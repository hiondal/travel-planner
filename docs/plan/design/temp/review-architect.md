# 유저스토리 리뷰 -- 아키 (소프트웨어 아키텍트)

> 리뷰 관점: 기술 실현 가능성, 시스템 아키텍처, Story Points 적절성
> 리뷰일: 2026-02-22
> 리뷰 대상: `docs/plan/design/userstory.md`
> 참조: 이벤트 스토밍 결과, 시퀀스 다이어그램(01~08), 리뷰 반영 결과

## 긍정 평가

### 1. 이벤트 스토밍 결과와 유저스토리의 정합성이 뛰어나다 (전체)

8개 유저플로우 -> 8개 Epic -> 22개 UFR 매핑이 빈틈없이 수행되었다. 섹션 10의 추적 매트릭스에서 UF-01~08의 모든 이벤트와 커맨드가 UFR로 커버됨을 확인할 수 있다. 정책 P1~P22 전체 22개가 UFR/NFR에 1:1로 매핑되어 설계-구현 간 추적성이 확보되었다.

### 2. MVP 모놀리스 우선 전략과 모듈 경계가 일관된다 (NFR-SCAL-010, UFR 서비스 태깅)

모든 UFR에 서비스 태그(SCHD/PLCE/MNTR/BRIF/ALTN)가 명시되어 있고, NFR-SCAL-010에서 "모듈 간 직접 의존 금지, 인터페이스 기반 간접 의존"을 명시하여 v0 모놀리스 -> v1 모니터링 분리 -> v2+ 점진적 분리 로드맵과 정합한다. 이는 이벤트 스토밍 섹션 10(MVP 배포 전략)의 분리 준비 설계 원칙과도 일치한다.

### 3. 외부 API 장애 내성 설계가 체계적이다 (UFR-MNTR-010, NFR-REL-010)

UFR-MNTR-010의 3개 시나리오(정상 수집, 부분 실패, 연속 실패)가 정책 P3/P20을 정확히 반영하고 있다. 특히 부분 실패 시 `is_cached`, `data_age_seconds`, `source_status=FALLBACK` 메타데이터 설계는 향후 AI 고도화(A2)의 기반이 된다. NFR-REL-010에서 서킷 브레이커 패턴까지 명시한 점이 적절하다.

### 4. MoSCoW 분류와 스프린트 배치가 솔루션 우선순위와 일치한다

Sprint 1(S05 배지) -> Sprint 2(S04 브리핑) -> Sprint 3(S06 대안카드) 순서가 통합 UX 흐름 "배지(Pull) -> 브리핑(Push) -> 대안카드(Action)"과 정확히 대응한다. 각 스프린트 목표가 독립적으로 데모 가능한 단위(Potentially Shippable Increment)로 구성되어 있다.

### 5. DoD(Definition of Done)가 구체적이고 실행 가능하다

Story Level 8개 항목, Sprint Level 5개 항목 모두 측정 가능한 기준으로 정의되었다. 특히 "커버리지 80% 이상", "WCAG 2.1 AA 접근성 확인", "p95 응답시간 기준 충족" 등 정량적 기준이 명확하다.

## 개선 의견

### G1. [UFR-MNTR-010] 외부 API 병렬 호출 전략이 명시되지 않았다

**대상**: UFR-MNTR-010 시나리오 1

**기술적 근거**: 현재 시퀀스 다이어그램(`02-외부데이터수집.puml`)에서 Google Places API, 날씨 API, 교통 API를 순차적으로 호출하는 구조다. 각 API 응답시간을 p95 기준 500ms~1.5s로 가정하면, 순차 호출 시 최악의 경우 3~4.5초가 소요된다. 이는 NFR-PERF-010(p95 2초 이내)과 직접 충돌한다. 3개 API 호출은 상호 독립적이므로 반드시 병렬 호출(CompletableFuture/WebFlux)로 설계해야 한다.

**수정 제안**: UFR-MNTR-010 시나리오 1 체크리스트에 다음 항목을 추가한다:
- "영업상태, 날씨, 이동시간 3개 API는 병렬로 동시 호출한다"
- "전체 수집 타임아웃은 2초로 설정하고, 타임아웃 초과 시 P20(부분 실패) 정책을 적용한다"

### G2. [UFR-SCHD-040] 이동시간 재계산의 외부 API 호출이 SP에 반영되지 않았다

**대상**: UFR-SCHD-040 (SP 8)

**기술적 근거**: "후속 일정의 이동시간 자동 재계산(P13)"은 단순 데이터 업데이트가 아니다. 대안 장소 교체 시 해당 장소의 전후 장소 간 이동시간을 재계산하려면 Google Directions API를 최소 2회(새 장소로 가는 경로 + 새 장소에서 다음 장소로 가는 경로) 호출해야 한다. 이 비동기 API 호출 + 실패 시 캐시 fallback + 일정표 원자적 갱신이 SP 8에 포함되어야 하는데, 현재 체크리스트에는 "후속 일정의 이동시간이 새 장소 기준으로 재계산된다"만 있어 구현 복잡도가 과소평가될 수 있다.

**수정 제안**: 체크리스트에 다음을 추가한다:
- "이동시간 재계산은 Google Directions API를 통해 수행되며, API 호출 실패 시 직선거리 기반 추정값을 사용한다"
- "재계산 대상은 교체된 장소의 직전 장소->교체 장소, 교체 장소->직후 장소 2구간이다"

### G3. [UFR-BRIF-010] 브리핑 생성 시 동기 호출 체인이 길어 성능 리스크가 있다

**대상**: UFR-BRIF-010 (SP 8)

**기술적 근거**: 시퀀스 다이어그램(`05-출발전브리핑생성.puml`)에서 브리핑 생성 흐름은 `브리핑업무 -> 데이터수집업무(최신 데이터 수집) -> 상태판정업무(종합 판정) -> 브리핑 생성`으로 3단계 동기 체인이다. 리뷰 의견 C20(데브-백, 파이프)에서도 지적되었듯, 출발 15~30분 전 트리거 시점에 온디맨드 재수집을 수행하면 외부 API 호출이 추가되어 브리핑 생성 지연이 발생할 수 있다.

**수정 제안**: UFR-BRIF-010 체크리스트에 다음을 추가한다:
- "브리핑 생성 시 최신 수집 데이터는 모니터링 서비스의 캐시된 데이터를 Read-Through 방식으로 조회한다 (온디맨드 외부 API 재호출 아님)"
- "캐시 데이터가 5분 이내이면 재수집 없이 사용, 5분 초과이면 비동기 수집 후 사용"

### G4. [UFR-ALTN-010] 대안 검색 파이프라인의 3초 제약이 기술적으로 도전적이다

**대상**: UFR-ALTN-010 (SP 8), NFR-PERF-020

**기술적 근거**: 대안 검색 전체 파이프라인은 `구독 티어 확인 -> 주변 장소 검색(Google Places API) -> 후보 상태 일괄 조회(모니터링) -> 정렬/카드 생성`이다. 반경 확장(P22) 발생 시 Google Places API를 최대 3회(1km, 2km, 3km) + 매 단계마다 후보 상태 조회가 필요하다. NFR-PERF-020에서 "반경 확장 포함 최악의 경우 5초 이내"로 완화했지만, 현재 UFR-ALTN-010 체크리스트에는 이 성능 제약이 명시되어 있지 않다.

**수정 제안**: UFR-ALTN-010 시나리오 2 체크리스트에 다음을 추가한다:
- "반경 확장 시 이전 단계 검색 결과를 누적하여 재검색 범위를 최소화한다 (1km 결과 유지 + 1km~2km 추가 검색)"
- "전체 파이프라인 타임아웃을 설정한다 (정상: 3초, 반경 확장 포함: 5초)"

### G5. [UFR-SCHD-010] 첫 일정 등록 시 권한 동의 플로우의 서비스 경계가 모호하다

**대상**: UFR-SCHD-010 시나리오 2

**기술적 근거**: 위치정보 수집 동의(P15)와 Push 알림 권한 요청은 클라이언트(프론트엔드) 영역의 네이티브 API 호출이다. 그러나 "동의 일시가 기록된다"는 서버 사이드 저장이 필요하다. 현재 이 UFR은 SCHD 서비스에 태깅되어 있는데, 동의 이력 저장은 사용자 프로필/인증 영역의 책임이다. MVP에서 별도 사용자 서비스가 없으므로(이벤트 스토밍 섹션 9), SCHD에서 처리하는 것이 맞지만, 체크리스트에 서버-클라이언트 책임 분리를 명시해야 한다.

**수정 제안**: 체크리스트에 다음을 추가한다:
- "위치정보 동의 UI 및 Push 권한 요청은 클라이언트에서 처리하고, 동의 결과(동의/거부, 일시)만 서버에 전송하여 저장한다"

### G6. [UFR-MNTR-020] 상태 판정 로직의 임계값 정의가 누락되었다

**대상**: UFR-MNTR-020 (SP 8)

**기술적 근거**: 체크리스트에 "초록: 영업 중 + 날씨 양호 + 혼잡도 보통 이하 + 이동시간 정상"으로 정의되어 있으나, "양호", "보통 이하", "정상"의 정량적 임계값이 없다. 예를 들어 날씨의 경우 강수확률 30% 미만이 "양호"인지, 기온 범위는 어떻게 되는지, 혼잡도는 어느 레벨까지가 "보통 이하"인지 등이 정의되어야 구현 시 해석 차이를 방지할 수 있다.

**수정 제안**: UFR-MNTR-020에 다음 체크리스트 항목을 추가하거나, 별도 부속 문서로 임계값 정의를 참조한다:
- "상태 판정 임계값은 설정 파일(config)로 관리하여 운영 중 조정 가능하게 한다"
- "MVP 초기 임계값: 날씨(강수확률 40% 이상 = 주의, 70% 이상 = 위험), 혼잡도(Google popular_times 기준 60% 이상 = 주의, 80% 이상 = 위험), 이동시간(예상 대비 1.5배 = 주의, 2배 = 위험)"

### G7. [NFR-SEC-010] JWT 토큰에 구독 티어 포함 시 정합성 리스크가 있다

**대상**: NFR-SEC-010 (SP 5)

**기술적 근거**: "구독 티어 정보가 토큰 클레임에 포함되어 각 서비스에서 참조"라고 명시되어 있다. 그러나 JWT는 발급 시점의 스냅샷이므로, 사용자가 여행 중 구독을 업그레이드하면 기존 토큰의 티어 정보가 구버전으로 남는다. Access Token의 만료 시간(일반적으로 15~30분)이 경과해야 갱신되므로, 그 사이에 Paywall이 잘못 표시되거나 브리핑 한도가 부정확하게 적용될 수 있다.

**수정 제안**: 체크리스트에 다음을 추가한다:
- "구독 티어 변경 시 기존 Access Token을 즉시 무효화하고 새 토큰을 재발급한다"
- "또는 구독 티어는 토큰 클레임이 아닌 서비스 내부 조회(캐시 포함)로 확인하되, MVP에서는 토큰 클레임 방식으로 시작하고 정합성 이슈 발생 시 전환한다"

## 기술적 리스크

### R1. [높음] 대안 검색 3초 SLA 달성 불확실 -- 기술 스파이크 필요

**대상**: UFR-ALTN-010, NFR-PERF-020

대안 검색 파이프라인은 최소 2회의 외부 API 호출(장소 검색 + 상태 조회)이 필수이며, 반경 확장 시 최대 6회까지 증가한다. Google Places API의 p95 응답시간이 지역과 시간대에 따라 500ms~2s로 변동되는 점을 고려하면, 3초 SLA 달성이 불확실하다.

**권고**: Sprint 3 착수 전에 기술 스파이크(1~2일)를 수행하여 다음을 검증한다:
- Google Places API Nearby Search의 실제 응답시간 측정 (도쿄, 오사카 등 목표 도시)
- 장소 데이터 사전 캐싱 전략의 효과 측정
- 반경 확장 시 누적 검색(이전 결과 유지 + 추가 범위만 검색)의 실현 가능성

### R2. [높음] 외부 API 병렬 호출 + 부분 실패 처리의 구현 복잡도

**대상**: UFR-MNTR-010

3개 외부 API의 병렬 호출, 개별 타임아웃, 부분 실패 시 항목별 캐시 fallback, 품질 메타데이터 첨부는 비동기 프로그래밍과 에러 핸들링의 복잡도가 높다. Spring Boot 기반이라면 WebClient + CompletableFuture 조합이 필요하며, 테스트 커버리지 확보도 까다롭다.

**권고**: Sprint 1 초반에 외부 API 호출 공통 모듈(서킷 브레이커 + 캐시 fallback + 타임아웃)을 먼저 구현하고, 이를 기반으로 개별 API 호출을 구현한다.

### R3. [중간] 스케줄러 기반 주기적 수집의 운영 안정성

**대상**: UFR-MNTR-010, UFR-BRIF-010

15분 주기 데이터 수집과 출발 15~30분 전 브리핑 트리거는 모두 스케줄러에 의존한다. v0 모놀리스에서 단일 인스턴스로 운영 시, 스케줄러 장애가 전체 서비스 중단으로 이어진다. 또한 동시 모니터링 장소가 증가하면 15분 윈도우 내에 모든 수집을 완료하지 못할 수 있다.

**권고**:
- 스케줄러는 ShedLock 등을 사용하여 단일 인스턴스에서만 실행되도록 보장한다
- 수집 작업을 비동기 워커 큐로 분산하여 동시 수집 가능하게 설계한다
- 리뷰 의견 D4(파이프)의 Watchdog 패턴을 Sprint 2에서 고려한다

### R4. [중간] Google Places API `popular_times` 필드 접근성

**대상**: UFR-MNTR-010

리뷰 의견 C10에서도 지적되었듯, Google Places API의 `popular_times`/`current_popularity` 필드는 공식 REST API에서 제한적으로 제공된다. New Places API(v1)에서는 지원 범위가 다를 수 있으며, 일부 장소에서는 데이터가 없을 수 있다. 이 경우 혼잡도 항목이 항상 "미확인"이 되어 상태 판정의 정확도가 낮아진다.

**권고**:
- 혼잡도 데이터 미제공 장소에 대한 fallback 정책(해당 항목을 "정상"으로 간주하고 나머지 3개 항목으로 판정)을 UFR-MNTR-020에 추가한다
- MVP 기술 스파이크에서 목표 도시의 주요 장소 카테고리별 `popular_times` 제공률을 조사한다

### R5. [낮음] FCM Push 알림의 플랫폼별 동작 차이

**대상**: UFR-BRIF-040, UFR-MNTR-050

iOS와 Android에서 FCM의 동작 방식(포어그라운드/백그라운드/콜드스타트)이 다르며, iOS의 경우 APNs를 경유한다. 리뷰 의견 C13(데브-프론트)에서 지적한 3케이스 분기가 프론트엔드 설계 단계로 미뤄져 있으므로, Sprint 2 시작 전까지 프론트엔드 설계에서 반드시 다뤄야 한다.

## 의존성 이슈

### D1. Sprint 1 내 MNTR-PLCE 순환 의존 리스크

Sprint 1에 PLCE-010(장소 검색)과 MNTR-010(외부 데이터 수집)이 함께 배치되어 있다. 이벤트 스토밍 결과에서 MNTR -> PLCE는 동기 호출(장소 서비스를 통한 외부 API 데이터 수집)로 정의되어 있는데, 시퀀스 다이어그램(`02-외부데이터수집.puml`)에서는 MNTR이 Google Places API를 직접 호출한다. 즉, MNTR이 PLCE를 경유하는지 직접 호출하는지 불일치가 있다.

**권고**: 두 가지 선택지를 명확히 한다:
- (A) MNTR이 PLCE를 경유: UFR-PLCE-010이 MNTR-010보다 먼저 완료되어야 함 (Sprint 1 내 순서 조정)
- (B) MNTR이 외부 API를 직접 호출: PLCE는 사용자 대면 검색 전용, MNTR은 백엔드 수집 전용으로 역할 분리
- MVP 모놀리스에서는 (B)가 구현이 단순하지만, 추후 분리 시 API 키 관리와 호출 집계가 분산되는 단점이 있으므로 (A)를 권장한다.

### D2. Sprint 3의 ALTN -> SCHD -> MNTR 3단 의존 체인

UFR-ALTN-030(대안 선택/일정 반영)은 SCHD-040(일정 장소 교체)에 동기 의존하고, SCHD-040은 모니터링 대상 변경을 비동기 발행한다. Sprint 3에 SCHD-040이 배치되어 있으나, 이 UFR은 Sprint 1에서 구현한 SCHD, MNTR 모듈의 이벤트 발행 인프라에 의존한다.

**권고**: Sprint 1에서 비동기 이벤트 발행 인프라(인메모리 이벤트 버스)를 공통 모듈로 먼저 구현하고, Sprint 1의 SCHD-030(장소 추가 시 모니터링 등록)에서 이를 검증한다. Sprint 3에서는 동일 인프라를 재활용하면 된다.

### D3. NFR-PERF-010이 Sprint 2에 배치된 시점 리스크

NFR-PERF-010(API 응답시간 p95 2초)이 Sprint 2에 배치되어 있으나, Sprint 1에서 구현하는 일정표 조회(SCHD-050)와 배지 조회(MNTR-030)도 사용자 대면 API이다. Sprint 1 완료 시 성능 기준을 검증하지 않으면 Sprint 2에서 뒤늦게 성능 이슈를 발견하여 리팩토링이 발생할 수 있다.

**권고**: NFR-PERF-010을 Sprint 1로 이동하거나, 최소한 Sprint 1의 DoD에 "사용자 대면 API의 p95 응답시간 2초 이내 확인"을 포함시킨다.

### D4. UFR-PLCE-030(주변 장소 검색)의 스프린트 배치 재검토

UFR-PLCE-030은 Sprint 3에 배치되어 있으나, 이는 UFR-ALTN-010(대안 장소 검색)의 선행 조건이다. ALTN-010과 PLCE-030이 같은 Sprint 3에 있으므로 구현 순서를 Sprint 3 내에서 명확히 해야 한다.

**권고**: Sprint 3 계획에 구현 순서를 명시한다: PLCE-030 -> ALTN-010 -> ALTN-020 -> ALTN-030 -> SCHD-040 -> ALTN-040/050.

## 총평

도그냥(서비스 기획자)이 작성한 유저스토리 초안은 이벤트 스토밍 결과를 충실히 반영하였으며, 22개 정책의 UFR/NFR 추적성, MoSCoW 분류의 적절성, 스프린트 단위의 독립 데모 가능성 측면에서 높은 완성도를 보인다.

아키텍처 관점에서 핵심 보완 사항은 다음 3가지로 요약한다:

1. **외부 API 호출 전략의 구체화**: 병렬 호출, 타임아웃, 부분 실패 처리가 유저스토리 체크리스트에 명시되어야 구현 시 해석 차이와 성능 리스크를 방지할 수 있다 (G1, G2, G3, G4).

2. **SP 재배분 (MNTR-010: 8->13, ALTN-010: 8->13)**: 외부 API 연동이 집중된 두 UFR의 복잡도가 과소평가되어 있다. 총 SP는 거의 변동 없지만, 해당 UFR에 대한 팀의 공수 예측이 현실적이어야 스프린트 완주율이 높아진다.

3. **기술 스파이크 선행**: 대안 검색 3초 SLA(R1)와 Google popular_times 데이터 가용성(R4)은 Sprint 3 착수 전 반드시 검증해야 한다. Sprint 2 기간 중 1~2일을 기술 스파이크에 할당할 것을 권고한다.

의존성 측면에서 NFR-PERF-010의 Sprint 1 이동(D3)과 Sprint 3 내 구현 순서 명시(D4)가 필요하다.

전체적으로 유저스토리의 구조와 품질은 다음 단계(설계)를 진행하기에 충분하며, 상기 보완 사항을 반영하면 구현 리스크를 크게 줄일 수 있을 것으로 판단한다.
