## 2. 미반영 항목 및 사유

### 2-1. MVP 이후로 미룬 항목

| # | 항목 | 제안자 | 미룬 사유 | 반영 시점 |
|:-:|------|--------|----------|----------|
| M6 | 오프라인 모드 동작 정의 | 인사이트 | MVP 범위 과다. 해외 여행 중 오프라인은 빈번하나, 최초 출시에서는 네트워크 연결 전제 서비스로 시작하고 사용자 피드백에 따라 Phase 2에서 추가한다. 오프라인 상태 감지 및 로컬 캐시 설계는 기술 복잡도가 높아 MVP 일정을 위협할 수 있다. | Phase 2 (Should) |
| M7 | 동일 장소 중복 등록 방지 | 인사이트 | 의도적 재방문(같은 카페 아침/저녁)과 실수 중복을 구분해야 하는 UX 설계가 필요하다. MVP에서는 경고 수준으로 처리하되 별도 UFR까지는 불필요하다. Sprint 후반에 간단한 중복 확인 경고를 SCHD-030 체크리스트에 추가하는 것으로 대체 가능. | Phase 2 (Should) |
| M8 | 알림 수신 시간대 설정 (DND) | 인사이트 | MVP 사용자 규모에서는 알림 빈도가 제한적이다(하루 4~6회 브리핑). DND 설정은 사용자 수 증가 후 앱스토어 리뷰 피드백에 따라 우선순위를 판단한다. | Phase 2 (Should) |
| M9 | 현지 공휴일/특수 이벤트 처리 | 인사이트 | 공휴일 DB 연동은 도시별 커스터마이징이 필요하고 데이터 유지 비용이 높다. MVP 초기 5개 도시에 한해 Google Places API의 exceptional_opening_hours 필드를 실험적으로 활용하되, 별도 UFR로 분리하지 않는다. SCHD-030의 영업시간 경고에서 "공휴일 여부 현지 확인" 안내 텍스트로 대체한다. | Phase 2 (Should) |
| M10 | 긴급 상황 대응 흐름 | 인사이트 | 자연재해/테러 경보 등 비정상 상황은 빈도가 낮고 대응 로직의 복잡도가 매우 높다. MVP에서 일반 브리핑으로 오안내하는 리스크는 낮으며, Phase 2에서 외교부 해외안전여행 경보 API 연동을 검토한다. | Phase 2 (Could) |
| M11 | 결제/환율/가격대 정보 | 인사이트 | Google Places API의 price_level 필드 활용은 간단하나, 대안 카드에 가격대를 추가하면 정보 과부하로 "열고, 보고, 탭한다" 3단계 UX가 훼손될 수 있다. Phase 2에서 카드 확장 정보로 검토한다. | Phase 2 (Could) |
| T6 | 가명화 사용자 ID 연결 구조 | 마법사 | AI 개인화 학습용 행동 이력 + 가명처리 양립 구조는 데이터 설계 단계에서 아키와 협의하여 스키마 수준에서 반영한다. 유저스토리 체크리스트보다 데이터 모델 문서가 적합한 위치이다. | 데이터 설계 단계 |
| T7 | 카드별 화면 거주 시간 수집 | 마법사 | 화면 거주 시간 수집은 프론트엔드 이벤트 트래킹 인프라가 필요하다. MVP P18 분석 이벤트 4종으로 충분한 행동 시그널을 확보할 수 있으며, Phase 2에서 프론트엔드 분석 고도화 시 함께 추가한다. | Phase 2 (선택) |
