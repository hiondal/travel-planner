## 3. 유저스토리 (UFR)

### 3-0. 인증 서비스 (AUTH)

---

#### UFR-AUTH-010: [소셜 로그인] 여행자로서 | 나는, 빠르게 서비스를 시작하기 위해 | Google 또는 Apple 계정으로 간편 로그인할 수 있기를 원한다.

- 시나리오 1: Google 계정 로그인 성공
  여행자가 앱을 처음 실행한 상태에서 | Google 로그인 버튼을 탭하면 | Google OAuth 인증 화면이 표시되고, 인증 완료 후 JWT Access Token + Refresh Token이 발급되어 로그인 상태가 된다.

  [체크리스트]
  - Google OAuth 2.0 기반 인증을 지원한다
  - 인증 완료 후 JWT Access Token(만료 30분) + Refresh Token이 발급된다
  - 최초 로그인 시 닉네임, 프로필 이미지가 Google 계정에서 자동 설정된다
  - 로그인 상태는 Refresh Token 유효 기간 동안 유지된다

- 시나리오 2: Apple 계정 로그인 성공
  여행자가 앱을 처음 실행한 상태에서 | Apple 로그인 버튼을 탭하면 | Apple Sign In 인증이 진행되고, 인증 완료 후 JWT Access Token + Refresh Token이 발급되어 로그인 상태가 된다.

  [체크리스트]
  - Apple Sign In을 지원한다 (iOS 필수, Android 선택)
  - Apple의 이메일 숨기기(Hide My Email) 옵션을 지원한다
  - 인증 실패 시 "로그인에 실패했습니다. 다시 시도해주세요" 에러 메시지가 표시된다
  - 네트워크 오류 시 "인터넷 연결을 확인해주세요" 안내가 표시된다

- M/5

### 3-1. 일정 서비스 (SCHD)

---

#### UFR-SCHD-005: [온보딩] 여행자로서 | 나는, 앱의 핵심 가치를 빠르게 이해하기 위해 | 첫 실행 시 3단계 온보딩(배지 -> 브리핑 -> 대안 카드)과 샘플 일정 체험을 제공받기를 원한다.

- 시나리오: 첫 실행 온보딩 가이드
  여행자가 로그인 후 처음 앱을 사용하는 상태에서 | 메인 화면이 로드되면 | 3단계 온보딩(1: 상태 배지 소개, 2: 출발 전 브리핑 소개, 3: 대안 카드 소개)이 순서대로 표시되고, 완료 후 샘플 일정("도쿄 체험 일정")을 확인할 수 있다.

  [체크리스트]
  - 3단계 온보딩은 스와이프 또는 다음 버튼으로 진행한다
  - 1단계: 상태 배지(초록/노랑/빨강) 의미 설명
  - 2단계: 출발 전 브리핑 Push 알림 설명
  - 3단계: 대안 카드 탭 한 번 일정 반영 설명
  - "건너뛰기" 버튼으로 온보딩을 스킵할 수 있다
  - 온보딩 완료 후 샘플 일정이 제공되어 배지/브리핑 흐름을 미리 체험할 수 있다
  - 온보딩은 최초 1회만 표시되며, 설정에서 다시 볼 수 있다

- M/3

---

#### UFR-SCHD-010: [여행 일정 생성] 여행자로서 | 나는, 해외 자유여행 일정을 체계적으로 관리하기 위해 | 여행명, 여행 기간, 방문 도시를 입력하여 새로운 여행 일정을 생성할 수 있기를 원한다.

- 시나리오 1: 신규 여행 일정 생성 성공
  여행자가 앱에 로그인한 상태에서 | 여행명 "도쿄 3박4일", 기간 "2026-03-15 ~ 2026-03-18", 도시 "도쿄"를 입력하고 생성 버튼을 탭하면 | 일정 ID가 발급되고 빈 일정표가 생성된다.

  [체크리스트]
  - 여행명은 1~50자 사이여야 한다
  - 여행 기간은 시작일이 종료일 이전이어야 한다
  - 도시는 지원 도시 목록에서 선택해야 한다
  - 일정 생성 후 빈 일정표 화면으로 이동한다

- 시나리오 2: 첫 일정 등록 시 권한 동의 (P15)
  여행자가 첫 번째 여행 일정을 생성하는 경우 | 일정 생성 버튼을 탭하면 | 위치정보 수집 동의 화면이 먼저 표시되고, 동의 후 Push 알림 권한 요청이 표시된다.

  [체크리스트]
  - 위치정보 수집 동의 화면에 수집 항목, 이용 목적, 보유 기간이 명시된다
  - Push 알림 권한 요청에 알림 유형(브리핑, 상태 변경)이 표시된다
  - 위치정보 동의를 거부하면 구체적 제한 범위가 안내된다: "수동 위치 입력으로 일정 등록 가능, 실시간 현재 위치 기반 이동시간 계산 불가, 대안 카드의 현재 위치 기반 거리 기능 불가" [U20 반영]
  - 위치정보 없이 이용 가능한 기능 목록이 동의 화면에서 명확히 안내된다 [U20 반영]
  - Push 알림을 거부해도 일정 생성은 가능하다 (브리핑 수신 불가 안내)
  - 동의 일시가 기록된다
  - 위치정보 동의 UI 및 Push 권한 요청은 클라이언트에서 처리하고, 동의 결과(동의/거부, 일시)만 서버에 전송하여 저장한다 [U12 반영]

- M/5

#### UFR-SCHD-020: [장소 검색 및 선택] 여행자로서 | 나는, 방문하고 싶은 장소를 일정에 추가하기 위해 | 장소명을 검색하고 검색 결과에서 원하는 장소를 선택할 수 있기를 원한다.

- 시나리오: 장소 검색 및 선택
  여행자가 일정표 화면에서 장소 추가 버튼을 탭한 상태에서 | 검색창에 "이치란 라멘 시부야"를 입력하면 | 장소 후보 목록(이름, 주소, 평점)이 표시되고, 원하는 장소를 탭하여 선택할 수 있다.

  [체크리스트]
  - 검색 결과는 장소명, 주소, 평점을 포함한다
  - 검색 결과는 최대 10개까지 표시된다
  - 검색어는 최소 2글자 이상이어야 한다
  - 해당 도시 범위 내 장소만 검색된다

- M/3

---

#### UFR-SCHD-030: [일정에 장소 추가] 여행자로서 | 나는, 선택한 장소를 원하는 시간에 방문하기 위해 | 장소와 방문 일시를 지정하여 일정표에 추가할 수 있기를 원한다.

- 시나리오 1: 영업시간 내 장소 추가 성공 (P1, P16)
  여행자가 "이치란 라멘 시부야"를 선택한 상태에서 | 방문 일시를 "2026-03-16 12:00 (현지시간)"으로 지정하고 추가 버튼을 탭하면 | 장소가 일정표에 추가되고 모니터링 대상으로 자동 등록된다.

  [체크리스트]
  - 방문 일시는 장소 현지 시간(IANA timezone) 기준으로 저장된다 (P16)
  - 방문 시간이 장소 영업시간 범위 내인지 검증한다 (P1)
  - 영업시간 외 시간을 선택하면 경고 메시지가 표시된다
  - 장소 추가 시 모니터링 대상 등록 이벤트가 발행된다 (P14)
  - 일정표에 추가된 장소가 시간순으로 정렬되어 표시된다

- 시나리오 2: 영업시간 외 장소 추가 경고
  여행자가 영업시간이 11:00~22:00인 장소를 선택한 상태에서 | 방문 일시를 "09:00"으로 지정하면 | "영업시간 외입니다 (영업시간: 11:00~22:00)" 경고가 표시되고 추가 여부를 재확인한다.

  [체크리스트]
  - 경고 메시지에 실제 영업시간이 표시된다
  - 여행자가 경고를 무시하고 추가할 수 있다 (강제 차단 아님)
  - 영업시간 외로 등록된 장소는 모니터링 대상으로 등록하되, 상태 배지 판정 시 "영업시간 외 예정 방문" 마킹을 적용하여 오경보를 방지한다 [U18 반영]

- M/5

#### UFR-SCHD-040: [일정 장소 교체] 여행자로서 | 나는, 대안 장소를 선택했을 때 기존 일정을 빠르게 변경하기 위해 | 원래 장소가 대안 장소로 교체되고 후속 이동시간이 자동 재계산되기를 원한다.

- 시나리오: 대안 선택 후 일정 자동 교체 (P12, P13, P14)
  여행자가 대안 카드에서 장소를 선택한 상태에서 | 일정 교체가 실행되면 | 원래 장소가 대안 장소로 교체되고, 후속 일정의 이동시간이 자동으로 재계산되며, 모니터링 대상이 변경된다.

  [체크리스트]
  - 원래 장소의 모니터링이 해제된다
  - 대안 장소의 모니터링이 등록된다 (P14)
  - 후속 일정의 이동시간이 새 장소 기준으로 재계산된다 (P13)
  - 이동시간 재계산은 Google Directions API를 통해 수행되며, API 호출 실패 시 직선거리 기반 추정값을 사용한다 [U11 반영]
  - 재계산 대상은 교체된 장소의 직전 장소->교체 장소, 교체 장소->직후 장소 2구간이다 [U11 반영]
  - 변경 요약(원래 장소 -> 대안 장소, 변경된 이동시간)이 여행자에게 표시된다
  - 모니터링 대상 변경은 비동기 이벤트로 발행된다

- M/8

#### UFR-SCHD-050: [일정표 조회] 여행자로서 | 나는, 오늘의 여행 일정과 각 장소의 현재 상태를 한눈에 파악하기 위해 | 일정표에 장소별 상태 배지가 함께 표시되기를 원한다.

- 시나리오: 배지 포함 일정표 조회 (P4, P17)
  여행자가 앱을 열고 일정표를 조회하면 | 장소 목록이 시간순으로 표시되고 | 각 장소명 옆에 상태 배지(초록+체크/노랑+느낌표/빨강+X/회색+물음표)가 함께 표시된다.

  [체크리스트]
  - 장소별로 초록/노랑/빨강/회색 4단계 상태 배지가 표시된다 (P4)
  - 배지는 색상 + 아이콘 조합으로 색약 사용자를 지원한다 (P17, WCAG 2.1)
  - 회색 배지에는 "데이터 미확인" 문구가 투명하게 표시된다 (P6)
  - 일정표 로딩 시 모니터링 서비스에서 최신 배지 상태를 조회한다

- M/3

### 3-2. 장소 서비스 (PLCE)

---

#### UFR-PLCE-010: [장소 검색] 여행자로서 | 나는, 방문하고 싶은 장소를 찾기 위해 | 키워드로 장소를 검색하고 결과 목록을 확인할 수 있기를 원한다.

- 시나리오: 키워드 기반 장소 검색
  여행자가 장소 추가 화면에서 | "시부야 라멘"을 검색하면 | Google Places API를 통해 검색된 장소 목록(이름, 주소, 영업시간, 좌표, 평점)이 반환된다.

  [체크리스트]
  - 검색 결과에 장소명, 주소, 영업시간, 평점이 포함된다
  - 검색은 현재 여행 도시 범위 내로 한정된다
  - API 응답 지연 시 로딩 인디케이터가 표시된다
  - 검색 결과가 없으면 "검색 결과가 없습니다" 메시지가 표시된다

- M/3

---

#### UFR-PLCE-020: [장소 상세 정보 조회] 여행자로서 | 나는, 장소를 일정에 추가하기 전에 상세 정보를 확인하기 위해 | 장소의 영업시간, 위치, 카테고리 등 상세 정보를 조회할 수 있기를 원한다.

- 시나리오: 장소 상세 정보 조회
  여행자가 검색 결과에서 특정 장소를 탭한 상태에서 | 상세 보기를 요청하면 | 장소의 영업시간, 좌표, 카테고리, 사진, 리뷰 평점 등 상세 정보가 표시된다.

  - 최대 반경 검색 후에도 3장 미만이면 가용한 수량만 제시한다
  - 반경 확장 시 이전 단계 검색 결과를 누적하여 재검색 범위를 최소화한다 (1km 결과 유지 + 1km~2km 추가 검색) [U8 반영]

  [체크리스트]
  - 삭제 전 확인 다이얼로그가 표시된다
  - 삭제된 장소의 모니터링이 해제된다
  - 후속 일정의 이동시간이 재계산된다

- S/3

---

#### UFR-SCHD-070: [일정 장소 순서 변경] 여행자로서 | 나는, 방문 순서를 유연하게 조정하기 위해 | 일정표에서 장소의 방문 순서를 드래그하여 변경할 수 있기를 원한다.

- 시나리오: 드래그 앤 드롭으로 순서 변경
  여행자가 일정표에서 장소를 길게 눌러 드래그한 상태에서 | 원하는 위치에 드롭하면 | 방문 순서가 변경되고 이동시간이 재계산된다.

  [체크리스트]
  - 드래그 앤 드롭으로 순서 변경이 가능하다
  - 순서 변경 후 이동시간이 자동 재계산된다
  - 영업시간 범위를 벗어나는 경우 경고가 표시된다

- C/5

---

#### UFR-BRIF-060: [브리핑 목록 조회] 여행자로서 | 나는, 오늘 수신한 브리핑들을 다시 확인하기 위해 | 브리핑 이력 목록을 조회할 수 있기를 원한다.

- 시나리오: 브리핑 이력 목록
  여행자가 브리핑 목록 화면에 진입하면 | 오늘 수신한 브리핑들이 시간순으로 표시되고 | 각 브리핑의 유형(안심/주의), 장소명, 생성 시각이 표시된다.

  [체크리스트]
  - 브리핑이 시간순(최신 순)으로 정렬된다
  - 브리핑 유형(안심/주의)이 시각적으로 구분된다
  - 브리핑을 탭하면 상세 내용이 표시된다
  - 오늘 날짜 기준으로 필터링된다

## 4. 비기능적 요구사항 (NFR)

### 4-1. 성능

---

#### NFR-PERF-010: [API 응답 시간] 사용자 대면 API의 응답 시간은 95번째 백분위(p95) 기준 2초 이내여야 한다.

- 적용 대상: 일정표 조회, 배지 조회, 브리핑 상세 조회, 대안 카드 조회
- 측정 기준: 서버 응답 시간 (네트워크 지연 제외)
- 모니터링: APM 도구로 지속 모니터링

- M/5

---

#### NFR-PERF-020: [대안 검색 응답 시간] 대안 카드 검색 및 제시는 3초 이내에 완료되어야 한다.

- 적용 대상: 대안 장소 검색 -> 상태 확인 -> 카드 생성 전체 파이프라인
- 근거: "23.5분 -> 3분 이내" 핵심 KPI. 대안 카드 제시까지 3초 초과 시 사용자 이탈 위험
- 반경 확장(P22) 포함 최악의 경우에도 5초 이내

- M/5

---

#### NFR-PERF-030: [Push 알림 지연] 브리핑 생성 후 Push 알림 발송까지 지연 시간은 10초 이내여야 한다.

- 적용 대상: 브리핑 생성 -> FCM 발송
- 근거: 출발 15~30분 전 시간 제약 내에서 의미 있는 사전 인지 제공

- S/3

### 4-2. 보안

---

#### NFR-SEC-010: [인증 및 인가] 모든 API 요청은 JWT 토큰 기반 인증이 적용되어야 한다.

- 인증: 소셜 로그인(Google, Apple) 기반 JWT Access Token + Refresh Token
- 인가: 구독 티어 정보가 토큰 클레임에 포함되어 각 서비스에서 참조
- API Gateway 레벨에서 토큰 검증
- 구독 티어 변경 시 기존 Access Token을 즉시 무효화하고 새 토큰을 재발급한다. MVP에서는 토큰 클레임 방식으로 시작하고 정합성 이슈 발생 시 서비스 내부 조회(캐시 포함)로 전환한다 [U13 반영]

- M/5

---

#### NFR-SEC-020: [개인정보 보호] 위치정보 수집 및 처리는 위치정보법/GDPR을 준수해야 한다. (P15)

- 위치정보 수집 동의 절차 필수 (P15)
- 수집 항목, 이용 목적, 보유 기간 명시
- 사용자 요청 시 데이터 삭제 가능
- 분석 데이터는 가명처리 후 활용
- 위치정보 보유 기간: 여행 종료 후 30일, 이후 자동 삭제 [U21 반영]
- 분석 데이터(가명처리): 서비스 종료 시까지 보유 [U21 반영]
- GDPR Article 17 기준 삭제 요청 처리: 72시간 이내 [U21 반영]
- 한국 거주자 위치정보 국외 이전 시 개인정보보호법 제28조의8 준수 확인 (Google API 전송 시 별도 고지 동의 추가) [U21 반영]

- M/5

### 4-3. 사용성/접근성

---

#### NFR-UX-010: [접근성 WCAG 2.1 AA] 상태 배지는 색상 + 아이콘 조합으로 색약 사용자를 지원해야 한다. (P17)

- 초록+체크, 노랑+느낌표, 빨강+X, 회색+물음표 (P17)
- 색상만으로 정보를 전달하지 않는다
- 터치 타겟 최소 44x44px
- 텍스트 대비비 4.5:1 이상

- M/3

#### NFR-UX-020: [모바일 퍼스트] 모든 화면은 Mobile First로 설계되어야 한다.

- 최소 지원 해상도: 375x667 (iPhone SE)
- 반응형 레이아웃 (320px ~ 428px)
- 터치 기반 인터랙션 최적화 (스와이프, 탭, 길게 누르기)

- M/3

### 4-4. 확장성

---

#### NFR-SCAL-010: [모듈 분리 가능성] 5개 서비스는 논리적 모듈로 분리되어 추후 독립 배포 단위로 전환할 수 있어야 한다.

- 모듈 간 직접 의존 금지 (인터페이스 기반 간접 의존)
- 서비스 간 통신은 인터페이스로 추상화 (MVP에서는 인메모리)
- 데이터베이스 스키마는 모듈별 논리적 분리

- S/5

#### NFR-SCAL-020: [도시 확장] MVP 이후 지원 도시를 5개에서 15개 이상으로 확장할 수 있어야 한다.

- 도시별 설정(timezone, 지원 카테고리)이 설정 파일로 관리된다
- 외부 API 키/엔드포인트가 도시별로 구성 가능하다
- 장소 데이터 캐싱이 도시별로 분리된다

- S/5

### 4-5. 안정성

---

#### NFR-REL-010: [외부 API 장애 내성] 외부 API(Google Places, OpenWeatherMap, Google Directions) 장애 시 서비스가 완전 중단되지 않아야 한다. (P3, P20)

- 부분 실패 시 실패 항목만 캐시 fallback (P20)
- 3회 연속 전체 실패 시 회색(미확인) 상태로 투명하게 표시 (P3)
- 서킷 브레이커 패턴 적용
