### 순위 변경 근거

**B를 1위로 격상한 이유 -- 비즈니스 가치 관점의 재판단**

기존에는 A를 1위로 선정했다. "23.5분을 5분으로 줄인다"는 가치 제안의 즉각성과 측정 가능성이 이유였다. 그러나 인터뷰 검증 결과, 이 판단을 수정한다.

1. **검증 강도의 압도적 차이**: B는 중요도 5.0 + 불편도 5.0 + 동의율 100% + 만장일치 최심각이라는 4관왕이다. 3개 문제 중 유일하게 모든 지표가 만점이다. A는 불편도 4.3으로 B보다 낮다.

2. **인과 체인의 레버리지**: 3명의 인터뷰이가 모두 "B가 A와 C의 원인"이라는 인과 구조를 독립적으로 인식했다. S1은 "미리 알았으면 검색 자체를 안 해도 되니까"라고 명시했다. B를 해결하면 A의 발생 빈도 자체가 감소한다. 이것이 MVP에서 가장 큰 비즈니스 레버리지다.

3. **감정 보호 > 시간 절약**: 인터뷰에서 발견된 핵심 인사이트는 사용자의 핵심 동기가 "시간 절약"보다 "감정 보호"라는 점이다. 솔로의 감정 고립, 그룹의 관계 훼손, 초보의 자기 의심 -- 이 감정적 고통의 해소가 더 강력한 가치 제안이며, 리텐션과 유료 전환의 진짜 트리거다.

4. **A의 가치 제안은 B 해결 후에 완성된다**: "23.5분을 5분으로"라는 A의 가치 제안은, B(사전 알림)가 해결된 상태에서 더 빛난다. 사전에 알림을 받고 여유 있게 대안을 탐색하는 것과, 도착 후 당황한 상태에서 급히 탐색하는 것은 같은 기능이라도 사용자 경험의 질이 다르다.

## 문제가설 1 (피보팅): 사후 인지 -- 80%가 도착 후에야 알게 되어 감정 절벽이 발생한다

> 피보팅 유형: **강화** (기존 2위 → 1위 격상, MVP 1순위 기능 확정)

### 문제가설

여행자의 80%가 임시 휴업, 혼잡, 교통 중단 등의 돌발 상황을 목적지에 도착한 후에야 인지하여 이동 시간(15~75분)이 매몰 비용이 되고 감정 곡선이 -2.0점 급락하는데, 이는 여행 일정에 등록된 장소의 상태를 실시간으로 모니터링하여 이동 전에 선제적으로 알려주는 Push 시스템이 부재하기 때문이다.

### 5WHY 분석

- **Why1**: 여행자의 80%가 임시 휴업, 혼잡, 교통 중단 등을 도착 후에야 인지한다
  - **Why2**: 현재 모든 여행 도구가 사용자가 직접 검색해야 정보를 제공하는 Pull 모델이다
    - **Why3**: 여행 일정에 등록된 장소를 실시간으로 모니터링하는 기술적 파이프라인이 부재하다
      - **Why4**: 구글맵 영업시간은 사업자 자발적 등록에 의존하며(임시 휴업 갱신율 30% 미만), POI 실시간 상태 데이터의 수집-유통 체계가 없다
        - **Why5**: 관광 데이터 생태계가 "예약 전환" 수익에 종속되어 실시간 현장 데이터 수집/유통의 경제적 인센티브가 구조적으로 부재하다

### 다층적 근본원인 검토

| 측면 | 근본원인 | 중요도 |
|------|---------|:------:|
| 사용자 심리적 | "계획대로 될 것"이라는 확증 편향(confirmation bias)과 플랫폼 권위에 대한 과신. 구글맵 "영업 중" 표시를 무비판적으로 신뢰하고 추가 검증을 생략. 국내 경험에서 형성된 "앱 정보 = 현실" 신뢰 모델이 해외에서 반복적으로 깨지면서 도구 불신 축적. **[인터뷰 추가] 세그먼트별 감정 반응 차이 -- 솔로: 감정 고립(부정 감정을 전환해줄 동행 부재), 그룹: 책임 추궁("왜 미리 확인 안 했어?"), 초보: 자기 의심("자유여행이 나에게 안 맞나")** | ★★★★★ |
| 제도/시스템적 | 현재 모든 여행 도구가 Pull 모델. 여행자의 일정에 등록된 장소를 실시간 모니터링하여 이동 전에 선제적으로 알리는 Push 모델이 부재. 구글맵 영업시간은 사업자 자발적 등록에 의존(임시 휴업 갱신율 30% 미만 추정). 관광 데이터 생태계가 "예약 전환" 수익에 종속되어 실시간 현장 데이터 수집/유통의 경제적 인센티브가 부재 | ★★★★★ |
| 맥락적 | 해외 여행에서 현지 정보는 현지어(일본어, 태국어 등)로 제공되어 외국인 여행자의 사전 접근이 구조적으로 불가능. 임시 휴업, 국가 행사, 교통 점검 등은 현지어 공식 채널에서만 안내. 전화 확인도 언어/요금 문제로 비현실적 | ★★★★ |
| 커뮤니케이션 | 돌발 상황 정보의 전달 체계가 "현장 안내문"에 의존하여, 정보가 도착 시점에만 전달되는 구조적 지연 발생. POI 운영자(점포)는 임시 휴업을 자기 SNS 채널에만 발행하며, 이 정보가 여행자가 사용하는 서비스로 전파되는 채널이 없음. **[인터뷰 추가] SNS 콘텐츠 손실 -- 기대했던 장소를 못 가면 "인생샷"을 못 찍어서 이중 아쉬움 발생 (S2)** | ★★★ |

### 핵심 근본원인

**"여행 일정 기반 선제적 리스크 감지/알림(Push) 시스템"의 부재**

현재 모든 여행 도구는 사용자가 직접 정보를 찾아야 하는 Pull 모델이다. 여행자의 일정에 등록된 장소를 실시간으로 모니터링하고, 이동 전에 상태 변화를 선제적으로 알려주는 Push 모델이 없기 때문에, 여행자는 매번 도착해서야 당황하게 된다. 이것을 핵심 근본원인으로 선정한 이유는, 제도/시스템적 측면(★★★★★)이 이 문제의 본질이며, 심리적 과신(사용자가 검증을 생략하는 행동)과 맥락적 접근 불가(현지어 장벽) 모두 "시스템이 알아서 감시해주지 않기 때문에" 발생하는 파생 문제이기 때문이다. Push 시스템이 존재한다면, 사용자의 확증 편향이 존재하더라도 시스템이 이를 보정해준다.

**4. 정보 정확도 90% 이상 = 서비스 존재의 전제 조건**

S1이 명시적으로 "정확도 90% 이상은 보장되어야 신뢰가 쌓인다"고 발언. 3명 모두 구글맵/블로그의 정보 부정확성이 사후 인지의 직접 원인이라고 지적. 우리 서비스가 동일한 오류를 반복하면 기존 도구와 차별성이 없다. **정확도는 기능이 아니라 존재 이유(reason to exist)이다.**

## 문제가설 2 (피보팅): 대안 탐색에 23.5분+ 소요 -- 여행 시간이 "검색 노동"과 "합의 비용"으로 직접 손실된다

> 피보팅 유형: **유지 + 보완** (기존 1위 → 2위 조정, 그룹 합의 비용 범위 포함)

### 문제가설

여행 중 돌발 상황(휴업, 날씨 급변, 혼잡, 교통 장애)이 발생했을 때, 여행자가 대안 장소를 찾고 선택하기까지 평균 23.5분이 소요되며, **그룹 여행의 경우 검색 결과 논쟁과 합의에 추가 20~40분이 발생**하여 여행 시간이 직접 손실되는데, 이는 여행자의 현재 맥락(위치, 시간, 날씨, 취향)을 이해하고 통합된 대안을 즉시 제시하는 서비스가 시장에 부재하기 때문이다.

### 5WHY 분석

## 문제가설 3 (피보팅): 일정 변경 1건이 연쇄 붕괴 -- 감정 취약 시점에 수동 재계산을 강제한다

> 피보팅 유형: **수정** (근본원인을 단일 구조 → 복합 구조로 변경)

### 문제가설

돌발 상황으로 하나의 일정이 변경되면 평균 2~3곳의 추가 변경이 연쇄적으로 발생하며, 여행자가 감정적으로 가장 취약한 시점(감정 레벨 1.6~1.9)에 인지적으로 가장 복잡한 수동 재계산을 수행해야 하는데, 이는 **(1) 상황 변화 감지(B)와 연계된 선제적 재조정 파이프라인, (2) 세그먼트별 차별화된 재조정 UX, (3) 재조정 결과의 투명성/설명 가능성**이 모두 부재하기 때문이다.

### 5WHY 분석

- **Why1**: 하나의 일정 변경이 평균 2~3곳의 추가 변경을 연쇄적으로 유발한다("도미노 효과")
  - **Why2**: 일정 항목 간에 이동 동선, 영업시간, 예약 시간, 체류시간의 의존 관계가 존재하는데, 현재 도구는 이를 독립 항목의 나열로만 관리한다
    - **Why3**: 노션, 트리플, 메모 앱 등 기존 일정 도구가 "정적 리스트" 구조에 머물러 장소 간 의존성을 데이터로 관리하지 못한다. **[인터뷰 반영] 또한 상황 변화 감지(B: Push 알림)가 선행되어야 재조정이 의미를 가지나, 감지-재조정 파이프라인 자체가 부재하다 (S1 명시)**
      - **Why4**: 동적 일정 재조정은 다제약 최적화(multi-constraint optimization) 문제로, 정량적 제약(거리, 시간)과 정성적 맥락(취향, 감정, 동행자)을 동시에 처리해야 하는 기술적 난이도가 높다. **[인터뷰 반영] 세그먼트별로 원하는 재조정 방식이 근본적으로 달라 단일 UX로 해결 불가 (S1/S2/S3)**
        - **Why5**: "여행 중 동적 재조정"이라는 문제 정의 자체가 업계에서 간과되어 왔으며, 이를 해결할 기술(LLM + 제약 최적화)이 최근에야 실현 가능해졌다

```
[B: 상황 변화 감지] → [알림 전달] → [C: 재조정 옵션 생성] → [사용자 선택/합의] → [일정 업데이트]
```

**(2) 세그먼트별 차별화된 재조정 UX의 부재**

인터뷰 결과, 같은 "재조정" 기능이라도 세그먼트별로 전혀 다른 인터랙션 패턴이 필요하다:

| 세그먼트 | 재조정 모드 | 상세 UX |
|---------|-----------|---------|
| S1 솔로 (효율 추구형) | **선택형 (반자동)** | 대안 2~3개 제시 + 변경 이유 설명 + 사용자가 최종 선택 |
| S2 그룹 (유연한 계획형) | **합의형** | AI가 대안 3개 제시 + 각 옵션의 장단점(이동시간, 비용, 리뷰 점수) + 그룹원 실시간 투표 |
| S3 초보 (첫 자유여행) | **가이드형 (전자동)** | "이렇게 하세요" 수준의 안내 + 패키지 가이드 대체 경험 |

**(3) 재조정 결과의 투명성/설명 가능성 부재**

S1이 "왜 이렇게 바꿨지? 이유를 모르면 신뢰가 안 된다"고 명확히 지적했다. 자동 재조정의 투명성(변경 이유 설명)이 신뢰의 핵심 조건이다. 모든 재조정 결과는 "왜 이 대안을 추천하는가", "이 변경이 후속 일정에 어떤 영향을 미치는가"를 시각적으로 명시해야 한다.

이 수정의 핵심 이유: 기존의 "동적 재조정 엔진의 부재"는 기술적 솔루션에 가깝다. 인터뷰 결과는 근본원인이 기술만의 문제가 아니라, (1) 감지-재조정 연결 구조, (2) 사용자 유형별 인터랙션 설계, (3) 결과의 설명 가능성이라는 세 가지 차원의 복합적 부재임을 보여준다.

## 문제 간 구조적 관계

세 문제는 독립적이 아니라 **시간 순서상 연쇄적으로 발생하는 오류 연쇄 구조(Error Cascade)**를 형성한다.

```
[문제가설 1: 사후 인지 (B)]     [문제가설 2: 탐색 시간 소모 (A)]   [문제가설 3: 연쇄 일정 붕괴 (C)]
도착해서야 문제를 알게 됨  --->  대안을 찾느라 23.5분+ 허비  --->  하나를 바꾸면 전체가 무너짐
        |                              |                              |
        v                              v                              v
  감정 절벽 (3.6->1.6)           감정 바닥 유지 (1.9)            회복 지연 (2.5)
        |                              |                              |
        v                              v                              v
  Push 알림 시스템의 부재        통합 정보 허브의 부재          B연계 파이프라인 +
                                                              세그먼트별 UX +
                                                              투명성의 복합 부재
```

**인과 관계의 방향 (인터뷰로 재확인)**: 문제가설 1(사후 인지, B)이 오류 연쇄의 **트리거**다. 인터뷰 3명 전원이 이 인과 구조를 독립적으로 인식하고 있다.

- S1: "미리 알았으면 검색 자체를 안 해도 되니까" -- B 해결 시 A 발생 빈도 감소
- S2: "한 번의 사후 인지가 남은 여행 전체의 톤을 바꿔버린다" -- B의 감정적 파급 효과
- S3: "B가 문제 체인의 트리거 역할을 한다" -- 인과 구조를 명시적으로 표현

**피보팅 후 비즈니스 관점의 시사점**: 이 오류 연쇄는 "예방 -> 대응 -> 복구"라는 3단계 가치 제안 구조를 자연스럽게 형성한다. 인터뷰 결과, **예방(B)의 레버리지가 압도적**임이 검증되었다. B를 해결하면 A의 발생 빈도 자체가 감소하고, C의 트리거 빈도도 줄어든다. 따라서 MVP는 B(예방)를 1순위로, A(대응)를 2순위로, C(복구)를 3순위로 구현한다. 다만 돌발 상황을 100% 예방할 수 없으므로 3단계 모두를 해결해야 고객의 여행 경험이 끊김 없이 보호된다.

**[NEW] C의 해결은 B에 의존한다**: 인터뷰에서 C의 근본원인이 복합 구조로 수정됨에 따라, C의 첫 번째 근본원인이 "B와 연계된 파이프라인"이다. 이는 MVP 구현 순서에서도 B → A → C의 우선순위를 강화한다.

## 상위 공통 원인

3개 문제를 관통하는 상위 구조적 원인은 하나로 수렴한다:

**"여행 산업이 Before-trip(여행 전)에 최적화되어 있고, During-trip(여행 중)은 구조적 공백 상태"**

OTA 업계의 수익 모델(예약 수수료), 데이터 생태계(정적 데이터 중심), 기술 투자 방향(검색/예약 전환율 최적화) 모두가 Before-trip에 정렬되어 있다. During-trip에서 발생하는 고객의 고통 -- 사후 인지(문제 1), 검색 노동(문제 2), 연쇄 붕괴(문제 3) -- 은 이 산업 구조적 공백이 만들어낸 결과다.

이 공백이 바로 우리 MVP의 존재 이유다. "여행 중 실시간 일정 최적화 가이드 앱"은 During-trip이라는 미개척 카테고리를 정의하고 선점하는 제품이다.

**[인터뷰로 추가 검증된 사항]**:
- 3명 전원이 WTP(5,000~10,000원/여행건) 확인 -- During-trip 가치에 대한 지불 의사 존재
- 정보 정확도 90% 이상이 서비스 존재의 전제 조건 (S1 명시) -- 기존 도구와 같은 오류를 반복하면 차별성 없음
- 세그먼트별 "같은 문제, 다른 고통"이 존재 -- 단일 UX가 아닌 세그먼트별 경험 설계 필요
