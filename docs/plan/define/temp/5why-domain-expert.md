# 5WHY 분석 -- 인사이트 (도메인 전문가)

> **분석 관점**: 여행/OTA 도메인 전문성, 업계 관행, 규제 환경, 위치기반서비스(LBS) 생태계, 관광 데이터 인프라 측면
> **분석 대상**: 효율 추구형 솔로 트래블러(S1) 중심, 전체 세그먼트 교차 검증
> **데이터 기반**: 관찰 10회, 체험 10회, 심층 인터뷰 5명, 고객분석/유저저니맵 종합

---

## 현상문제 1: 여행 중 돌발 상황을 "도착 후에야" 인지하여 이동 시간 전체가 매몰 비용이 된다

> 인터뷰 4/5명이 목적지에 도착한 뒤에야 임시 휴업, 혼잡, 교통 중단 등을 인지한다. 관찰 6/10회, 체험 8/10회에서 "사후 인지" 패턴이 반복 확인되었다. 15분~75분의 이동 시간이 고스란히 매몰 비용으로 전환되며, 감정 곡선이 3.6에서 1.6으로 급락(-2.0)하는 직접적 트리거가 된다. 준혁(S1)은 "구글맵이 '영업 중'이라고 하는 걸 100% 믿을 수 없다는 게 제일 스트레스"라고 증언했고, 관찰1에서는 날씨 앱의 시간대별 예보를 확인했음에도 돌발 폭우를 사전에 감지하지 못했다.

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 여행자는 돌발 상황을 도착 후에야 인지하는가? | 이동 전에 목적지의 실시간 상태(영업 여부, 혼잡도, 날씨 영향)를 확인할 수 있는 통합 채널이 존재하지 않기 때문이다. 구글맵의 영업시간 정보는 정적 데이터(사업자 등록 기반)이며, 임시 휴업이나 실시간 변동을 반영하지 못한다. |
| **WHY 2** | 왜 목적지의 실시간 상태를 확인할 수 있는 통합 채널이 없는가? | OTA/여행 앱 업계가 "여행 전 예약(Booking)"에 수익 모델을 집중하고 있어, "여행 중 실행(During-trip)" 단계에 대한 투자가 구조적으로 부족하기 때문이다. 마이리얼트립, 클룩, 트리플 등 국내 주요 OTA는 액티비티/숙소 예약 수수료가 핵심 매출이며, 현지 실시간 정보 제공은 직접적 수익원이 아니라 우선순위에서 밀린다. |
| **WHY 3** | 왜 OTA 업계는 여행 중 실행 단계에 투자하지 않는가? | 실시간 영업 상태, 혼잡도, 날씨 영향 등의 데이터를 수집하려면 현지 POI(Point of Interest) 사업자와의 실시간 데이터 연동이 필요하나, 이는 국가/도시마다 데이터 표준이 다르고, 개별 소상공인의 참여율이 극히 낮아 데이터 공급망 구축 비용이 매우 높기 때문이다. Google조차 영업시간은 사업자 자가 신고(Google My Business)에 의존하며, 임시 휴업 갱신율은 전 세계 평균 30% 미만으로 추정된다. |
| **WHY 4** | 왜 현지 POI 실시간 데이터의 공급망 구축이 어려운가? | 관광 산업의 데이터 인프라가 "공급자 중심(사업자 등록 기반 정적 데이터)"으로 설계되어 있고, "소비자 중심(여행자가 필요한 시점에 필요한 실시간 데이터)" 관점의 데이터 표준과 유통 체계가 부재하기 때문이다. 한국관광공사의 Tour API도 관광지 기본 정보(주소, 소개문)는 제공하나 실시간 영업 상태나 혼잡도는 포함하지 않는다. 일본의 경우 Tabelog, 핫페퍼 등이 예약 기반 데이터를 보유하나 외부 API 공개가 제한적이다. |
| **WHY 5** | 왜 관광 데이터 인프라가 소비자 중심으로 전환되지 못하는가? | **근본원인**: 관광 데이터 생태계가 "예약 전환율" 중심의 비즈니스 모델에 종속되어 있어, 예약과 직접 연결되지 않는 실시간 현장 데이터(영업 상태, 혼잡도, 날씨 영향)는 수집-가공-유통의 경제적 인센티브가 부재하다. 데이터를 공급하는 현지 사업자 입장에서도 실시간 상태 갱신은 추가 업무 부담이며, 그에 상응하는 보상 체계가 없다. 결국 여행자는 구글맵의 부정확한 정적 데이터, 2~3년 전 블로그 후기, 현지어로 된 SNS 정보 등 신뢰도가 낮은 파편적 정보원에 의존할 수밖에 없는 구조가 고착된다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 | 근거 |
|------|---------|------|
| **사용자 심리적** | "계획대로 될 것"이라는 확증 편향으로 사전 검증 행동을 생략함. 특히 구글맵의 "영업 중" 표시를 무비판적으로 신뢰하는 플랫폼 의존 심리가 작동한다. | 인터뷰 4/5명 사후 인지. 준혁: "구글에는 영업 중이라고 떴는데." 관찰5: 리피터조차 정기휴일 미확인 |
| **제도/시스템적** | 관광 POI의 실시간 상태 데이터를 수집-유통하는 산업 표준이 부재하며, 각국 관광청의 데이터 공개(Open Data) 정책이 기본 정보에 국한되어 실시간 데이터를 포함하지 않는다. | Tour API의 한계, Google My Business의 낮은 갱신율, Tabelog/핫페퍼의 폐쇄적 API 정책 |
| **맥락적** | 해외 자유여행이라는 맥락에서 현지어 장벽, 시차에 의한 정보 접근 제약, 데이터 로밍 환경 등이 실시간 확인 행동의 마찰을 높인다. 현지 안내문이 현지어로만 제공되어 사전 정보 수집이 불가하다. | 관찰3: 영어/태국어 안내문만. 체험6: 스페인어 파업 안내. 인터뷰 4/5명 현지어 장벽 |
| **커뮤니케이션** | 돌발 상황 정보가 현지 사업자 -> 플랫폼 -> 여행자 경로로 전달되는 데 시간 지연(latency)이 발생하며, 대부분의 경우 전달 자체가 이루어지지 않는다. 여행자 간 실시간 정보 공유 채널도 구조화되어 있지 않다. | 관찰1: 날씨 앱 예보 변경이 반영되지 않음. 체험5: 정기휴일 정보가 온라인에 미갱신 |

**핵심 근본원인**: 관광 데이터 생태계가 "예약 전환" 중심으로 설계되어 있어, 여행자가 이동 중 실시간으로 필요로 하는 현장 상태 데이터(영업 여부, 혼잡도, 날씨 영향, 교통 상태)의 수집-유통 인센티브가 부재하고, 이로 인해 여행자는 부정확한 정적 데이터에 의존하여 "도착 후 사후 인지"라는 구조적 정보 비대칭에 놓이게 된다.

## 현상문제 2: 돌발 상황 발생 후 대안을 찾는 데 평균 23.5분이 소요되어 제한된 여행 시간이 "검색 노동"으로 소모된다

> 인터뷰 5/5명 전원이 대안 탐색에 30분~1시간을 소모하며, 체험 10회 평균 23.5분(최소 12분~최대 40분)이 대안 탐색에 소요된다. 3박 4일 여행에서 매일 반복되면 총 1.5~2시간, 즉 하루 일정 1개 분량이 "검색 노동"에 허비된다. 감정 곡선의 절대적 최저점(1.9/5.0)이 이 단계에서 발생한다. 구글맵-네이버 블로그-인스타그램을 오가는 "삼각 동선"(4/5명)이 보편적 패턴이며, 준혁(S1)은 "3박 4일에서 매일 30분씩 날리면 하루치 일정 하나가 통째로 사라지는 거예요"라고 표현했다.

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 대안 탐색에 평균 23.5분이나 소요되는가? | 여행자의 의사결정에 필요한 4가지 핵심 정보(거리/접근성, 리뷰/평판, 실시간 영업 상태, 시각적 정보/사진)가 각각 다른 앱에 분산되어 있어 "삼각 동선"(구글맵 -> 네이버 블로그 -> 인스타그램 -> 다시 구글맵)이 발생하기 때문이다. 후보 1건당 3개 앱을 오가며 5~8분이 소요되고, 3~5개 후보를 비교하면 20~40분에 이른다. |
| **WHY 2** | 왜 의사결정에 필요한 정보가 여러 앱에 분산되어 있는가? | 글로벌 지도 플랫폼(구글맵)은 위치/경로에 특화, OTA(트리플/마이리얼트립)는 예약에 특화, SNS(인스타그램)는 비주얼 콘텐츠에 특화되어 각 플랫폼이 자신의 핵심 가치에만 집중하는 "플랫폼 사일로(Platform Silo)" 현상 때문이다. 특히 한국인 여행자에게 결정적인 "한국어 리뷰"는 네이버 블로그에만 존재하는데, 네이버는 지도/위치 서비스가 해외에서 작동하지 않아 구글맵과의 교차 확인이 불가피하다. |
| **WHY 3** | 왜 플랫폼 사일로가 해소되지 않는가? | 각 플랫폼의 비즈니스 모델이 "자사 생태계 내 체류 시간 극대화"에 기반하고 있어, 경쟁 플랫폼과의 데이터 연동이 전략적으로 배제되기 때문이다. 구글은 구글맵 내에서 리뷰/예약까지 완결시키려 하고, 네이버는 블로그 트래픽을 네이버 내부로 유지하려 한다. 여행자 관점의 "정보 통합"은 어느 플랫폼에도 인센티브가 되지 않는다. 또한 한국어 리뷰 데이터는 네이버의 핵심 자산이므로 외부 플랫폼에 API로 공개하지 않는다. |
| **WHY 4** | 왜 "여행 중 실시간 맥락"을 반영한 통합 추천 서비스가 등장하지 않았는가? | "여행 중(During-trip)" 맥락에서의 추천은 (1) 실시간 위치, (2) 현재 날씨, (3) 시간대별 영업 상태, (4) 혼잡도, (5) 개인 취향, (6) 동행자 구성이라는 6가지 변수를 동시에 처리해야 하는 고도의 맥락 인식(Context-Awareness) 기술이 필요하다. 이는 최근 LLM/AI 기술이 상용화되기 전까지는 기술적으로 난이도가 높았고, 기존 OTA는 "검색 키워드 -> 결과 나열" 방식의 정적 추천에 머물러 있었기 때문이다. |
| **WHY 5** | 왜 기존 OTA와 지도 앱은 맥락 기반 추천으로 진화하지 못했는가? | **근본원인**: 기존 여행 서비스 생태계가 "여행 전 계획/예약"이라는 수익 모델에 최적화되어 있고, "여행 중 실행"은 수익화 모델이 불명확하여 투자 우선순위에서 체계적으로 배제되어 왔다. 동시에, 여행 중 실시간 맥락 추천에 필요한 다차원 데이터(위치 x 날씨 x 영업 상태 x 혼잡도 x 취향 x 동행 구성)의 통합 처리가 기술적으로 최근에서야 가능해졌다(LLM, 실시간 데이터 파이프라인). 그 결과 여행자는 "구글맵으로 거리 보고, 네이버로 리뷰 보고, 인스타로 사진 보고"라는 수작업 정보 조합을 반복하며, 이 과정이 23.5분의 "검색 노동"으로 귀결된다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 | 근거 |
|------|---------|------|
| **사용자 심리적** | "최선의 선택을 해야 한다"는 최적화 강박(Maximizer 성향)이 탐색 시간을 늘린다. 선택지가 20개 이상이면 오히려 결정 마비(Decision Paralysis)가 발생하며, "더 좋은 곳이 있을 수도"라는 FOMO가 탐색을 중단하지 못하게 만든다. | 인터뷰 5/5명 "이게 최선인지 모르겠다". 준혁: "선택지가 20개면 오히려 못 고르거든요." 솔로 여행자 3/5명 상의 대상 부재 |
| **제도/시스템적** | 여행 정보 플랫폼 간 데이터 상호운용성(Interoperability)이 없다. 구글맵-네이버-인스타그램 각각이 폐쇄적 데이터 정책을 유지하며, 여행자 관점의 크로스 플랫폼 정보 통합을 지원하는 산업 표준이나 오픈 API가 부재하다. | 소윤: "인스타에서 찾고, 블로그에서 확인하고, 구글맵에서 위치 잡고... 삼각 동선이 진짜 피곤해요." 인터뷰 4/5명 탭 5~6개 전환 |
| **맥락적** | 해외 여행 중이라는 맥락에서 현지어 리뷰 접근 불가(4/5명), 로밍 환경에서의 느린 데이터 속도, 이동 중 스마트폰 조작의 물리적 제약(관찰1: 젖은 손, 관찰3: 33도 폭염 속 35분 검색)이 탐색 마찰을 가중시킨다. | 관찰5: 일본어 리뷰 수동 번역. 체험4: 프랑스어 리뷰 번역 품질 저하. 관찰3: 방콕 33도에서 길에서 35분 검색 |
| **커뮤니케이션** | 여행 정보가 "일방향 콘텐츠"(블로그 후기, SNS 게시물) 형태로만 유통되며, 여행자의 현재 맥락(위치, 시간, 상황)에 반응하는 "양방향 대화형" 정보 제공 체계가 부재한다. 블로그 정보는 2~3년 전 시점의 정적 스냅샷이며 현재 상태를 반영하지 못한다. | 체험3: 2022년 블로그 정보 믿고 예약 안 한 대영박물관 입장 거부. 시우: "블로그에서 본 코스를 그대로 따라갈 수 있을 줄 알았는데, 현실은 변수가 너무 많아요." |

**핵심 근본원인**: 여행 정보 생태계가 "플랫폼 사일로"와 "일방향 정적 콘텐츠" 구조에 고착되어 있어, 여행자의 실시간 맥락(위치 x 시간 x 날씨 x 취향)을 이해하고 통합된 대안을 즉시 제시하는 서비스가 부재하다. 이로 인해 여행자는 3개 이상의 앱을 수작업으로 오가며 정보를 조합하는 "삼각 동선"을 반복하게 되고, 이 과정이 매 돌발 상황마다 23.5분의 검색 노동으로 귀결된다.

## 현상문제 3: 하나의 일정 변경이 전체 여행에 연쇄적으로 파급되어 수동 재조정의 인지적 부담과 피로가 누적된다

> 인터뷰 4/5명이 하나의 일정 변경이 이후 전체 일정에 연쇄 영향을 미친다고 응답했다. 체험 8/10회에서 일정 수동 재조정의 번거로움이 지적되었으며, 1곳 변경 시 평균 2~3곳의 연쇄 변경이 발생한다. 소윤(S1)은 "하나가 밀리니까 연쇄적으로 다 밀려서, 결국 마지막 날에 원래 첫째 날 일정을 하게 됐어요. 이걸 머릿속으로 퍼즐 맞추듯이 재조정하는 게 진짜 피곤했어요"라고 증언했다. 관찰4에서는 팀랩 예약금 11,400엔이 손실되는 금전적 피해도 발생했다.

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 하나의 일정 변경이 전체 여행에 연쇄적으로 파급되는가? | 자유여행 일정은 "장소 A(10:00~11:30) -> 이동(30분) -> 장소 B(12:00~13:30) -> ..."처럼 시간적 의존성과 공간적 의존성(동선)이 복합적으로 얽혀 있기 때문이다. 하나의 장소가 변경되면 이동 시간, 영업 시간 제약, 예약 시간 등 연결된 모든 변수가 동시에 영향을 받는다. |
| **WHY 2** | 왜 연쇄 변경의 영향을 빠르게 처리할 수 없는가? | 기존 여행 일정 도구(트리플, 노션, 구글맵 저장 목록)가 "정적 리스트" 형태로 설계되어 있어, 장소 간 시간적/공간적 의존성을 데이터로 관리하지 않기 때문이다. 하나를 수정하면 나머지를 수동으로 재계산해야 한다. 트리플의 일정 기능은 장소를 시간대에 배치할 수 있지만, 한 곳을 바꿨을 때 나머지를 자동 재배치하는 "동적 일정 엔진"은 제공하지 않는다. |
| **WHY 3** | 왜 기존 여행 앱은 동적 일정 재조정 기능을 제공하지 않는가? | 동적 일정 재조정은 (1) 실시간 이동 시간 계산(교통 API), (2) 영업 시간 데이터, (3) 예약 제약 조건, (4) 개인 선호(식사 시간, 체력), (5) 날씨 변수를 동시에 고려하는 다제약 최적화(Multi-Constraint Optimization) 문제이다. 이는 단순 일정 관리 앱의 기술 범위를 넘어서며, 물류/배송 경로 최적화 수준의 알고리즘 역량이 필요하다. 기존 OTA는 이러한 기술 투자보다 예약 전환율 최적화에 리소스를 집중해왔다. |
| **WHY 4** | 왜 여행 영역에서 경로/일정 최적화 기술이 적용되지 않았는가? | 물류/배송 분야의 경로 최적화는 "차량-도로-거점" 같은 정형화된 제약 조건 하에서 작동하지만, 여행 일정은 "개인 취향", "감정 상태", "즉흥성", "동행자 관계 역학" 같은 비정형적/감성적 변수가 개입하여 순수한 알고리즘적 최적화가 어렵기 때문이다. 또한 여행 일정의 "최적"이란 효율만이 아니라 "경험의 질", "여유", "뜻밖의 발견"까지 포함하는 다차원적 개념이어서, 정량적 최적화만으로는 여행자 만족을 보장할 수 없었다. |
| **WHY 5** | 왜 비정형적 변수를 포함한 여행 일정 최적화가 해결되지 못했는가? | **근본원인**: 여행 일정의 본질이 "효율적 경로"가 아니라 "맥락적 경험의 연속"이어서, 시간/공간 최적화만으로는 해결할 수 없는 복합적 문제였다. 기존 기술로는 정량적 제약(거리, 시간, 영업 여부)과 정성적 맥락(취향, 기분, 동행자 관계, 즉흥성)을 동시에 처리하는 것이 불가능했다. 그러나 이제 LLM 기반 AI가 자연어로 사용자의 맥락을 이해하고, 정량적 제약 조건과 정성적 선호를 동시에 반영하여 일정을 재구성할 수 있는 기술적 기반이 마련되었다. 핵심은 기술의 부재가 아니라, "여행 중 동적 재조정"이라는 문제 정의 자체가 업계에서 간과되어 왔다는 점이다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 | 근거 |
|------|---------|------|
| **사용자 심리적** | 여행자는 일정을 "한 번 세우면 완성"이라고 인식하며, 실시간 재조정이 필요하다는 전제를 갖고 일정을 설계하지 않는다. 백업 플랜(Plan B)을 사전에 준비하는 여행자는 극소수이며, 돌발 상황이 발생하면 "머릿속 퍼즐 맞추기"라는 높은 인지 부하가 걸리면서 피로와 스트레스가 급증한다. | 소윤: "퍼즐 맞추듯이 재조정하는 게 진짜 피곤." 인터뷰 4/5명 수동 재계산 부담. 관찰1: 노션 일정 수동 수정 |
| **제도/시스템적** | 여행 일정 관리 도구가 "정적 리스트" 구조에 머물러 있어, 장소 간 시간/공간 의존성(dependency graph)을 데이터로 표현하지 못한다. 물류 업계에서 일상적인 실시간 경로 재최적화(Vehicle Routing Problem) 기술이 여행 일정 도메인에는 적용되지 않고 있다. | 관찰3: 방콕 짐 톰슨 하우스 포기 -> 연쇄 일정 변경. 관찰4: 팀랩 예약금 11,400엔 손실. 체험1: 노션 일정 수정에 10분 추가 |
| **맥락적** | 여행 일정의 의존성은 단순 시간 순서가 아니라 "예약 유무", "영업시간 제약", "날씨 의존도(실내/실외)", "식사 타이밍", "체력 소모도", "동행자 컨디션" 등 다차원 변수가 얽혀 있어, 머릿속 재계산이 구조적으로 한계에 부딪힌다. 특히 아이 동반(S4)이나 복수 일정(3일 이상)에서 복잡도가 기하급수적으로 증가한다. | 소윤: "마지막 날에 원래 첫째 날 일정을 하게 됐어요." 관찰4: 팀랩 이전 -> 아이 낮잠 -> 저녁 식당 예약 -> 연쇄 붕괴 |
| **커뮤니케이션** | 일정 변경 사항이 동행자에게 실시간으로 공유되지 않아, 동행자 간 일정 인식의 불일치가 추가적인 합의 비용을 발생시킨다. 커플/그룹 여행에서 한쪽이 재조정하면 다른 쪽은 "원래 일정"을 기억하고 있어 혼선이 발생한다. | 하은: "제가 검색하는 20분 동안 남자친구는 옆에서 폰 게임을 해요." 관찰2: A는 네이버, B는 인스타로 각자 검색 -> 합의 지연 |

**핵심 근본원인**: 여행 일정이 시간적/공간적/맥락적 의존성이 복합적으로 얽힌 "다제약 동적 시스템"임에도 불구하고, 기존 여행 도구는 이를 "정적 리스트"로만 관리한다. 업계가 "여행 전 계획" 단계에 집중하면서 "여행 중 동적 재조정"이라는 문제 자체를 정의하지 않았고, 정량적 제약(거리, 시간)과 정성적 맥락(취향, 감정, 동행자)을 동시에 처리하는 기술적 해법도 최근에서야 현실화되었다. 그 결과 여행자는 돌발 상황마다 "머릿속 퍼즐 맞추기"를 반복하며, 이 인지 부하가 여행 전체의 피로와 만족도 하락의 구조적 원인이 된다.

## 종합: 핵심 근본원인 3가지와 도메인 관점의 시사점

| # | 현상문제 | 핵심 근본원인 | 도메인 관점의 구조적 배경 |
|---|---------|-------------|----------------------|
| 1 | 도착 후 사후 인지 | **관광 데이터 생태계의 "예약 전환" 편향** -- 실시간 현장 데이터의 수집-유통 인센티브 부재 | OTA 수익 모델이 예약 수수료에 집중, 실시간 POI 상태 데이터의 경제적 가치가 산업 내에서 인식되지 않음 |
| 2 | 23.5분의 검색 노동 | **플랫폼 사일로와 정적 콘텐츠 구조** -- 여행자 맥락을 이해하는 통합 추천 부재 | 구글/네이버/인스타 각각의 폐쇄적 데이터 정책, 한국어 리뷰의 네이버 독점, "여행 중" 추천의 수익 모델 부재 |
| 3 | 연쇄적 일정 붕괴 | **"여행 중 동적 재조정" 문제의 미정의** -- 정량+정성 다제약 최적화의 기술적 난이도 | 업계가 "계획/예약" 단계에만 집중하여 "실행/재조정" 단계의 문제를 정의조차 하지 않았음 |

### 도메인 전문가 소견

세 가지 근본원인은 모두 **"여행 산업이 Before-trip(여행 전)에 최적화되어 있고, During-trip(여행 중)은 구조적 공백 상태"**라는 하나의 상위 원인으로 수렴한다. 이는 OTA 업계의 수익 모델(예약 수수료), 데이터 생태계(정적 데이터 중심), 기술 투자 방향(검색/예약 전환율 최적화) 모두가 Before-trip에 정렬되어 있기 때문이다.

여행자 관점에서 "여행의 질"을 결정하는 것은 사전 계획이 아니라 **현장에서의 실행과 대응**이다. 인터뷰 5/5명 전원이 "결과적으로 나쁘지 않았다 / 오히려 좋았다. 하지만 대안 찾는 시간이 아까웠다"라고 응답한 것은, 문제의 본질이 "좋은 대안의 부재"가 아니라 **"좋은 대안에 도달하는 과정의 비효율"**에 있음을 방증한다.

이는 곧, 이 시장에서 During-trip 단계에 진입하는 서비스가 **여행자의 시간을 되돌려주는(Time-saving)** 명확한 가치 제안을 할 수 있으며, "30분을 3분으로"라는 정량적 가치가 유료 전환의 강력한 트리거가 될 수 있음을 의미한다. 소윤의 "하나를 고르면 나머지 일정이 알아서 재배열되면... 그건 진짜 돈 내고 쓸 것 같아요"는 이 시장 기회의 직접적 근거이다.

규제 관점에서는 위치정보법(위치정보의 보호 및 이용 등에 관한 법률)에 따른 실시간 위치 수집 동의, 개인정보 보호법에 따른 취향/행동 데이터 처리 기준, 각국 관광 데이터의 크로스보더 활용 시 데이터 이전 규정(GDPR 등)을 사전에 설계에 반영해야 한다.
