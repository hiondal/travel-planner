## 수렴 기준

- **현상문제 유사도 판단 기준**: 현상문제가 묘사하는 고객 행동과 고통의 핵심 장면(scene)이 동일하거나 실질적으로 같은 경험을 다른 관점에서 표현한 것인지를 기준으로 판단한다. 예를 들어, "대안 탐색에 23.5분 소요"와 "여러 앱을 오가며 23.5분 소모"는 동일한 고객 경험의 다른 표현이므로 같은 그룹으로 분류한다.
- **근본원인 유사도 판단 기준**: 5WHY 분석의 최종 도달점(WHY5)이 가리키는 구조적 결함의 방향성이 동일한지를 기준으로 판단한다. 각 팀원의 전문 관점(비즈니스, UX, 아키텍처, AI, 백엔드, 프론트엔드, QA, DevOps, 도메인)에 따라 표현은 다르지만, 가리키는 구조적 공백이 같으면 동일 그룹으로 통합한다.
- **그룹핑 원칙**: 솔루션이 아닌 고객의 문제에 집중한다. 각 관점의 근본원인은 하나의 그룹 안에서 다층적 근본원인의 풍부한 근거로 활용한다.

## 그룹핑 결과

27개 현상문제는 **3개의 핵심 그룹**으로 수렴된다. 9명의 팀원이 각자의 전문 관점에서 분석했음에도 불구하고, 고객이 겪는 핵심 고통은 동일한 3가지로 귀결되었다. 이는 이 문제들이 개별 관점의 편향이 아닌, 실제 고객 경험의 구조적 문제임을 강하게 시사한다.

### 그룹 A: 돌발 상황 발생 시 대안 탐색에 평균 23.5분이 소요되어, 여행 시간이 "검색 노동"으로 직접 손실된다

- **포함 팀원/문제**:
  - 피오 #1 (비즈니스 관점: 카테고리 공백, 선점 기회)
  - 도그냥 #4 (서비스 흐름 관점: 통합 정보 허브 부재)
  - 인사이트 #8 (도메인 관점: 플랫폼 사일로, OTA 수익 모델 편향)
  - 아키 #10 (아키텍처 관점: 이기종 데이터 통합 아키텍처 부재)
  - 마법사 #13 (AI 관점: 맥락 인식 추천 시스템 부재)
  - 데브-백 #16 (백엔드 관점: 저지연 통합 집계 레이어 부재)
  - 데브-프론트 #19 (프론트엔드 관점: 스트레스 맥락 UX 설계 미스매치)
  - 가디언 #23 (QA 관점: 통합 정보 인터페이스 부재, 품질 결함)
  - 파이프 #26 (DevOps 관점: API 오케스트레이션 레이어 부재)

- **통합 현상문제**: 여행 중 돌발 상황(휴업, 날씨 급변, 혼잡, 교통 장애)이 발생했을 때, 여행자가 대안 장소를 찾고 선택하기까지 평균 23.5분(최소 12분~최대 40분)이 소요된다. 거리, 한국어 리뷰, 실시간 영업 상태, 비주얼이라는 4가지 판단 기준이 구글맵, 네이버 블로그, 인스타그램에 각각 흩어져 있어 "삼각 동선"을 반복해야 하며, 3박 4일 여행 기준 하루 일정 1개 분량(약 2시간)이 "검색 노동"에 소모된다. (인터뷰 5/5명, 체험 10/10회, 심각도 5.0/5)

- **다층적 근본원인**:

  | 측면 | 근본 원인 | 중요도 |
  |------|----------|:------:|
  | 사용자 심리적 | "최선의 선택을 해야 한다"는 완벽주의적 탐색 행동(Maximizer 성향). 혼자 여행 시 의사결정을 검증해줄 상대가 없어 후보를 반복 비교하며 확신을 얻으려 함. 선택지 20개 이상이면 결정 마비(choice paralysis) 발생 (인터뷰 5/5명 "이게 최선인지 모르겠다") | ★★★★ |
  | 제도/시스템적 | 여행 정보 플랫폼 간 데이터 상호운용성(interoperability)이 없다. 구글맵(위치/거리), 네이버(한국어 리뷰), 인스타그램(비주얼)이 각각 폐쇄적 데이터 사일로를 유지하며, 여행자의 현재 맥락(위치, 시간, 날씨, 취향, 영업 상태)을 통합하여 대안을 제시하는 서비스가 시장에 부재. 기존 OTA는 "예약 수수료" 수익 모델에 집중하여 "여행 중 실행" 단계 투자 인센티브가 구조적으로 부재 | ★★★★★ |
  | 맥락적 | 돌발 상황은 "지금, 여기서, 즉시"라는 시간-공간-긴급도의 삼중 제약 하에서 발생. 배고픔, 더위(관찰3: 33도에서 35분 검색), 비(관찰1: 젖은 손 터치), 피로 등 물리적 스트레스가 동시에 발생하여 인지 자원이 고갈된 상태에서 복잡한 탐색 수행. 기존 도구는 정적 정보만 제공하여 맥락 불일치 | ★★★★ |
  | 커뮤니케이션 | 현지어 리뷰(일본어, 태국어, 프랑스어) 접근 불가(4/5명). 블로그 정보의 시차 문제(2~3년 전 정보). 기존 서비스는 "선택지 20개 나열"만 하고 "왜 이것을 추천하는가"의 맥락을 제공하지 않아, 여행자가 추천 이유 없이 스스로 판단해야 함. 동행자 간 서로 다른 앱/기준으로 검색하여 합의 비용 추가 발생 | ★★★ |

- **핵심 근본원인**: **"여행 중 실행 단계"를 위한 실시간 맥락 기반 통합 정보 허브의 구조적 부재** -- 거리, 한국어 리뷰, 실시간 영업 상태, 비주얼을 여행자의 현재 맥락(위치, 시간, 날씨, 취향)에 맞춰 하나의 화면에서 비교 판단할 수 있는 서비스가 없다. 기존 여행 플랫폼들의 수익 모델이 "여행 전 예약"에 집중되어 있어, 이 카테고리는 시장에서 구조적 공백 상태이다.

  - 파이프 #25 (DevOps 관점: POI 실시간 상태 수집/발행 파이프라인 부재)

- **통합 현상문제**: 여행자가 임시 휴업, 대기 행렬, 교통 중단, 날씨 급변 등의 돌발 상황을 "목적지에 도착한 후에야" 인지한다. 이동에 투자한 15~75분이 매몰 비용이 되고, 유저 저니 맵 감정 곡선에서 2단계(3.6점)에서 3단계(1.6점)로 -2.0점의 최대 감정 낙차("감정 절벽")가 발생한다. 구글맵의 "영업 중" 표시를 신뢰했으나 실제와 다른 경험이 반복되면서, 디지털 도구 전반에 대한 불신이 축적된다. (인터뷰 4/5명, 관찰 6/10회, 체험 8/10회, 심각도 4.8/5)

- **다층적 근본원인**:

  | 측면 | 근본 원인 | 중요도 |
  |------|----------|:------:|
  | 사용자 심리적 | "계획대로 될 것"이라는 확증 편향(confirmation bias)과 플랫폼 권위에 대한 과신. 구글맵 "영업 중" 표시를 무비판적으로 신뢰하고 추가 검증을 생략. 국내 경험에서 형성된 "앱 정보 = 현실" 신뢰 모델이 해외에서 반복적으로 깨지면서 도구 불신 축적. 높은 만족감(3.6) 직후의 갑작스러운 추락이 충격을 극대화 | ★★★★ |
  | 제도/시스템적 | 현재 모든 여행 도구가 사용자가 직접 검색해야 정보를 제공하는 Pull 모델. 여행자의 일정에 등록된 장소를 실시간으로 모니터링하여 이동 전에 "이 장소는 현재 임시 휴업입니다"라고 선제적으로 알리는 Push 모델이 부재. 구글맵 영업시간은 사업자 자발적 등록에 의존(임시 휴업 갱신율 30% 미만 추정). 관광 데이터 생태계가 "예약 전환" 수익에 종속되어 실시간 현장 데이터 수집/유통의 경제적 인센티브가 부재 | ★★★★★ |
  | 맥락적 | 해외 여행에서 현지 정보는 현지어(일본어, 태국어 등)로 제공되어 외국인 여행자의 사전 접근이 구조적으로 불가능. 임시 휴업, 국가 행사, 교통 점검 등은 현지어 공식 채널에서만 안내(관찰3: 태국어/영어 안내문만, 체험6: 스페인어 파업 안내문). 전화 확인도 언어/요금 문제로 비현실적 | ★★★★ |
  | 커뮤니케이션 | 돌발 상황 정보의 전달 체계가 "현장 안내문"에 의존하여, 정보가 도착 시점에만 전달되는 구조적 지연 발생. POI 운영자(점포)는 임시 휴업을 자기 SNS 채널에만 발행하며, 이 정보가 여행자가 사용하는 서비스로 전파되는 채널이 없음. 정보 생산자와 소비자 사이의 커뮤니케이션 레이어 단절 | ★★★ |

- **핵심 근본원인**: **"여행 일정 기반 선제적 리스크 감지/알림(Push) 시스템"의 부재** -- 현재 모든 여행 도구는 사용자가 직접 정보를 찾아야 하는 Pull 모델이다. 여행자의 일정에 등록된 장소를 실시간으로 모니터링하고, 이동 전에 상태 변화를 선제적으로 알려주는 Push 모델이 없기 때문에, 여행자는 매번 도착해서야 당황하게 된다. 이는 관광 데이터 생태계가 "예약 전환" 수익에 종속되어 실시간 현장 데이터의 수집-유통 인센티브가 구조적으로 부재하기 때문이다.

### 그룹 C: 하나의 일정 변경이 전체 여행을 연쇄적으로 붕괴시키며, 여행자가 감정적으로 가장 취약한 순간에 인지적으로 가장 부담이 큰 수동 재계산을 수행해야 한다

- **포함 팀원/문제**:
  - 피오 #3 (비즈니스 관점: 손실 회피 심리, "인지적 해방"이 유료 전환 트리거)
  - 도그냥 #6 (서비스 흐름 관점: 감정-인지 미스매치, 동적 일정 엔진 부재)
  - 인사이트 #9 (도메인 관점: 정량+정성 다제약 최적화의 기술적 난이도, 문제 미정의)
  - 아키 #12 (아키텍처 관점: 제약 기반 동적 스케줄링 엔진, CSP/TSP 복잡도)
  - 마법사 #15 (AI 관점: 의존성 그래프 + LLM 기반 동적 재최적화)
  - 데브-백 #18 (백엔드 관점: 실시간 최적화 연산과 저지연 응답의 동시 달성)
  - 데브-프론트 #21 (프론트엔드 관점: 정적 문서에서 동적 스케줄링 인터페이스로 전환)
  - 가디언 #24 (QA 관점: 의존 관계 관리 부재로 오류 연쇄, Working Memory 한계)
  - 파이프 #27 (DevOps 관점: 이벤트 기반 워크플로우 부재, 정적 YAML 같은 일정 관리)

- **통합 현상문제**: 돌발 상황으로 하나의 일정이 변경되면, 이동 동선, 영업시간, 예약 시간, 체류시간의 의존성 때문에 평균 2~3곳의 추가 변경이 연쇄적으로 발생한다("도미노 효과"). 여행자는 이를 머릿속이나 메모장에서 수동으로 재계산해야 하며, 추가 5~15분이 소요된다. 이 재계산은 감정적으로 가장 취약한 시점(감정 레벨 1.6~1.9)에 인지적으로 가장 복잡한 작업(다변수 최적화)을 수행해야 하는 "감정-인지 미스매치"를 유발한다. 심한 경우 당일 일정을 넘어 여행 전체 일정이 재편된다. (인터뷰 4/5명, 체험 8/10회, 심각도 4.4/5. 소윤: "마지막 날에 원래 첫째 날 일정을 하게 됐어요")

- **다층적 근본원인**:

  | 측면 | 근본 원인 | 중요도 |
  |------|----------|:------:|
  | 사용자 심리적 | 일정 변경 시 "포기해야 하는 장소"에 대한 손실 회피(loss aversion) 심리. 감정적으로 가장 취약한 시점(1.6~1.9)에 인지적으로 가장 복잡한 작업을 수행해야 하는 "감정-인지 미스매치". "이게 최선인가?"라는 확신 부족이 만성화(5/5명 전원). "전부 망했다"는 인지적 과장(catastrophizing). 촘촘한 일정이 오히려 유연성을 극도로 낮추는 역설. 초보 여행자는 "자유여행이 나에게 맞지 않나"라는 자기 의심까지 유발 | ★★★★★ |
  | 제도/시스템적 | 현재의 일정 관리 도구(노션, 트리플, 메모 앱)가 "정적 리스트" 구조에 머물러 있어, 장소 간 시간/위치/영업시간의 의존성을 데이터로 관리하지 못함. 하나를 바꿔도 나머지가 자동 조정되지 않는 구조. 동적 일정 재조정은 다제약 최적화(multi-constraint optimization) 문제로, 물류/배송 수준의 알고리즘 역량이 필요하나 여행 도메인에는 적용 전무 | ★★★★★ |
  | 맥락적 | 여행 일정의 의존성이 단순 시간 순서가 아니라 "영업시간 제약, 날씨 의존도(실내/실외), 예약 유무, 식사 타이밍, 체력 소모도, 동행자 컨디션" 등 다차원 변수가 얽혀 있어 머릿속 재계산이 구조적 한계. 해외라는 맥락이 교통/영업 관행/예약 정책의 불확실성을 가중. 복수 일(3일 이상)에 걸친 도미노 효과는 단일 일 최적화로 해결 불가 | ★★★★ |
  | 커뮤니케이션 | 일정 변경의 연쇄 영향이 여행자에게 투명하게 전달되지 않음. "이 장소를 바꾸면 다음 3개 일정에 이런 영향이 있습니다"라는 시각적 피드백 부재. 동행자 간 변경 사항 공유가 구두나 수기 메모에 의존하여 혼선 발생. 재조정 결과의 변경 전/후 비교, 이유, 포기 장소의 대안 날짜 제안 등 투명한 설명이 없음 | ★★★ |

- **핵심 근본원인**: **일정 항목 간 시간적/공간적/맥락적 의존 관계를 인식하고, 변경 발생 시 나머지를 자동으로 재최적화하는 "동적 일정 재조정 엔진"의 부재** -- 여행 일정은 시간, 위치, 영업시간, 예약, 날씨 등 다수의 의존 관계로 연결된 복합 시스템이나, 현재의 모든 일정 도구는 이를 독립적인 항목의 나열(정적 목록)로만 관리한다. "여행 중 동적 재조정"이라는 문제 정의 자체가 업계에서 간과되어 왔으며, 정량적 제약(거리, 시간)과 정성적 맥락(취향, 감정, 동행자)을 동시에 처리하는 기술적 해법도 최근에야 실현 가능해졌다.

## 3개 그룹의 구조적 관계

세 그룹은 독립적이 아니라 **시간 순서상 연쇄적으로 발생**하며, **하나의 오류 연쇄 구조(Error Cascade)**를 형성한다.

```
[그룹 B: 사후 인지]               [그룹 A: 탐색 시간 소모]           [그룹 C: 연쇄 일정 붕괴]
도착해서야 문제를 알게 됨    --->   대안을 찾느라 23.5분 허비    --->   하나를 바꾸면 전체가 무너짐
        |                               |                               |
        v                               v                               v
  감정 절벽 (3.6->1.6)            감정 바닥 유지 (1.9)             회복 지연 (2.5)
        |                               |                               |
        v                               v                               v
  [핵심 근본원인]                 [핵심 근본원인]                  [핵심 근본원인]
  선제적 Push 알림               실시간 맥락 기반                 동적 일정 재조정
  시스템의 부재                  통합 정보 허브의 부재             엔진의 부재
```

**상위 공통 원인**: "여행 산업이 Before-trip(여행 전)에 최적화되어 있고, During-trip(여행 중)은 구조적 공백 상태"라는 하나의 산업 구조적 원인으로 수렴한다. OTA 업계의 수익 모델(예약 수수료), 데이터 생태계(정적 데이터 중심), 기술 투자 방향(검색/예약 전환율 최적화) 모두가 Before-trip에 정렬되어 있다.
