# 5WHY 분석 -- 아키 (소프트웨어 아키텍트)

## 분석 관점

소프트웨어 아키텍트로서 고객이 겪는 문제를 **기술 아키텍처 관점**에서 분석한다. 기존 여행 정보 시스템의 구조적 한계, 데이터 통합의 부재, 실시간 처리 아키텍처의 결핍이 고객 경험의 근본적인 병목을 만들어내고 있다는 시각에서 접근한다.

---

## 현상문제 1: 여행 중 대안을 찾는 데 평균 23.5분이 소요되며, 이 시간이 여행 경험을 직접 잠식한다

> 인터뷰 5/5명 전원, 체험 10/10회에서 확인. "3박 4일 여행에서 매일 30분씩 날리면 하루치 일정 하나가 통째로 사라지는 거예요." (준혁, S1)

### 5WHY 분석

| WHY | 질문 | 응답 |
|-----|------|------|
| **WHY 1** | 왜 대안을 찾는 데 23.5분이나 걸리는가? | 여행자가 거리, 리뷰, 영업 상태, 사진이라는 4가지 판단 기준을 확인하기 위해 구글맵, 네이버, 인스타그램 등 3개 이상의 앱을 번갈아 사용하는 "삼각 동선"을 반복하기 때문이다. (인터뷰 4/5명, 관찰 7/10회) |
| **WHY 2** | 왜 여러 앱을 오가야 하는가? | 각 앱이 제공하는 정보가 부분적이고 파편화되어 있기 때문이다. 구글맵은 위치와 거리에 강하지만 한국어 리뷰가 부족하고, 네이버는 한국어 리뷰가 있지만 실시간 영업 상태가 없으며, 인스타그램은 비주얼 정보만 제공하고 위치 정확도가 떨어진다. |
| **WHY 3** | 왜 하나의 플랫폼에서 통합 정보를 제공하지 못하는가? | 기존 여행 플랫폼들이 **이기종 데이터 소스 통합 아키텍처**를 갖추지 못했기 때문이다. 위치 데이터(Google Places API), 리뷰 데이터(네이버/Tabelog), 비주얼 데이터(인스타그램), 실시간 영업 데이터(각 점포 POS/예약 시스템)가 서로 다른 데이터 스키마, 접근 프로토콜, 갱신 주기를 가지고 있어 단일 파이프라인으로 통합하기 어렵다. |
| **WHY 4** | 왜 이기종 데이터 통합이 어려운가? | 여행 도메인의 데이터가 구조적으로 **높은 이질성과 낮은 표준화** 문제를 가지기 때문이다. (1) 각국 POI(Point of Interest) 데이터의 스키마가 다르고, (2) 영업 상태의 갱신 빈도가 플랫폼마다 다르며(구글은 크라우드소싱 기반으로 수일~수주 지연), (3) 리뷰 데이터의 언어/문화적 맥락이 달라 단순 번역으로는 품질이 보장되지 않고, (4) 실시간 혼잡도 데이터는 대부분의 소규모 점포에서 아예 제공되지 않는다. |
| **WHY 5** | 왜 여행 도메인에서 데이터 표준화와 실시간 통합이 이루어지지 않았는가? | 여행 중 실시간 의사결정 지원이라는 문제 영역 자체가 기존 시장에서 **비즈니스 가치가 검증되지 않은 영역**이었기 때문이다. 기존 여행 플랫폼들은 "여행 전 계획" 단계에 집중하여 수익 모델(예약 수수료, 광고)을 구축했고, "여행 중 실행" 단계의 실시간 데이터 통합은 기술적 난이도(다국어 NLP, 실시간 스트리밍, 위치 기반 컨텍스트 엔진)가 높은 반면 수익 모델이 불명확하여 투자 우선순위에서 밀려왔다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| **사용자 심리적** | 여행자는 "잘못된 선택의 리스크"를 혼자 감당해야 하므로, 확신을 얻기 위해 여러 소스를 교차 검증하는 행동을 반복한다. 정보를 더 모을수록 더 나은 결정을 할 수 있다는 "정보 과잉 수집 편향"이 탐색 시간을 늘린다. (인터뷰 5/5명 "이게 최선인지 모르겠다") |
| **제도/시스템적** | 글로벌 POI 데이터에 대한 오픈 표준(스키마, API 규격, 갱신 프로토콜)이 부재하다. Google Places, Foursquare, Yelp, Tabelog, 네이버 플레이스 등이 각각 폐쇄적인 데이터 사일로를 구축하고 있으며, 데이터 공유를 위한 산업 표준 프로토콜이 없다. 이로 인해 통합 서비스 구축의 기술적/법적 장벽이 높다. |
| **맥락적** | 여행 중이라는 맥락이 탐색을 어렵게 만든다. (1) 모바일 환경의 작은 화면에서 다중 탭을 전환하는 물리적 불편, (2) 비/더위 등 열악한 물리적 환경(관찰1: 젖은 손 터치, 관찰3: 33도에서 35분 검색), (3) 배고픔/피로 등 신체 상태가 인지 능력을 저하시킴, (4) 현지 네트워크 환경(로밍/Wi-Fi 불안정)이 검색 속도를 늦춘다. |
| **커뮤니케이션** | 동행자가 있는 경우 각자 다른 앱에서 다른 기준으로 검색하여 정보가 공유되지 않는다. A는 네이버+리뷰점수, B는 인스타+비주얼을 기준으로 판단하며, 이 불일치가 합의 시간을 추가로 20분 이상 늘린다. (관찰2: 28분간 비효율적 논쟁) 솔로 여행자는 반대로 검증해줄 상대가 없어 "이게 맞나?" 반복 고민에 시간을 소모한다. |

### 핵심 근본원인

> **여행 중 실시간 의사결정에 필요한 이기종 데이터(위치, 리뷰, 영업 상태, 혼잡도, 날씨)를 단일 컨텍스트로 통합하는 데이터 통합 아키텍처가 부재하여, 여행자가 인간 미들웨어 역할을 수동으로 수행하고 있다.**

## 현상문제 2: 방문 예정 장소의 휴업/혼잡/교통 장애를 도착 후에야 인지하여 이동 시간이 매몰 비용이 된다

> 인터뷰 4/5명이 도착 후 인지. 관찰 6/10회, 체험 8/10회에서 확인. "구글맵이 '영업 중'이라고 하는 걸 100% 믿을 수 없다는 게 제일 스트레스예요." (준혁, S1)

### 5WHY 분석

| WHY | 질문 | 응답 |
|-----|------|------|
| **WHY 1** | 왜 여행자는 장소의 상태 변화를 도착 후에야 알게 되는가? | 이동 중에 도착 예정 장소의 실시간 상태(임시 휴업, 혼잡도, 교통 장애)를 선제적으로 알려주는 시스템이 없기 때문이다. 현재 여행자는 "도착 -> 확인 -> 대응"의 사후 반응형(reactive) 패턴으로 행동할 수밖에 없다. |
| **WHY 2** | 왜 선제적 알림 시스템이 없는가? | 선제적 알림을 위해서는 (1) 여행자의 현재 위치와 이동 중인 목적지를 실시간 추적하고, (2) 목적지의 상태를 지속적으로 모니터링하며, (3) 두 정보를 교차 분석하여 "도착 전 최적 시점"에 알림을 발행하는 **이벤트 드리븐 아키텍처**가 필요한데, 기존 여행 앱은 이러한 실시간 이벤트 처리 파이프라인을 갖추고 있지 않다. |
| **WHY 3** | 왜 기존 앱은 실시간 이벤트 처리를 하지 못하는가? | 기존 여행 앱(트리플, 구글맵 등)의 아키텍처가 **요청-응답(Request-Response) 기반의 정적 정보 제공 모델**로 설계되었기 때문이다. 사용자가 검색을 실행해야 정보가 반환되는 풀(Pull) 방식이며, 시스템이 먼저 상태 변화를 감지하여 사용자에게 보내주는 푸시(Push) 방식이 아니다. 구글맵의 영업시간은 크라우드소싱 기반으로 수일~수주 주기로 갱신되어 실시간성이 근본적으로 부족하다. |
| **WHY 4** | 왜 실시간 상태 모니터링이 기술적으로 어려운가? | 여행 목적지의 실시간 상태를 파악하려면 다수의 이질적 신호 소스를 병합해야 하기 때문이다. (1) 날씨 데이터는 기상청 API에서 제공되지만 시간대별 정밀도가 낮고, (2) 영업 상태는 개별 점포가 실시간으로 공유하는 체계가 없으며(대부분 수동 업데이트), (3) 혼잡도는 GPS 밀집도/Wi-Fi 접속 수 등 간접 지표로 추정해야 하고, (4) 교통 장애는 각국 교통 당국의 데이터 공개 수준이 천차만별이다. 이 모든 소스를 실시간으로 수집-정규화-분석하는 **스트림 프로세싱 인프라**를 구축하는 비용이 매우 높다. |
| **WHY 5** | 왜 이 인프라에 대한 투자가 이루어지지 않았는가? | "여행 중" 단계에서의 실시간 데이터 품질 보장은 **높은 데이터 수집 비용 대비 직접적 수익 모델이 약하다**는 구조적 문제가 있기 때문이다. 예약/결제 시점에 수수료를 취하는 OTA(Online Travel Agency) 모델에서는 "여행 전"에 트래픽이 집중되고, "여행 중"은 이미 결제가 완료된 이후라 수익 창출 기회가 제한적이었다. 따라서 실시간 인프라에 대한 투자 인센티브가 시장 구조적으로 부족했다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| **사용자 심리적** | 여행자는 "구글맵에 영업 중이라고 뜨니까 괜찮겠지"라는 **플랫폼 권위에 대한 과신**을 가지고 있다. 대형 플랫폼의 정보가 정확할 것이라는 기대가 사전 교차 검증 행동을 억제하고, 도착 후 사실과 다를 때의 배신감과 불신을 증폭시킨다. (인터뷰 3/5명 "구글맵 영업시간을 100% 믿을 수 없다") |
| **제도/시스템적** | 상점의 영업 상태를 실시간으로 공유하는 표준 프로토콜이 없다. 임시 휴업, 특별 행사로 인한 폐쇄, 정기 휴일 변경 등을 디지털 채널로 즉시 전파하는 시스템이 각국 소상공인 수준에서는 사실상 존재하지 않는다. 대형 체인을 제외하면 영업 상태 갱신은 구글 비즈니스 프로필의 수동 업데이트에 의존한다. |
| **맥락적** | 돌발 상황의 유형이 날씨(폭우, 폭염), 영업 상태(임시 휴업, 정기 휴일), 혼잡도(대기 90분), 교통(BTS 점검, 나루터 운행 중단) 등 다양하며, 각각의 데이터 소스와 감지 방법이 완전히 다르다. 단일 시스템으로 이 모든 유형의 돌발 상황을 감지하려면 다차원 컨텍스트 인식이 필요한데, 이는 기존의 단일 도메인 서비스(날씨 앱, 지도 앱, 교통 앱)가 각각 분리되어 있는 현 구조에서는 불가능하다. |
| **커뮤니케이션** | 돌발 상황 정보가 현지어로만 제공되는 경우가 빈번하다. (관찰3: 영어/태국어 안내문, 체험6: 스페인어 파업 안내문) 현장 안내문, 전광판, 공식 웹사이트가 현지어 중심이라 외국인 여행자가 상황을 정확히 파악하는 데 언어 장벽이 추가적인 시간 지연을 발생시킨다. 이는 **다국어 실시간 정보 전달 채널**의 부재 문제이다. |

### 핵심 근본원인

> **기존 여행 정보 시스템이 요청-응답 기반의 정적 아키텍처로 설계되어 있어, 여행자의 이동 컨텍스트와 목적지의 상태 변화를 실시간으로 교차 분석하여 선제적으로 알림을 발행하는 이벤트 드리븐 아키텍처가 부재하다.**

## 현상문제 3: 하나의 일정이 변경되면 나머지 일정이 연쇄적으로 무너지고, 이를 머릿속에서 수동으로 재계산해야 한다

> 인터뷰 4/5명, 체험 8/10회에서 확인. "하나가 밀리니까 연쇄적으로 다 밀려서, 결국 마지막 날에 원래 첫째 날 일정을 하게 됐어요. 이걸 머릿속으로 퍼즐 맞추듯이 재조정하는 게 진짜 피곤했어요." (소윤, S1)

### 5WHY 분석

| WHY | 질문 | 응답 |
|-----|------|------|
| **WHY 1** | 왜 하나의 일정 변경이 전체 일정의 연쇄 붕괴를 일으키는가? | 여행 일정의 각 항목이 시간(영업시간, 일몰 시각, 예약 시간), 위치(이동 동선), 의존 관계(A를 먼저 봐야 B로 이동 가능)로 서로 강하게 결합(tightly coupled)되어 있기 때문이다. 하나의 노드가 변경되면 연결된 모든 노드에 전파(propagation)가 발생한다. (체험5: 빈티지숍 2곳 휴업으로 시모키타자와 일정 40분에 종료, 이후 전체 재배치) |
| **WHY 2** | 왜 여행자는 이 연쇄 영향을 수동으로 재계산해야 하는가? | 현재 여행 일정 관리 도구(노션, 트리플, 메모 앱)가 **정적 목록(static list) 구조**로 되어 있어, 항목 간 시간적/공간적 제약 조건(constraint)을 모델링하지 못하기 때문이다. 일정을 "할 일 목록"처럼 나열할 뿐, 각 항목의 영업시간, 소요 시간, 이동 거리, 의존 관계를 데이터로 관리하지 않는다. |
| **WHY 3** | 왜 기존 도구는 제약 조건을 모델링하지 못하는가? | 여행 일정의 제약 조건을 컴퓨터가 처리하려면 **제약 충족 문제(Constraint Satisfaction Problem, CSP)** 를 풀어야 하는데, 이를 위해서는 (1) 각 장소의 영업시간/위치/소요시간이 구조화된 데이터로 존재해야 하고, (2) 장소 간 이동 시간이 실시간으로 계산 가능해야 하며, (3) 변경이 발생했을 때 나머지 일정을 실시간으로 재최적화하는 솔버(solver)가 필요하다. 기존 도구는 이 세 가지를 모두 갖추지 못했다. |
| **WHY 4** | 왜 이 최적화 엔진이 구현되지 않았는가? | 여행 일정 최적화는 **다변수 조합 최적화 문제**로, 변수가 많고(장소 수 x 시간 슬롯 x 이동 수단 x 영업 상태), 제약 조건이 실시간으로 변하며(날씨 변화, 혼잡도 변동, 임시 휴업), 사용자의 주관적 선호(분위기, 피로도, 동행자 의견)까지 반영해야 하는 복잡도를 갖기 때문이다. 전통적인 알고리즘(TSP, VRP)으로는 이 복잡도를 실시간으로 처리하기 어렵고, LLM 기반 추론과 결합해야 사용자 맥락을 반영한 유연한 재스케줄링이 가능하다. |
| **WHY 5** | 왜 이 기술적 도전이 해결되지 않았는가? | "여행 중 동적 일정 재최적화"라는 문제를 풀기 위해서는 **실시간 데이터 파이프라인(날씨/영업/혼잡도) + 경로 최적화 엔진 + LLM 기반 맥락 이해 + 사용자 선호 학습**이 하나의 시스템으로 통합되어야 하는데, 이는 최근에야 기술적으로 실현 가능해진 영역이다. 특히 LLM의 맥락 이해 능력이 2023년 이후 급격히 발전하면서, "비가 오니까 실내 일정으로 전환하되 동선을 최소화하고 아이가 좋아할 만한 곳을 우선 배치"와 같은 복합 제약을 자연어로 표현하고 처리하는 것이 비로소 가능해졌다. |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| **사용자 심리적** | 여행자는 일정을 "완벽하게 짜야 한다"는 강박을 가지고 있으며(특히 S3 초보: 분 단위 일정), 이 촘촘한 계획이 오히려 유연성을 극도로 낮춘다. 하나가 틀어졌을 때 "전부 망했다"는 인지적 과장(catastrophizing)이 발생하고, 수동 재계산의 인지 부하가 "전체가 엉켰다"는 불안을 현실보다 크게 느끼게 만든다. (인터뷰 4/5명 "나머지 일정이 다 엉키는 거 아니야?") |
| **제도/시스템적** | 현재의 여행 일정 도구(트리플, 노션, 구글 캘린더)는 **일정을 정적 문서로 취급**한다. 시간을 배정하고 장소를 나열하는 기능만 있을 뿐, 장소 간 이동 시간, 영업시간 제약, 일정 간 의존 관계를 데이터로 관리하는 구조가 아니다. 따라서 하나를 수정하면 나머지를 수동으로 재계산할 수밖에 없는 도구 구조적 한계가 있다. |
| **맥락적** | 일정 재계산이 필요한 시점은 이미 돌발 상황으로 인한 심리적 충격과 시간 압박이 겹쳐 있는 상태이다. 감정 레벨이 1.6~1.9인 최저점에서 복잡한 조합 최적화를 머릿속으로 수행해야 하므로, 인지 능력이 가장 낮은 시점에 가장 높은 인지 부하가 걸리는 최악의 조합이 발생한다. |
| **커뮤니케이션** | 일정 변경의 연쇄 영향이 동행자에게 적시에 공유되지 않는다. 커플/그룹 여행에서 일정 변경을 구두로만 합의하거나(관찰2), 카카오톡 메모에 수기로 작성하여(체험2) 추후 혼선이 발생한다. 일정 변경의 영향 범위를 시각적으로 보여주고 동행자 간 실시간 동기화하는 수단이 없다. |

### 핵심 근본원인

> **여행 일정 관리 도구가 항목 간 시간적/공간적 제약 조건을 데이터로 모델링하지 못하는 정적 목록 구조로 설계되어 있어, 변경 전파와 재최적화를 자동으로 수행하는 동적 스케줄링 엔진이 부재하다.**

## 핵심 근본원인 종합

| # | 현상문제 | 핵심 근본원인 | 기술 아키텍처 키워드 |
|---|---------|-------------|-------------------|
| 1 | 대안 탐색에 23.5분 소요 | 이기종 데이터 소스를 단일 컨텍스트로 통합하는 데이터 통합 아키텍처 부재 | Data Integration, Context Aggregation, Multi-source Fusion |
| 2 | 도착 후 사후 인지 | 이동 컨텍스트와 목적지 상태를 실시간 교차 분석하는 이벤트 드리븐 아키텍처 부재 | Event-Driven Architecture, Real-time Stream Processing, Proactive Push |
| 3 | 일정 연쇄 붕괴와 수동 재계산 | 제약 조건 기반 동적 스케줄링 엔진과 실시간 재최적화 솔버 부재 | Constraint Satisfaction, Dynamic Scheduling, Optimization Engine |

### 세 가지 근본원인의 구조적 관계

세 가지 근본원인은 독립적이 아니라 **계층적 의존 관계**를 가진다.

```
[계층 1: 데이터 기반]
  이기종 데이터 통합 아키텍처 (근본원인 1)
      |
      v
[계층 2: 실시간 처리]
  이벤트 드리븐 실시간 모니터링 (근본원인 2)
      |
      v
[계층 3: 지능형 의사결정]
  제약 기반 동적 스케줄링 엔진 (근본원인 3)
```

- **계층 1(데이터 통합)**이 해결되지 않으면 계층 2, 3은 작동할 수 없다. 통합된 데이터가 있어야 실시간 모니터링이 가능하고, 모니터링 결과가 있어야 동적 재스케줄링이 의미를 가진다.
- 이는 곧 시스템 아키텍처의 **설계 우선순위**를 결정한다: 데이터 통합 계층을 먼저 구축하고, 그 위에 이벤트 처리 계층을, 그 위에 최적화 엔진을 올리는 계층적 아키텍처가 필요하다.
- 궁극적으로 이 세 계층을 관통하는 것은 **사용자의 실시간 맥락(위치, 날씨, 시간, 동행자, 선호도)을 이해하는 컨텍스트 엔진**이며, 이것이 travel-planner 서비스의 아키텍처적 핵심 차별점이 된다.
