# 5WHY 분석 — 파이프 (DevOps 엔지니어)

> 작성자: 송주영 / 파이프 (DevOps 엔지니어)
> 작성일: 2026-02-22
> 관점: 서비스 가용성 · 실시간 데이터 파이프라인 · 모니터링 · 글로벌 운영

---

## 분석 배경

고객 관찰(10회), 체험(10회), 심층 인터뷰(5명) 데이터를 DevOps/운영 관점에서 재해석했다.
고객이 "구글맵 영업 중인데 도착하니 문 닫혔다", "대안 찾는데 35분 날렸다", "비 온다는 걸 현장에서야 알았다"고 말할 때, 이것은 단순한 UX 문제가 아니다. 근저에는 **실시간 데이터 파이프라인 부재**, **글로벌 멀티소스 데이터 동기화 실패**, **서비스 장애를 고객이 대신 감지하는 구조적 결함**이 있다. 수동 배포가 없듯이, 수동 정보 탐색도 허용되어선 안 된다.

---

## 현상문제 1

### "고객이 목적지에 도착한 후에야 휴업·혼잡·운행중단을 인지한다"

> 인터뷰 4/5명이 "도착 후 사후 인지" 패턴. 감정 낙차 -2.0 (3.6 -> 1.6). 심각도 4.8/5.
> "구글맵이 영업 중이라고 해도 100% 믿을 수 없다" (준혁)

---

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 고객은 목적지에 도착해서야 문제를 인지하는가? | 이동 중에는 목적지의 실시간 상태(휴업·혼잡·운행중단)를 알려주는 알림이 존재하지 않기 때문이다 |
| **WHY 2** | 왜 이동 중에 실시간 상태 알림이 없는가? | 여행 앱들이 목적지 상태 데이터를 실시간으로 수집·처리하는 파이프라인을 보유하지 않기 때문이다 |
| **WHY 3** | 왜 여행 앱들은 실시간 상태 파이프라인을 보유하지 않는가? | 영업시간·휴업·혼잡도 데이터는 POI(Point of Interest) 운영자가 비정형적으로 업데이트하며, 이를 자동 수집할 표준화된 API나 데이터 소스가 존재하지 않기 때문이다 |
| **WHY 4** | 왜 POI 상태 데이터에 표준화된 API가 없는가? | 전 세계 수백만 개의 개별 점포·관광지는 각자 다른 플랫폼(SNS, 자체 홈페이지, 구글 비즈니스)을 사용하고, 그 어떤 플랫폼도 실시간 상태를 기계가 읽을 수 있는 형식으로 발행할 인센티브가 없기 때문이다 |
| **WHY 5** | 왜 어떤 플랫폼도 POI 실시간 상태를 표준 발행할 인센티브가 없는가? | 이 데이터의 최종 소비자(여행자)와 데이터 생산자(점포 운영자) 사이에 가치 교환 구조가 형성되지 않았으며, 수집 비용이 수익보다 크다고 판단되기 때문이다 |

---

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|---------|
| **사용자 심리적** | 고객은 온라인 정보(구글맵 영업 중)를 신뢰하도록 학습되어 있으나, 실제 정확도가 낮다는 것을 경험으로 알고 있다. "믿고 싶지만 믿을 수 없는" 인지 부조화 상태이며, 불확실성을 해소할 대안 수단이 없으므로 헛걸음을 감수하고 직접 확인한다 |
| **제도·시스템적** | 구글, 네이버 등 기존 지도 서비스의 POI 데이터 갱신 주기는 수일~수주 단위이다. 실시간 상태(임시 휴업, 오늘의 혼잡도)를 반영하는 표준 데이터 스펙과 API가 없으며, 각국 규제·언어가 달라 글로벌 단일 파이프라인 구축이 구조적으로 어렵다 |
| **맥락적** | 여행자는 이동 중이라는 맥락(이미 15~75분 이동 비용 지불 완료)에서 문제를 인지한다. 이 시점에서 대안을 찾으면 매몰 비용 효과로 심리적 고통이 증폭된다. 사전 알림이었다면 감정 낙차가 -2.0이 아니라 -0.5 수준으로 억제될 수 있다 |
| **커뮤니케이션** | POI 운영자(점포)는 임시 휴업을 인스타그램·카카오채널 등 자기 채널에만 발행하며, 이 정보가 여행자가 사용하는 지도 서비스로 전파되는 채널이 없다. 정보 생산자와 소비자 사이의 커뮤니케이션 레이어가 단절되어 있다 |

---

### 핵심 근본원인

> **전 세계 POI의 실시간 상태 데이터를 자동 수집·정규화·발행하는 표준화된 데이터 파이프라인이 존재하지 않으며, 이를 구축할 경제적 인센티브 구조도 형성되지 않았다.**
>
> DevOps 관점 요약: "헛걸음"은 UX 실패가 아니라 **데이터 파이프라인 SLA(Service Level Agreement) 부재**로 인한 시스템 장애다. 고객이 직접 목적지를 방문해 상태를 확인하는 행위는, 자동화 모니터링이 없을 때 사람이 수동으로 서버 상태를 확인하는 것과 동일한 구조다.

## 현상문제 2

### "돌발 상황 발생 시 고객이 대안을 찾는 데 평균 23.5분을 소모한다"

> 체험 10회 평균 23.5분 (최소 12분~최대 40분). 심각도 5.0/5.
> "3박 4일에서 매일 30분씩 날리면 하루치 일정 하나가 통째로 사라지는 거예요" (준혁)
> 구글맵·네이버블로그·인스타그램 3개 앱을 5~6탭 왕복하는 "삼각 동선" 패턴

---

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 고객은 대안 탐색에 평균 23.5분이나 소모하는가? | 거리·리뷰·영업 상태·사진이라는 4가지 판단 기준이 서로 다른 앱(구글맵·네이버·인스타그램)에 분산되어 있어 앱 간 전환을 반복해야 하기 때문이다 |
| **WHY 2** | 왜 판단에 필요한 정보가 여러 앱에 분산되어 있는가? | 각 플랫폼이 자신의 강점(위치 정보, 한국어 리뷰, 비주얼 콘텐츠)에 특화된 사일로(silo) 구조로 운영되며, 플랫폼 간 데이터 공유 API를 제공하지 않기 때문이다 |
| **WHY 3** | 왜 플랫폼들은 데이터 공유 API를 제공하지 않는가? | 각 플랫폼에게 데이터는 경쟁 우위 자산이므로, 외부에 공유하면 트래픽·체류시간·광고 수익이 감소한다. 공유의 비용이 공유의 이익보다 크다 |
| **WHY 4** | 왜 고객 입장에서 이 분산 구조를 해소할 통합 레이어가 존재하지 않는가? | 통합 레이어를 만들려면 복수 플랫폼의 데이터를 실시간으로 크롤링·정규화·캐싱하는 백엔드 인프라가 필요하며, 이 비용과 법적 리스크(이용약관 위반)를 감수할 사업자가 없었기 때문이다 |
| **WHY 5** | 왜 이 통합 레이어 사업이 아직 등장하지 않았는가? | 여행 중 실시간 대안 탐색이라는 니즈는 여행 전 계획 도구나 여행 후 리뷰 플랫폼에 비해 시장이 작고 사용 빈도가 산발적으로 보이기 때문에, 기존 플랫폼 사업자들이 전략적 우선순위를 부여하지 않았다 |

---

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|---------|
| **사용자 심리적** | 고객은 "이 선택이 정말 최선인가"라는 확인 욕구를 충족시키기 위해 다중 소스를 검증한다 (인터뷰 5/5명 "이게 최선인지 모르겠다"). 정보를 더 많이 볼수록 안심하는 심리가 탐색 시간을 연장하며, 이는 정보의 신뢰도가 낮을수록 더 심화된다 |
| **제도·시스템적** | 플랫폼 간 데이터 인터오퍼러빌리티(interoperability) 표준이 없다. 구글 Places API, 네이버 Local API, 인스타그램 Graph API는 각각 다른 스펙과 요금 정책을 가지며, 이를 하나의 파이프라인으로 통합하면 API 비용·Rate Limit·데이터 신선도 관리가 모두 엔지니어링 과제로 전환된다 |
| **맥락적** | 대안 탐색은 길거리·비·더위 등 열악한 물리적 환경에서 이루어진다 (관찰: 33도 야외에서 35분, 젖은 손으로 터치). 인지 부하가 높은 환경에서 다중 앱 전환은 물리적·심리적 피로를 기하급수적으로 증가시키며, 이것이 탐색 포기(일정 축소)로 이어진다 |
| **커뮤니케이션** | 추천 결과에 "왜 이것을 추천하는가"에 대한 맥락이 없다. 구글맵의 별점 순 나열은 고객의 현재 상황(날씨·피로도·동행자·선호)을 반영하지 않으므로, 고객은 결과를 수동으로 필터링해야 한다. 기계가 해야 할 컨텍스트 매칭을 사람이 대신 수행하고 있다 |

---

### 핵심 근본원인

> **여행 중 실시간 맥락(위치·날씨·영업상태·선호)을 종합하여 즉시 대안을 제시하는 통합 데이터 처리 레이어가 존재하지 않으며, 기존 플랫폼들의 데이터 사일로 구조가 이를 만들지 못하게 막고 있다.**
>
> DevOps 관점 요약: 고객이 5~6개 탭을 왕복하는 행위는 **API 오케스트레이션 레이어 부재**의 증거다. 마이크로서비스 환경에서 서비스 간 통신을 수동으로 처리하게 두는 것과 동일한 구조적 결함이다. 23.5분은 자동화가 해결해야 할 처리 시간이다.

## 현상문제 3

### "날씨·교통·혼잡도 등 실시간 외부 변수가 일정에 반영되지 않아 고객이 현장에서 도미노식 일정 붕괴를 경험한다"

> 하나의 변경 시 평균 2~3곳 연쇄 변경. 수동 재조정 5~15분 추가. 심각도 4.4/5.
> "마지막 날에 원래 첫째 날 일정을 하게 됐어요. 머릿속으로 퍼즐 맞추듯이 재조정하는 게 진짜 피곤했어요" (소윤)
> 관찰: 교통체증으로 이동 30분이 1시간 15분으로 연장. BTS 운행 중단으로 대안 이동 1시간 추가

---

### 5WHY 분석

| 단계 | 질문 | 응답 |
|------|------|------|
| **WHY 1** | 왜 하나의 일정 변경이 전체 일정의 도미노식 붕괴로 이어지는가? | 고객의 일정은 시간·위치 의존성을 가진 연속 이벤트 체인이지만, 이를 관리하는 도구가 노션·메모장 같은 정적 텍스트 기반이어서 외부 변수 변화가 자동으로 반영되지 않기 때문이다 |
| **WHY 2** | 왜 여행 일정 도구는 외부 변수를 자동으로 반영하지 않는가? | 기존 여행 앱(트리플 등)은 "여행 전 계획 작성" 도구로 설계되었으며, 실행 중 실시간 외부 데이터(날씨 API, 교통 API, 혼잡도 데이터)와 연동하는 런타임 엔진이 없기 때문이다 |
| **WHY 3** | 왜 여행 앱들은 실시간 외부 데이터와 연동하는 런타임 엔진을 만들지 않았는가? | 날씨·교통·혼잡도를 실시간으로 수집하고 일정 최적화 알고리즘에 입력하는 것은 단순 CRUD 앱 개발 수준을 넘는 복잡한 스트리밍 데이터 파이프라인과 최적화 엔진이 필요하며, 기존 여행 앱들의 기술 역량과 비즈니스 모델이 이를 지원하지 않았기 때문이다 |
| **WHY 4** | 왜 여행 앱들의 기술 역량과 비즈니스 모델이 스트리밍 파이프라인을 지원하지 않았는가? | 여행 앱 시장의 수익 모델이 광고·예약 수수료 중심으로 형성되었으며, "여행 중 실행 단계" 사용자의 이탈률이 높아(여행 전 계획 완료 후 이탈) 이 구간에 투자할 ROI가 낮다고 판단했기 때문이다 |
| **WHY 5** | 왜 "여행 중 실행 단계"의 투자 ROI가 낮다고 판단되는가? | 여행 중 사용자의 세션이 짧고(이동 중 단발 조회), 실시간 처리 인프라 비용(스트리밍·글로벌 CDN·멀티소스 API)이 단일 여행 앱의 수익으로 회수하기 어려운 구조이기 때문이다 |

---

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|---------|
| **사용자 심리적** | 고객은 일정 재조정 시 "이게 최선인지 모르겠다"는 불확실성(5/5명)과 "원래 가려던 곳을 포기해야 하는 아쉬움"(4/5명)을 동시에 경험한다. 인간의 손실 회피 심리로 인해 이미 틀어진 일정에 대한 집착이 생기고, 새 계획에 에너지를 투입하는 것이 심리적으로 어렵다 |
| **제도·시스템적** | 글로벌 실시간 교통·날씨 데이터는 공개 API로 일부 접근 가능하지만, 혼잡도·임시 운행 변경·특정 관광지 상태 같은 데이터는 표준화된 공개 소스가 없다. 특히 교통 파업·자연재해 등 예외 상황은 현지 언어 소스를 실시간으로 파싱해야 하며, 이는 NLP와 글로벌 멀티언어 파이프라인이 필요한 영역이다 |
| **맥락적** | 도미노 붕괴는 고객이 촘촘한 일정을 세울수록 심화된다. 버퍼 시간 없이 최대한 많은 장소를 가려는 효율 추구형 여행자(S1 세그먼트가 1순위)일수록 하나의 지연이 전체를 무너뜨린다. 즉, 가장 가치 있는 고객이 가장 큰 고통을 겪는 구조다 |
| **커뮤니케이션** | 일정 변경이 필요한 시점에 고객에게 전달되는 정보가 없다. 기상청 API는 날씨 변화를 2~3시간 전에 예측할 수 있고, 구글 혼잡도 데이터는 예상 혼잡 시간대를 제공한다. 이 데이터가 고객의 일정과 연결되지 않으므로, 고객이 현장에서 충격을 받을 때까지 어떤 예경고도 발생하지 않는다 |

### 핵심 근본원인

> **여행 일정이 정적인 텍스트 데이터로만 존재하며, 날씨·교통·혼잡도 같은 실시간 외부 이벤트 스트림과 연동되는 동적 재조정 엔진이 없다. 기존 여행 앱들은 "여행 중 실행" 구간의 수익 모델을 만들지 못했기 때문에 이 문제를 해결할 기술 투자를 하지 않았다.**
>
> DevOps 관점 요약: 여행 일정은 외부 이벤트 트리거에 반응해야 하는 **이벤트 기반 워크플로우(Event-Driven Workflow)**다. 현재 고객의 일정은 이를 정적 YAML 파일처럼 관리하고 있다. 쿠버네티스가 노드 장애에 자동으로 Pod를 재스케줄링하듯이, 여행 일정도 외부 변수에 자동 재스케줄링되어야 한다. 그 오케스트레이터가 존재하지 않는 것이 근본 문제다.

## 종합 핵심 근본원인

| 현상문제 | 핵심 근본원인 | DevOps 관점 메타포 |
|----------|-------------|-------------------|
| 현상 1: 헛걸음 (사후 인지) | 전 세계 POI 실시간 상태 수집·발행 파이프라인 부재 + 경제적 인센티브 구조 미형성 | 프로덕션 서버에 모니터링 에이전트가 없어 장애를 고객 신고로 인지하는 구조 |
| 현상 2: 23.5분 탐색 낭비 | 다중 플랫폼 데이터 사일로 + 실시간 맥락 기반 통합 오케스트레이션 레이어 부재 | API 게이트웨이 없이 클라이언트가 각 마이크로서비스를 개별 호출하는 구조 |
| 현상 3: 도미노 일정 붕괴 | 정적 일정과 동적 외부 이벤트 스트림의 비연동 + 이벤트 기반 재스케줄링 엔진 부재 | 이벤트 브로커 없이 단일 CRON 스케줄로만 운영되는 배치 파이프라인 구조 |

**DevOps 엔지니어 파이프의 최종 진단:**

세 현상문제의 공통 근원은 하나다.
"여행 중 실행" 구간을 지원하는 **실시간 데이터 수집-처리-전달 인프라가 존재하지 않는다.**
날씨·교통·POI 상태·혼잡도라는 이종(異種) 데이터 스트림을 수집하고, 개인화 컨텍스트와 결합하고, 3분 이내에 액션 가능한 형태로 고객에게 전달하는 파이프라인이 없기 때문에, 그 공백을 고객이 23.5분의 수동 노동으로 채우고 있다.

이것은 수동 배포와 같다. 반드시 자동화되어야 한다.
