컨텍스트 파일(고객분석, 유저저니맵, 관찰결과, 체험결과, 인터뷰결과, 인터뷰취합) 분석을 통해

- 인터뷰 4/5명에서 "도착 후에야 휴업·혼잡을 인지"하는 사후 인지 패턴 확인
- 인터뷰 4/5명에서 "1곳 변경 시 평균 2~3곳 연쇄 재조정 발생" 확인

## 현상문제 1: 고객이 여러 앱을 전전하며 대안 장소 정보를 찾는 데 평균 23.5분을 소비한다

> 배경: 인터뷰 5/5명, 체험 10/10회 공통 확인. 구글맵 → 네이버 블로그 → 인스타그램의
> "삼각 동선" 패턴이 4/5명에서 반복됨. 거리·영업상태·리뷰·사진이라는 4가지 판단 기준이
> 각각 다른 플랫폼에 분산된 것이 직접 원인.

### 5WHY 테이블

| 단계 | 질문 | 응답 |
|------|------|------|
| WHY 1 | 왜 고객은 대안을 찾는 데 23.5분이나 소모하는가? | 거리·영업상태·리뷰·사진이라는 4가지 판단 기준이 구글맵·네이버·인스타그램에 각각 흩어져 있어, 앱을 5~6번 오가며 정보를 수동으로 조합해야 하기 때문이다 |
| WHY 2 | 왜 판단 기준이 여러 플랫폼에 흩어져 있는가? | 각 플랫폼은 자기 도메인(지도, 블로그, 사진)에 최적화되어 있고, 여행 현지 실시간 맥락(현재 위치·현재 영업 여부·한국어 리뷰)을 통합 제공하는 단일 API가 존재하지 않기 때문이다 |
| WHY 3 | 왜 실시간 맥락을 통합 제공하는 API가 없는가? | 영업시간 실시간 검증, 혼잡도, 날씨 연계, 다국어 리뷰 요약은 각각 이종(異種) 외부 데이터 소스(Google Places, 기상청, SNS, 리뷰 플랫폼)를 조합해야 하며, 이를 저지연으로 집계하는 통합 백엔드 없이는 단일 응답으로 만들 수 없기 때문이다 |
| WHY 4 | 왜 이종 외부 소스를 통합하는 백엔드가 구축되지 않았는가? | 이종 API 호출은 각각의 레이트리밋·응답지연·데이터 포맷 불일치 문제가 있고, 이를 병렬 집계하여 수백 밀리초 이내에 응답하는 설계는 단순 CRUD 앱 수준의 구현으로는 불가능하기 때문이다 |
| WHY 5 | 왜 병렬 집계 설계가 적용되지 않았는가? | 기존 여행 앱들은 "여행 전 계획" 단계를 위한 정적 데이터 제공에 집중하여 설계되었고, 여행 중 돌발 상황이라는 "실시간·맥락 의존적 요구"를 충족하는 고성능 집계 레이어를 서비스 핵심으로 설정하지 않았기 때문이다 |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| 사용자 심리적 | 고객은 "틀린 선택을 하면 제한된 여행 시간을 낭비한다"는 손실 회피 심리로 인해, 충분한 정보를 확보하기 전까지 결정을 미루게 된다. 정보가 파편화될수록 확인 행동이 늘어나고 탐색 시간이 선형적으로 증가한다 |
| 제도·시스템적 | Google Places API, 네이버 지역 API, SNS 플랫폼은 각자 독립된 생태계를 유지하며 상호 연동을 공식 지원하지 않는다. 크로스-플랫폼 데이터 통합은 이용약관상 제약이 따르며, 이를 해소하는 미들웨어 레이어가 시스템적으로 부재하다 |
| 맥락적 | 돌발 상황은 "지금, 여기, 이 조건에서"라는 극도로 맥락 의존적인 요구를 발생시킨다. 그러나 기존 여행 앱은 사전 계획 단계를 위한 범용 정보를 제공하도록 설계되어, 실시간 맥락 파라미터(현재 날씨, 현재 위치, 현재 영업 여부)를 쿼리 조건으로 받는 API를 제공하지 않는다 |
| 커뮤니케이션 | 정보를 제공하는 각 플랫폼이 "지금 이 사람에게 필요한 것"을 고객에게 선제적으로 전달하는 푸시 메커니즘이 없다. 고객이 직접 여러 채널에 능동적으로 요청(풀)해야 하므로, 정보 탐색 자체가 여행 시간을 잠식하는 별도의 작업이 된다 |

### 핵심 근본원인

> **여행 중 실시간 맥락(위치·날씨·영업상태·혼잡도)을 단일 요청으로 통합 집계하여
> 수백 밀리초 이내에 응답하는 백엔드 집계 레이어가 존재하지 않기 때문이다.**
> 이 레이어 없이는 어떤 클라이언트 앱을 만들더라도 고객은 정보를 수동으로 조합하는
> 23.5분의 탐색 과정을 반복할 수밖에 없다.

## 현상문제 2: 고객이 목적지에 도착한 후에야 휴업·혼잡·운행 중단을 인지하여 이동 시간 전체를 낭비한다

> 배경: 인터뷰 4/5명, 관찰 6/10회, 체험 8/10회에서 확인. "사후 인지" 패턴이 감정 곡선
> 3.6→1.6의 급락(-2.0)을 직접 유발함. 구글맵의 영업 정보가 실제와 다른 경우가 빈번하며
> (준혁: "100% 믿을 수 없다"), 교통 운행 중단·임시 휴업은 사전 알림 없이 현장에서만 인지된다.

### 5WHY 테이블

| 단계 | 질문 | 응답 |
|------|------|------|
| WHY 1 | 왜 고객은 목적지에 도착한 후에야 휴업·혼잡·운행 중단을 알게 되는가? | 기존 플랫폼(구글맵 등)의 영업 정보는 점주가 수동으로 등록한 고정 데이터이고, 임시 휴업·당일 혼잡도·교통 돌발 중단은 실시간으로 반영되지 않기 때문이다 |
| WHY 2 | 왜 영업 정보가 실시간으로 반영되지 않는가? | 영업 상태를 실시간으로 갱신하려면 점포로부터 상태 변경 신호를 수신하거나, 외부 신호(SNS 언급, 결제 데이터, 리뷰 패턴)를 분석하여 영업 여부를 추론하는 데이터 파이프라인이 필요한데, 기존 플랫폼은 이를 구축하지 않았기 때문이다 |
| WHY 3 | 왜 실시간 영업 상태 갱신 파이프라인이 구축되지 않았는가? | 수십만~수백만 개 장소의 상태를 실시간으로 폴링·이벤트 수신하려면 고빈도 외부 API 호출, 스트리밍 데이터 처리, 점포별 이상 탐지 로직이 필요하며, 이는 단순 CRUD 기반 플랫폼의 아키텍처 범위를 초과하기 때문이다 |
| WHY 4 | 왜 스트리밍 데이터 처리와 이상 탐지가 적용되지 않았는가? | 기존 여행 정보 플랫폼은 "정확한 정적 정보의 DB 구축"을 핵심 가치로 삼아 설계되었고, "돌발 상황의 선제적 감지와 사용자 알림"은 서비스 범위에 포함되지 않았기 때문이다 |
| WHY 5 | 왜 선제적 감지와 알림이 서비스 범위에서 배제되었는가? | 고객이 "여행 전" 정보를 찾는 수요는 명확하게 식별되었지만, "여행 중" 이동 경로상의 장소를 실시간 감시하여 도달 전에 알림을 보내는 것은 기술적 난이도가 높고 수익 모델과의 연계가 불분명하여 제품 우선순위에서 후순위로 밀렸기 때문이다 |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| 사용자 심리적 | 고객은 이동 전에 영업 여부를 직접 확인하는 행동을 번거롭게 여기며, 플랫폼 표기를 신뢰한다. 그러나 한 번 헛걸음을 경험하면 "구글맵을 100% 믿을 수 없다"는 불신이 형성되고, 이후 매번 추가 확인 행동을 반복하게 되어 전체 탐색 부담이 증가한다 |
| 제도·시스템적 | 점포 영업 상태는 점주가 자발적으로 업데이트하지 않으면 플랫폼이 강제로 갱신할 수단이 없다. 임시 휴업·특별 이벤트·정기 휴일 변경은 SNS, 점포 홈페이지, 현장 안내문 등 분산된 채널에 개별적으로 공지되며 표준화된 데이터 포맷이 없다 |
| 맥락적 | 고객이 이동 중인 시점에는 스마트폰으로 추가 확인을 하기 어렵고(이동 중 조작 불편), 목적지 도착 직전에야 이상을 인지할 가능성이 높다. 선제적 알림은 고객의 이동 동선과 예상 도착 시간을 서버가 추적하고 있어야 가능한데, 이를 위한 실시간 위치 기반 모니터링 기능이 기존 앱에는 없다 |
| 커뮤니케이션 | 장소의 영업 상태 변경 정보는 점포 → 플랫폼 → 고객으로 전달되는 체계가 없다. 점포는 SNS에 임시 휴업을 공지하지만, 여행자는 해당 SNS를 팔로우하지 않으므로 정보가 전달되지 않는다. 중간에서 이를 수집·정규화·배포하는 시스템이 부재하다 |

### 핵심 근본원인

> **고객의 이동 경로상 예정 장소를 실시간으로 모니터링하여, 도착 전에 영업상태 변화·혼잡도
> 급등·교통 이상을 감지하고 푸시 알림을 발송하는 이벤트 기반 백엔드 파이프라인이
> 존재하지 않기 때문이다.**
> 이 파이프라인 없이는 고객은 언제나 현장 도착 후 문제를 인지하는 "사후 대응" 구조에서
> 벗어날 수 없으며, 감정 곡선의 급락(-2.0)이 반복된다.

## 현상문제 3: 하나의 일정이 변경되면 나머지 일정을 고객이 머릿속으로 수동 재계산해야 하며, 평균 2~3곳의 연쇄 변경이 발생한다

> 배경: 인터뷰 4/5명, 체험 8/10회에서 확인. 일정 수동 재조정에 추가 5~15분 소요.
> 소윤: "하나가 밀리니까 연쇄적으로 다 밀려서 마지막 날에 첫째 날 일정을 하게 됐어요."
> 1곳 변경 → 당일+익일 연쇄 2~3곳 변경 발생. "자동 재조정이면 돈 내고 쓸 것 같다" (소윤).

### 5WHY 테이블

| 단계 | 질문 | 응답 |
|------|------|------|
| WHY 1 | 왜 하나의 일정 변경이 2~3곳의 연쇄 재계산을 유발하는가? | 일정 간에는 이동 시간·영업시간·예약 슬롯이라는 시간적 의존성이 있고, 하나가 밀리면 이후 일정의 도착 시간이 연쇄적으로 밀려 실현 가능성을 벗어나기 때문이다 |
| WHY 2 | 왜 시간적 의존성이 있는 일정 재조정을 고객이 머릿속으로 직접 계산해야 하는가? | 현재 일정 관리 앱(노션, 트리플 등)은 장소 목록을 저장·표시할 뿐, 장소 간의 이동 시간·현재 영업 여부·실시간 혼잡도를 변수로 한 "최적화 재배치" 연산을 수행하지 않기 때문이다 |
| WHY 3 | 왜 일정 앱이 최적화 재배치 연산을 수행하지 않는가? | 재배치 연산은 TSP(Traveling Salesman Problem) 계열의 최적화 문제로, 방문 후보 장소들의 실시간 영업 여부·이동 소요 시간(교통 상황 반영)·체류 시간 추정·고객 선호도를 동시에 고려해야 한다. 이는 단순 리스트 저장 수준의 앱 아키텍처에서는 구현이 불가능하기 때문이다 |
| WHY 4 | 왜 이런 최적화 연산을 수행하는 백엔드가 없는가? | 최적화 연산에 필요한 실시간 교통 API, 장소별 평균 체류 시간 데이터, 사용자 선호 패턴, 현재 영업 상태를 하나의 컨텍스트로 조합하는 데이터 모델 설계가 어렵고, 요청마다 실시간으로 최적화를 수행하면 응답 지연이 발생해 사용자 경험을 해치기 때문이다 |
| WHY 5 | 왜 실시간 최적화와 낮은 응답 지연을 동시에 달성하기 어려운가? | 최적화 연산은 후보 장소 수가 늘어날수록 계산 복잡도가 지수적으로 증가하며, 이를 사용자 요청 시점(온라인)에 처리하면 수 초~수십 초가 소요된다. 사전 캐싱·근사 알고리즘·사용자 맥락 기반 범위 제한 설계가 없으면 실사용 불가능한 응답 시간이 나오기 때문이다 |

### 다층적 근본원인 검토

| 측면 | 근본원인 |
|------|----------|
| 사용자 심리적 | 고객은 직접 만든 일정에 소유감과 애착을 느끼며, "알고리즘이 임의로 바꾼 일정"보다 "내가 직접 조정한 일정"을 더 신뢰한다. 그러나 실제로는 수동 재계산이 너무 피곤하다는 인지 부하를 동시에 경험한다. "자동 재조정이면 돈 낼 것 같다"는 발언은 이 심리적 장벽이 충분한 편의성으로 해소될 수 있음을 시사한다 |
| 제도·시스템적 | 실시간 경로 최적화에 필요한 Google Directions API, 대중교통 정보, 장소별 평균 체류 시간 데이터는 각각 유료 API이거나 직접 수집해야 하는 데이터다. 이 데이터들을 통합하여 최적화 연산의 입력으로 사용하는 것은 API 비용·데이터 수집 체계 구축이 선행되어야 한다 |
| 맥락적 | 일정 재조정은 "지금 이 순간의 상황"을 반영해야 한다. 현재 위치, 현재 시각, 현재 날씨, 남은 시간을 입력으로 하는 실시간 재최적화는 사전에 계산해 둘 수 없으며, 요청 시점에 즉시 계산되어야 한다. 이는 매 요청마다 최신 외부 데이터를 수집하고 연산하는 백엔드 설계를 요구한다 |
| 커뮤니케이션 | 고객은 "나머지 일정이 어떻게 달라지는지"를 시각적으로 확인하고 수용한 뒤에 대안을 확정하고 싶어 한다. 재조정 결과를 투명하게 설명하는 응답 포맷(변경 전/후 비교, 이유, 포기 장소의 대안 날짜 제안)이 갖춰지지 않으면 "자동 재조정"은 고객의 불신을 초래한다 |

### 핵심 근본원인

> **일정 간 시간적 의존성(이동 시간·영업시간·체류 시간)을 실시간 외부 데이터(교통 API·
> 영업 상태·날씨)와 함께 고려하여 변경 일정의 최적 배치를 즉시 계산하고 응답하는
> 최적화 연산 백엔드가 존재하지 않기 때문이다.**
> 이 연산 레이어 없이는 고객은 언제나 머릿속 퍼즐 맞추기를 반복하며, 이 피로가 누적되어
> 여행 후반부의 전체 경험 품질을 저하시킨다.

## 종합: 3개 현상문제의 백엔드 관점 핵심 근본원인 요약

| 현상문제 | 핵심 근본원인 (백엔드 관점) |
|----------|---------------------------|
| 문제 1: 대안 탐색에 평균 23.5분 소모 | 이종 외부 소스(위치·영업·리뷰·날씨)를 단일 요청으로 통합 집계하는 저지연 백엔드 집계 레이어 부재 |

> 어떤 프론트엔드를 만들더라도 고객은 정보 탐색·사후 인지·수동 재계산이라는
> 3가지 시간 낭비를 반복할 수밖에 없다.
