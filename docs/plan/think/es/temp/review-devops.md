# 이벤트 스토밍 리뷰 - 파이프 (DevOps 엔지니어)

> 리뷰 관점: 배포, 확장성, 운영 안정성, 비용 최적화
> 리뷰일: 2026-02-22

---

## 긍정 평가

- **모니터링 서비스의 단일 책임 분리가 적절하다.**
  데이터 수집과 상태 판정을 하나의 서비스로 묶되, 독립 서비스로 분리한 결정은 타당하다. 이 서비스는 "방문 예정 시간 2시간 전부터 15분 주기 수집"이라는 높은 부하 변동 특성을 가지며, HPA(Horizontal Pod Autoscaler) 대상으로 가장 먼저 지정해야 할 컴포넌트다. 독립 배포 단위로 만들어 둔 덕분에 스케일 인/아웃이 다른 서비스에 영향을 주지 않는다.

- **비동기 이벤트 경로와 동기 조회 경로를 명확히 구분했다.**
  컨텍스트 맵(5-2)에서 일정 서비스 -> 모니터링 서비스 경로는 비동기, 브리핑/대안 서비스 -> 모니터링/장소 서비스 경로는 동기로 구분한 설계는 운영 측면에서 장애 전파 범위를 사전에 의식했음을 보여준다. 비동기 경로가 있다는 것 자체가 일정 변경 시 모니터링 서비스 장애가 일정 서비스 가용성에 직접 영향을 주지 않음을 의미한다.

- **API 실패 정책(P3)이 명시되어 있다.**
  "API 실패 시 최근 캐시 사용, 3회 연속 실패 시 회색 상태"라는 명시적 정책(`02-외부데이터수집.puml` note)은 외부 의존성 장애를 앱 레벨에서 격리하는 Graceful Degradation 패턴이다. 운영 중 외부 API 장애 시 전체 서비스가 멈추지 않고 "회색(미확인)" 상태로 계속 서비스할 수 있어 SLA 유지에 유리하다.

- **브리핑+알림 통합 결정(MVP 통합)이 현실적이다.**
  MVP 단계에서 알림 채널이 FCM Push 1종뿐인 상황에서 브리핑 서비스와 알림을 합친 결정은 운영 복잡도를 줄이는 올바른 선택이다. Phase 2에서 SMS/카카오톡 채널 추가 시 분리한다는 확장 로드맵까지 명시되어 있어 기술 부채 인식도 적절하다.

- **외부 시스템 의존성 목록이 한 곳에 정리되어 있다.**
  Google Places, OpenWeatherMap, Google Popular Times, Google Directions, FCM 5개 외부 시스템이 명시되어 있어 API Key 관리, Secret 로테이션, 비용 모니터링 알람 설정 대상을 즉시 파악할 수 있다.

## 개선 의견

### 1. 스케줄러 단일 장애점(SPOF) 문제 — `02-외부데이터수집.puml`, `05-출발전브리핑생성.puml`

현재 설계에서 "스케줄러"는 단일 참여자로 표현되어 있으나, 두 가지 트리거(15분 주기 수집, 출발 전 15~30분 브리핑)를 모두 담당한다. 이 스케줄러가 단일 Pod로 운영되면 재시작 시 발화 누락이 발생하고, 다중 Pod로 운영하면 동일 잡이 중복 실행된다.

**제안:** Kubernetes CronJob 대신 분산 잡 큐(예: BullMQ + Redis, 또는 Kubernetes Job with leader election)를 도입하고, 스케줄러 컴포넌트를 모니터링 서비스 내 독립 Deployment로 분리하여 `spec.replicas: 1` + `livenessProbe`로 SPOF를 관리하거나, Redlock 기반 분산 락을 적용해 중복 실행을 방지해야 한다.

### 2. 동기 호출 연쇄가 만드는 장애 전파 경로 — `05-출발전브리핑생성.puml`, `event-storming-결과.md` 5-2

브리핑 생성 흐름을 보면 `브리핑업무 -> 데이터수집업무(동기) -> 상태판정업무(동기)`의 연쇄 동기 호출이 발생한다. 대안 서비스도 `대안 -> 장소 서비스(동기) -> 모니터링 서비스(동기)`를 두 번 연쇄 호출한다. 어느 한 서비스의 응답 지연이 전체 체인 타임아웃으로 이어지는 구조다.

**제안:** 브리핑 생성 시점에 필요한 "최신 데이터"는 이미 15분 주기로 수집된 데이터를 캐시에서 읽는 것으로 충분하다. 브리핑 서비스가 모니터링 서비스에 동기 요청을 보내는 대신, 수집된 데이터를 Redis 또는 이벤트 스트림(Kafka/NATS)에서 직접 읽는 방식으로 전환하면 동기 체인 깊이를 줄일 수 있다. 최소한 Timeout + Retry 정책과 Circuit Breaker(Resilience4j 또는 서비스 메시 수준)를 각 동기 호출에 명시해야 한다.

### 3. 외부 API 호출 볼륨과 비용 추정이 누락되어 있다 — `02-외부데이터수집.puml`, `event-storming-결과.md` 8

한 장소당 15분 주기로 4개 외부 API를 호출한다. 사용자 1명이 3일 여행에 장소 5개를 등록하면, 장소별 모니터링 활성화 구간이 방문 2시간 전이므로 장소당 최대 8회 주기 발화 x 4 API = 32회 호출이다. 동시 접속 사용자 1,000명, 장소 5개 기준이면 피크 시 분당 약 1,333회 외부 API 호출이 발생한다. Google Places API 무료 티어는 월 $200 크레딧이며, 피크 트래픽에서 일 수만 건 호출 시 비용이 선형으로 증가한다.

**제안:** 이벤트 스토밍 결과에 다음 비용 제어 장치를 추가해야 한다.
- 동일 장소 ID에 대한 수집 결과를 TTL 15분으로 Redis에 캐싱하여 중복 호출 방지
- 날씨/교통 API는 좌표 기반 Grid 단위로 집계 캐싱 (동일 블록 내 여러 장소 공유 가능)
- Google Popular Times(혼잡도)는 실시간 API가 아닌 Historical 데이터이므로 일 1회 수집으로 충분한지 검토 필요

### 4. 상태 변경 알림 전달 경로의 신뢰성 — `03-실시간상태모니터링.puml`

상태 변경 시 `상태판정업무 -> 알림업무`로 이벤트가 전달되는데, 현재 다이어그램에서 이 호출이 동기인지 비동기인지 불명확하다. 만약 동기 호출이면 FCM 응답 지연이 상태 판정 루프를 블로킹한다. 또한 알림 발송 실패 시 재처리 메커니즘이 정의되어 있지 않다.

**제안:** 상태 변경 이벤트 발행을 메시지 브로커(Redis Streams 또는 Kafka)를 통한 비동기 발행으로 변경하고, 알림 서비스는 이를 구독하여 독립적으로 처리해야 한다. 발송 실패 시 Dead Letter Queue + 재시도 정책(Exponential Backoff, 최대 3회)을 명시해야 한다.

### 5. MVP에서 5개 서비스가 과한가 — 모놀리스 우선 접근 검토

현재 5개 마이크로서비스 구조는 K8s 배포를 전제로 하면 최소 5개 Deployment, 5개 Service, 5개 ConfigMap/Secret, API Gateway 또는 Ingress, 메시지 브로커 1종이 필요하다. MVP 팀이 소규모라면 이 인프라 운영 오버헤드가 제품 개발 속도를 저해할 수 있다.

**제안 (단계적 접근):**
- MVP v0: 단일 모노레포 NestJS 모노리스로 시작하되, 내부 모듈 경계를 바운디드 컨텍스트 기준으로 엄격하게 유지한다. 단일 Dockerfile, 단일 K8s Deployment로 배포한다.
- MVP v1: 부하가 가장 높은 모니터링 서비스만 먼저 분리한다. 나머지 4개 컨텍스트는 모노리스에 유지한다.
- Phase 2 이후: 트래픽 데이터를 기반으로 추가 분리 결정을 내린다.

이 접근법은 "Fast fail, Learn and Pivot" 팀 원칙에도 부합한다.

## 누락 사항

### N1. 헬스체크 및 Readiness/Liveness Probe 정의 부재

5개 서비스 각각에 대해 K8s `livenessProbe`(서비스 생존 여부)와 `readinessProbe`(트래픽 수용 가능 여부)가 설계에 포함되어 있지 않다. 특히 모니터링 서비스는 외부 API 연결 가능 여부를 Readiness 판단에 포함해야 한다. 스케줄러가 미준비 Pod에 잡을 발화하면 수집이 무음으로 누락된다.

### N2. 배포 환경별 설정 관리(Config Management) 전략 부재

개발/스테이징/프로덕션 환경별로 외부 API 엔드포인트, API Key, FCM 프로젝트 ID가 달라야 한다. 현재 설계에 환경별 설정 분리 전략(Kubernetes ConfigMap/Secret, Vault, 환경변수 주입 방식)이 명시되어 있지 않다. API Key가 컨테이너 이미지에 하드코딩되는 사고를 사전에 방지해야 한다.

### N3. 분산 추적(Distributed Tracing) 및 로그 상관관계 식별자 부재

브리핑 생성 한 건이 브리핑 서비스 -> 모니터링 서비스 -> 장소 서비스를 거치는 다중 서비스 흐름을 보인다. 장애 발생 시 어느 서비스에서 지연이 발생했는지 파악하려면 `TraceId`/`SpanId`를 요청 헤더로 전파하는 분산 추적(OpenTelemetry + Jaeger/Tempo)이 필수적이다. 현재 이벤트 스토밍 설계에는 이 관점이 전혀 없다.

### N4. 스케줄러 잡 누락 감지(Job Miss Detection) 메커니즘 부재

"출발 전 15~30분 브리핑 트리거"는 정시성이 핵심이다. Pod 재시작, 노드 교체, GC Pause 등으로 스케줄러가 특정 시간대에 발화를 놓쳤을 때 이를 감지하고 보정하는 메커니즘이 없다. 예를 들어 출발 20분 전 브리핑이 누락되면 사용자는 알림을 받지 못하고 그냥 출발하게 된다. Missed Job 감지용 Watchdog 또는 브리핑 발송 이력을 기반으로 한 보정 스캔 로직이 필요하다.

### N5. 사용자 인증/인가 서비스 및 API Gateway 설계 부재

5개 마이크로서비스가 각각 클라이언트의 요청을 직접 받는지, API Gateway를 통해 라우팅되는지 명시되어 있지 않다. JWT 검증, Rate Limiting, CORS 처리가 각 서비스에 중복 구현될 경우 유지보수 비용이 높아진다. API Gateway(NGINX Ingress, Kong, AWS API Gateway 등) 레이어가 이벤트 스토밍 외부 시스템 목록에 포함되어야 한다.

### N6. 데이터 보존 정책 및 스토리지 용량 계획 부재

모니터링 서비스는 15분 주기로 수집 데이터를 저장한다. 사용자 1명, 장소 5개 기준으로 여행 3일간 발생하는 수집 레코드는 약 1,440건이다. 사용자 10만 명 규모에서 장기 보존 시 스토리지 비용이 급증한다. 수집 원시 데이터의 TTL(예: 7일 후 삭제)과 상태 이력의 보존 기간(예: 여행 종료 후 30일)을 이벤트 스토밍 정책 목록에 추가해야 한다.
