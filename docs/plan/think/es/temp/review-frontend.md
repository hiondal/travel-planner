# 이벤트 스토밍 리뷰 - 데브-프론트 (프론트엔드 개발자)

> 리뷰 관점: 프론트엔드 인터랙션, 상태 관리, 접근성
> 리뷰일: 2026-02-22

---

## 긍정 평가

- **"열고, 보고, 탭한다" 3단계 UX 명시** (`08-대안카드선택및일정반영.puml`): 대안 카드 인터랙션을 단 3단계로 제약한 것은 모바일 환경에서 인지 부하를 최소화하는 탁월한 설계다. 프론트엔드 구현 관점에서도 낙관적 업데이트(Optimistic Update)를 적용하기 쉬운 단순한 흐름이다.

- **상태 배지 4색 체계와 회색(미확인) 명시** (`event-storming-결과.md` P3/P4/P6): 초록/노랑/빨강/회색의 4단계 상태와 "회색 = 데이터 미확인" 정책(P6)이 명확하다. 이는 Skeleton UI 및 에러 상태 UI를 별도 설계할 필요 없이 회색 배지가 그 역할을 겸할 수 있어 UI 상태 가짓수가 줄어든다.

- **안심/주의 브리핑 텍스트 명시** (`06-출발전브리핑수신.puml`): Push 알림 제목을 "다음 목적지 준비 완료" / "다음 목적지 확인 필요"로 명시한 것은 좋다. 알림 딥링크(deep link) 목적지 URL 설계 시 양 갈래 분기를 명확하게 구현할 수 있다.

- **대안 카드 → 일정 교체 → 모니터링 재등록의 연쇄 흐름 명시** (`08-대안카드선택및일정반영.puml`): 대안 선택 후 신규 장소가 자동으로 모니터링 대상에 등록되는 P14 정책이 시퀀스에 명시되어 있다. 프론트엔드는 일정 반영 완료 후 해당 장소의 배지 상태를 즉시 "회색(수집 대기)"으로 표시하는 낙관적 처리 기준이 생긴다.

- **상태 상세 팝업의 데이터 구조 명시** (`04-상태배지조회.puml`): 상태 상세 정보 응답에 영업상태, 혼잡도, 날씨, 이동시간, 판정사유가 포함됨을 명시하고 있어 팝업 컴포넌트의 Props 인터페이스를 바로 설계할 수 있다.

## 개선 의견

### 1. 상태 배지 갱신 통신 방식이 미정의 (`04-상태배지조회.puml`, `event-storming-결과.md` P2)
`userflow.puml`에서 "상태 변경 시 배지 갱신" 흐름이 있고 P2 정책에서 15분 주기 수집이 명시되어 있지만, **프론트엔드가 갱신된 배지를 어떻게 수신하는지**가 전혀 정의되어 있지 않다. Polling, WebSocket, SSE 세 가지 선택지 중 결정이 필요하다.

- **권장 방식: SSE(Server-Sent Events)** - 배지 상태 변경은 단방향(서버→클라이언트)이고 15분 주기의 저빈도 이벤트이므로, WebSocket의 양방향 오버헤드 없이 SSE로 충분하다. 단, 모바일 브라우저의 백그라운드 탭에서 SSE 연결이 끊기는 문제가 있으므로, 앱이 포어그라운드로 돌아올 때(Visibility Change 이벤트) 재연결 + 전체 배지 상태를 한 번 폴링하는 하이브리드 전략을 권장한다.
- 이벤트 스토밍 결과에 `[클라이언트 수신 방식]` 노트 한 줄을 각 상태 변경 이벤트에 추가하면 백엔드·프론트엔드 구현 계약이 명확해진다.

### 2. Push 알림 수신 후 앱 내 네비게이션 딥링크 흐름 누락 (`06-출발전브리핑수신.puml`)
현재 시퀀스는 "Push 알림 수신됨 → 브리핑 상세 조회 요청"으로 단순히 표현되어 있으나, 실제 프론트엔드에서는 다음 세 가지 케이스를 모두 처리해야 한다.

- **케이스 A**: 앱이 포어그라운드 상태에서 알림 배너를 탭 → 현재 화면에서 브리핑 모달을 슬라이드업
- **케이스 B**: 앱이 백그라운드 상태에서 알림을 탭 → 앱 재개(resume) 후 브리핑 상세 화면으로 라우팅
- **케이스 C**: 앱이 완전히 종료된 상태에서 알림을 탭 → 앱 콜드 스타트 후 딥링크 `/briefing/:briefingId`로 직행

이 세 케이스의 분기 처리가 시퀀스에 없으면 프론트엔드 개발 시 케이스 C(콜드 스타트)를 놓칠 위험이 높다. `06-출발전브리핑수신.puml`에 `alt` 블록으로 추가를 권장한다.

### 3. 대안 카드 선택 중 낙관적 업데이트 실패 롤백 흐름 누락 (`08-대안카드선택및일정반영.puml`)
"탭 한 번으로 일정 반영" P12 정책을 만족하려면 프론트엔드는 탭 즉시 UI를 변경(Optimistic Update)해야 한다. 그러나 현재 시퀀스에는 `대안검색업무 → 일정업무` 교체 요청이 실패하는 케이스가 없다. 네트워크 오류, 일정 서비스 장애, 동시 교체 충돌 등 실패 시 다음이 필요하다.

- 변경된 일정을 원래 상태로 되돌리는 롤백 UI 처리
- 사용자에게 재시도 가능한 Toast/Snackbar 에러 메시지 표시
- 시퀀스 내 `else` 실패 분기 추가

### 4. 색상 배지만으로 상태 전달 — 색약 사용자 대응 미정의 (`04-상태배지조회.puml`, `event-storming-결과.md` P4)
현재 상태 판정 정책(P4)은 **초록/노랑/빨강/회색** 색상 코드만 정의하고 있다. 적록색약(Deuteranopia, 남성의 약 8%) 사용자는 빨강과 초록을 구별하기 어렵다. WCAG 2.1 Success Criterion 1.4.1(Use of Color)은 색상만을 유일한 시각적 구별 수단으로 사용하는 것을 금지한다.

- 배지 컴포넌트에 색상 외 **보조 레이블 텍스트** 또는 **아이콘 심볼**을 함께 표시해야 한다. 예: 초록(체크 아이콘), 노랑(느낌표 아이콘), 빨강(X 아이콘), 회색(물음표 아이콘)
- 이벤트 스토밍 정책 P4에 "배지는 색상 + 아이콘 조합으로 표시한다" 조건을 추가할 것을 제안한다.

### 5. 오프라인/네트워크 불안정 상태의 UI 대응 전략 누락 (전체 흐름)
이벤트 스토밍 전체에서 "네트워크가 없을 때 프론트엔드가 어떻게 동작하는가"에 대한 정의가 없다. P3 정책(API 3회 실패 시 회색 상태)은 백엔드 관점의 정책이고, 프론트엔드 관점의 오프라인 대응이 별도로 필요하다.

- **배지 조회 실패 시**: 마지막으로 캐시된 배지 상태를 표시하되 "X분 전 기준" 타임스탬프를 함께 표시
- **대안 카드 탭 후 네트워크 단절 시**: 스피너 + 타임아웃 후 "연결 실패, 다시 시도" 인라인 메시지
- **앱 전체 오프라인 진입 시**: 상단 고정 배너("오프라인 상태 - 마지막 동기화 HH:MM")로 사용자에게 명확히 고지

이 정책들을 이벤트 스토밍 정책 목록(섹션 7)에 P15~P17로 추가하여 프론트엔드 구현 기준으로 삼을 것을 제안한다.

## 누락 사항

### A. 상태 배지 스켈레톤 UI / 로딩 상태의 미정의
`04-상태배지조회.puml`에서 일정표 조회 후 배지 목록이 반환되기 전 중간 상태(로딩 중)에 대한 정의가 없다. 배지가 없는 빈 상태로 잠깐 표시되었다가 갑자기 채워지는 화면 점프(layout shift)는 사용자 불안감을 유발한다. "배지 로딩 중" 상태를 회색 + 펄스 애니메이션 Skeleton으로 처리하는 정책이 필요하다.

### B. 브리핑 상세 화면의 "대안 보기" 버튼 노출 조건이 불명확
`06-출발전브리핑수신.puml`의 `else 주의/위험 브리핑` 분기에서 "대안 보기 요청"이 있지만, 브리핑 상세 화면 UI에서 해당 버튼이 항상 표시되는지, 아니면 위험 항목이 있는 경우에만 표시되는지 조건이 명시되지 않았다. 프론트엔드 컴포넌트의 조건부 렌더링 기준이 모호하다.

### C. 대안 카드 3장 중 선택하지 않은 나머지 카드의 상태 관리 누락
`08-대안카드선택및일정반영.puml`에서 사용자가 카드 1을 탭하면 일정이 반영되는 흐름은 명확하다. 그러나 카드 2, 3은 어떻게 처리되는가(화면에서 사라지는가, 비활성화되는가, 유지되는가)에 대한 정의가 없다. 선택 후 나머지 카드를 비활성화하고 "N초 후 화면 닫힘" 카운트다운 처리가 필요한지 여부가 프론트엔드 구현 판단에 영향을 준다.

### D. 브리핑 수신 시 이미 앱 화면이 열려있는 경우의 인앱 알림 처리 누락
`06-출발전브리핑수신.puml`은 Push 알림 경로만 정의했다. 사용자가 앱을 사용 중일 때 브리핑 트리거 시점이 도래하면, Push 알림 외에 인앱 알림(In-App Notification Banner 또는 모달)으로도 브리핑을 알려야 한다. 이 경우의 UI 인터랙션(자동 표시 vs 알림 아이콘 배지로 대체)이 정의되어 있지 않다.

### E. 일정 교체 완료 후 후속 이동시간 재계산 결과의 프론트엔드 표시 방식 미정의
P13 정책(후속 일정 이동시간 자동 재계산)은 백엔드 동작으로 명시되어 있지만, 재계산된 이동시간이 프론트엔드에 어떻게 전달되고 표시되는지 정의가 없다. 일정 반영 완료 응답 payload에 `updatedTravelTimes` 배열이 포함되어야 하며, 일정표 UI에서 변경된 이동시간 항목을 하이라이트(예: 노란색 배경 2초 후 정상 복귀)하는 피드백이 필요하다.
