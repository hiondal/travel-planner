# 이벤트 스토밍 리뷰 - 도그냥 (서비스 기획자)

> 리뷰 관점: UX 흐름, 사용자 시나리오, 인터랙션 설계
> 리뷰일: 2026-02-22

---

## 긍정 평가

### 1. "인지(Pull) -> 알림(Push) -> 행동(Action)" 통합 UX 흐름이 유저플로우에 정확히 반영되어 있다

`userflow.puml`에서 S05(상태 배지 조회) -> S04(출발 전 브리핑 수신) -> S06(대안 카드 선택)으로 이어지는 흐름이 핵심솔루션 문서의 통합 UX 흐름과 일관된다. 여행자가 일정표를 "열고"(Pull), 브리핑을 "보고"(Push), 대안 카드를 "탭한다"(Action)는 3단계 UX가 유저플로우 간 연결 구조에 명확히 드러난다.

### 2. 안심 브리핑의 습관 형성 흐름이 시퀀스에 잘 설계되어 있다

`05-출발전브리핑생성.puml`에서 종합 상태가 "정상"일 때에도 안심 브리핑을 생성하는 분기가 명시되어 있고, `06-출발전브리핑수신.puml`에서도 안심 브리핑 수신 시 "알림 = 안심 신호" 습관 형성 정책이 노트로 기재되어 있다. 정상 시에도 Push를 보내는 결정이 시퀀스 레벨까지 내려와 있어, 개발 시 "정상이면 알림 안 보내도 되지 않나?"라는 질문을 선제적으로 차단한다.

### 3. 신호등 메타포가 정책/규칙으로 일관되게 관리된다

`03-실시간상태모니터링.puml`에서 초록/노랑/빨강/회색 4단계 판정 기준이 명시되어 있고, `04-상태배지조회.puml`에서 회색 배지의 "데이터 미확인" 투명 표시 정책, `07-대안장소검색.puml`에서 후보 장소 상태 조회 시 "초록만 필터"하는 규칙까지 신호등 메타포가 전체 흐름에 걸쳐 일관된다. 특히 P6(회색 = 데이터 미확인 투명 표시) 정책은 서비스 신뢰도 관점에서 좋은 설계이다.

### 4. 대안 카드 선택 후 순환 루프가 완결되어 있다

`08-대안카드선택및일정반영.puml`에서 대안 장소 선택 -> 일정 교체 -> 새 장소 모니터링 등록으로 이어지고, `userflow.puml`에서도 08번 유저플로우가 03번(실시간 상태 모니터링)으로 다시 연결된다. 사용자가 대안을 선택한 후에도 새 장소에 대한 상태 감시가 자동으로 시작되므로, "대안 장소도 문제가 생기면 어떡하지?"라는 사용자 불안을 구조적으로 해소한다.

### 5. 정책/규칙이 시퀀스 다이어그램 내에 인라인으로 배치되어 있다

14개 정책/규칙(P1~P14)이 결과 문서에 정리되어 있을 뿐 아니라, 각 시퀀스 다이어그램의 해당 시점에 `note right` 형태로 직접 배치되어 있다. 개발자가 시퀀스를 따라가면서 자연스럽게 비즈니스 규칙을 확인할 수 있어 "문서는 따로, 코드는 따로" 문제를 예방한다.

## 개선 의견

### 1. [01-여행일정등록.puml] 장소 검색 결과에서 여행자의 "선택" 인터랙션이 누락되어 있다

현재 시퀀스에서 장소업무가 Google Places로부터 "장소 목록"을 반환받은 후, 바로 일정업무로 "장소 정보 조회 완료됨"이 전달된다. 그러나 실제 UX에서는 검색 결과 중 여행자가 원하는 장소를 **선택하는 단계**가 반드시 필요하다. 검색어 "카페"를 입력하면 여러 카페가 나올 텐데, 어떤 카페를 일정에 추가할지 여행자가 결정하는 인터랙션이 빠져 있다.

**제안**: `장소업무 --> 여행자 : [이벤트] 장소 검색 결과 제시됨` -> `여행자 -> 장소업무 : [커맨드] 장소 선택` 단계를 추가할 것.

### 2. [04-상태배지조회.puml] 배지 상세 조회 시 여행자가 상태판정업무에 직접 접근하는 구조가 UX와 맞지 않는다

현재 시퀀스에서 여행자가 일정표를 조회할 때는 "일정업무"를 통해 접근하지만, 배지를 탭하여 상세 정보를 볼 때는 "상태판정업무"에 직접 접근한다. 사용자 관점에서 이 두 행위는 모두 같은 일정표 화면 안에서 일어나는 것이다. 프론트엔드 구현 시 API 호출 구조가 달라질 수 있고, 사용자 경험의 일관성 관점에서도 일정업무가 중개하는 것이 자연스럽다.

**제안**: 상태 상세 조회도 `여행자 -> 일정업무 -> 상태판정업무` 경로로 통일할 것. 화면 단위와 API 게이트웨이 구조를 일치시키는 것이 UX-시스템 정합성에 유리하다.

### 3. [06-출발전브리핑수신.puml] 브리핑 알림을 "무시"하거나 "나중에 보기"하는 시나리오가 없다

현재 시퀀스는 여행자가 Push 알림을 수신한 후 반드시 "브리핑 상세 조회"를 하는 것으로 설계되어 있다. 그러나 실제 여행 중에는 알림을 바로 확인하지 못하는 상황이 빈번하다(이동 중, 식사 중, 수면 중 등). 알림을 놓쳤을 때 앱을 열면 어디서 브리핑을 다시 확인할 수 있는지, 브리핑의 유효 시간은 언제까지인지에 대한 흐름이 없다.

**제안**: (1) 앱 내 "브리핑 히스토리" 또는 일정표에서 해당 장소의 최근 브리핑을 확인할 수 있는 재진입 경로 추가, (2) 브리핑 유효 시간 정책(예: 출발 예정 시간 경과 후 자동 만료) 정의 필요.

### 4. [07-대안장소검색.puml] 대안 카드가 3장 미만일 때의 예외 흐름이 없다

P10 정책(동일 카테고리, 반경 1km, 영업 중 필터)과 초록 상태만 필터하는 조건을 모두 적용하면, 특히 야간이나 소도시에서는 후보가 3개 미만이거나 0개일 수 있다. 현재 시퀀스는 항상 "대안 카드 3장 생성됨"으로 끝나는데, 0~2장인 경우의 사용자 안내가 설계되어 있지 않다.

**제안**: (1) 후보 0건: "현재 조건에 맞는 대안이 없습니다. 검색 범위를 넓혀볼까요?" 안내 + 반경 확대 옵션, (2) 후보 1~2건: 있는 만큼만 제시하되 "추가 검색" 버튼 제공, (3) 카테고리를 유사 카테고리로 확장하는 폴백 로직 정책 추가.

### 5. [08-대안카드선택및일정반영.puml] "일정 반영 취소(되돌리기)" 시나리오가 없다

P12(탭 한 번으로 일정 반영)는 효율 추구형 사용자에게 매우 강력한 UX이지만, 실수로 잘못된 카드를 탭했을 때 되돌릴 방법이 시퀀스에 없다. "열고, 보고, 탭한다"의 빠른 UX일수록 실수 복구 경로가 중요하다. 특히 후속 일정 이동시간까지 자동 재계산(P13)되므로, 잘못된 선택의 파급 효과가 크다.

**제안**: (1) 일정 반영 직후 "변경 완료" 토스트에 "되돌리기" 버튼을 3~5초간 표시, (2) 되돌리기 시 원래 장소로 복원 + 모니터링 재등록하는 역방향 플로우 추가.

## 누락 사항

### 1. 여행 일정 수정/삭제 시나리오가 없다

01번 유저플로우에서 일정 "생성"과 장소 "추가"는 있지만, 여행자가 직접 장소를 삭제하거나 방문 순서를 변경하는 흐름이 없다. 실제 여행 준비 과정에서 일정 수정은 매우 빈번하게 발생하며, 장소 삭제 시 해당 장소의 모니터링 해제, 후속 일정 이동시간 재계산 등 연쇄 이벤트가 필요하다.

### 2. 여행자의 알림 설정/해제 시나리오가 없다

브리핑 Push 알림(S04)은 핵심 기능이지만, 여행자가 알림을 끄거나 시간대를 조정하는(예: 수면 시간 알림 차단) 설정 흐름이 없다. 알림이 과도하면 앱 삭제로 이어질 수 있으므로, 알림 빈도/시간대 제어는 리텐션에 직결된다.

### 3. 대안 카드를 "모두 거절"하는 시나리오가 없다

07~08번 유저플로우에서 대안 카드 3장이 제시된 후, 여행자가 3장 모두 마음에 들지 않을 때의 흐름이 없다. 이 경우 (1) 직접 검색으로 전환, (2) 원래 일정 유지, (3) 해당 시간대를 비워두기 등의 선택지가 필요하다. 현재는 반드시 카드를 선택해야만 흐름이 진행되는 구조여서 Dead End가 발생한다.

### 4. 초록 -> 초록 유지 상태에서의 사용자 경험이 정의되어 있지 않다

P5에서 상태 악화(초록->노랑, 초록->빨강, 노랑->빨강) 시 알림을 트리거하지만, 반대로 상태가 개선되는 경우(빨강->노랑, 노랑->초록)의 알림 정책이 없다. 예를 들어 "폭우 예보가 해제되었습니다"와 같은 긍정적 상태 변화 알림은 안심 경험을 강화할 수 있다. 또한 여행 기간 내내 모든 장소가 초록 상태를 유지하는 "아무 일도 일어나지 않는" 시나리오에서 서비스 가치를 어떻게 체감시킬 것인지에 대한 설계가 필요하다.

### 5. 온보딩 및 최초 사용 시나리오가 없다

여행자가 앱을 처음 실행했을 때 신호등 배지 메타포를 이해시키고, 알림 권한을 획득하고, 첫 여행 일정을 등록하도록 유도하는 흐름이 유저플로우에 포함되어 있지 않다. 신호등 메타포가 "학습 비용 제로"라고 했지만, 회색(미확인)의 의미나 브리핑 알림의 가치는 최초 경험에서 안내가 필요하다. 특히 Push 알림 권한 획득은 OS 레벨 퍼미션이므로, 권한 요청 타이밍과 사전 설명 화면이 MVP에서도 중요하다.

## 총평

아키가 작성한 이벤트 스토밍은 핵심 솔루션 3개(S04/S05/S06)의 시스템 이벤트 흐름을 체계적으로 분석했으며, 특히 "인지 -> 알림 -> 행동"의 통합 UX 흐름과 신호등 메타포의 일관성은 높이 평가한다. 다만 서비스 기획 관점에서 보면, 시스템 간 이벤트 흐름(Happy Path)은 잘 정의되어 있으나, **사용자가 예상과 다르게 행동하는 시나리오**(알림 무시, 카드 거절, 실수 복구, 설정 변경)에 대한 흐름이 부족하다. MVP에서 모든 예외를 다룰 수는 없지만, 최소한 "대안 카드 거절"과 "일정 반영 되돌리기"는 Dead End를 방지하기 위해 반드시 추가되어야 한다.
