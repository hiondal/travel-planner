# 이벤트 스토밍 리뷰 - 마법사 (AI 엔지니어)

> 리뷰 관점: AI 활용 기회, 지능화 가능성, 데이터 파이프라인
> 리뷰일: 2026-02-22

---

## 긍정 평가

- **상태 판정 로직의 단계적 고도화 경로가 명확하다.**
  `03-실시간상태모니터링.puml`의 초록/노랑/빨강/회색 4단계 분류 체계는 규칙 기반으로 시작하면서 이후 ML 분류기로 교체하기에 이상적인 인터페이스 설계다. 판정 결과를 Aggregate(`PlaceStatus`)로 격리해 둔 덕분에 내부 알고리즘을 교체해도 외부 인터페이스가 흔들리지 않는다.

- **`SearchContext` Aggregate 설계가 개인화 학습의 씨앗이다.**
  클러스터 7(대안 검색)에서 `AlternativeCard`와 함께 `SearchContext`를 별도 Aggregate로 두었다. 검색 당시의 맥락(카테고리, 위험 사유, 위치, 시간대)이 구조화된 형태로 남아 향후 사용자 선택 이력과 결합하면 개인화 추천 모델의 학습 데이터로 바로 활용 가능하다.

- **안심 브리핑 발송(P8) 정책이 행동 데이터 수집 루프를 만든다.**
  `05-출발전브리핑생성.puml`에서 정상 상황에도 브리핑을 발송하는 정책(P8)은 단순한 UX 습관 형성 이상의 효과를 가진다. 모든 출발 전 상황에 대한 사용자 반응(열람, 무시, 대안 탭)이 축적되어 "어떤 상황에서 어떤 사용자가 어떤 행동을 취하는가"에 대한 레이블된 데이터셋이 자연스럽게 쌓인다.

- **5개 마이크로서비스 경계가 AI 기능 추가에 적합하다.**
  모니터링/브리핑/대안 서비스가 분리되어 있어, 각 서비스에 AI 모듈을 점진적으로 삽입할 수 있다. 특히 Phase 3에 "AI 재조정 서비스"를 별도 추가하는 확장 로드맵(6-4절)은 AI 기능을 코어 도메인과 격리해 실험하기 좋은 구조다.

- **외부 데이터 소스가 4종(Places, Weather, Popular Times, Directions)으로 다양하다.**
  멀티소스 수집 구조는 단일 지표 의존 없이 복합 신호 기반 판단을 가능하게 한다. 이는 Anomaly Detection(예: "혼잡도 급상승 + 날씨 이상 + 이동시간 급증이 동시에 발생")과 같은 다변량 이상 감지의 토대가 된다.

## 개선 의견

- **[03-실시간상태모니터링.puml] 상태 판정 로직에 신뢰도 점수(Confidence Score)를 추가해야 한다.**
  현재 정책(P4)은 "1개 이상 주의 -> 노랑"처럼 항목 수 기반의 단순 규칙이다. 예를 들어 혼잡도 데이터의 신뢰도가 낮은 상태에서 노랑 판정이 나면 사용자에게 불필요한 불안을 줄 수 있다. 이벤트 데이터에 `confidence_score: Float` 필드를 추가하고, 상태 변경 이벤트에 "판정 근거 항목 목록"을 포함할 것을 제안한다. MVP에서는 규칙 기반 신뢰도(데이터 수집 성공 비율, 캐시 나이)로 시작하고 이후 ML 기반 신뢰도 추정으로 고도화한다.

- **[07-대안장소검색.puml] 대안 카드 3장 선정 기준(P11)이 고정 가중치 랭킹으로 설계되어 있다.**
  "거리 > 평점 > 혼잡도" 우선순위는 모든 사용자, 모든 상황에 동일하게 적용된다. 그러나 솔로 트래블러는 이동 효율보다 분위기나 평점을 더 중시할 수 있고, 상황(비가 오는 날 vs 맑은 날)에 따라 가중치가 달라져야 한다. MVP에서는 현행 고정 가중치를 유지하되, 대안 카드 선택/미선택 이벤트에 `rank_at_selection: Int`와 `rejection_reason: Enum` 필드를 기록해 두면 이후 학습 기반 개인화 랭킹으로 전환하는 데 필요한 레이블 데이터를 확보할 수 있다.

- **[05-출발전브리핑생성.puml] 브리핑 텍스트 생성 로직이 완전히 규칙 기반으로 가정되어 있다.**
  "위험 항목, 주의 사유, 권장 행동, 총평" 데이터가 구조화되어 있지만, 이를 자연어 브리핑으로 렌더링하는 흐름이 다이어그램에 없다. 두 가지 접근이 가능하다. (1) 단기: 템플릿 기반 문장 조합, (2) 중기: Claude API를 활용한 자연어 브리핑 생성. `브리핑업무`와 실제 텍스트 렌더링 책임 사이에 `브리핑콘텐츠생성` 단계를 명시적으로 분리하는 커맨드/이벤트를 추가하면, 나중에 AI 생성으로 교체할 때 서비스 로직을 건드리지 않아도 된다.

- **[event-storming-결과.md, 클러스터 3] 수집 데이터 저장 이벤트에 데이터 품질 지표가 없다.**
  `수집 데이터 저장됨` 이벤트의 데이터 필드가 "장소 ID, 수집 시각"에 그친다. AI 이상 감지 또는 예측 모델을 적용하려면 수집 시점의 `source_latency_ms`, `is_cached: Boolean`, `data_age_seconds: Int` 같은 메타데이터가 원시 이벤트에 함께 기록되어야 한다. MVP에서 이 필드를 누락하면 나중에 히스토리 데이터를 소급하여 품질 보정하는 비용이 매우 커진다.

- **[event-storming-결과.md, 클러스터 4/5] 브리핑과 상태 판정 이벤트에 사용자 반응 피드백 루프가 없다.**
  현재 이벤트 흐름은 시스템 -> 사용자 방향의 단방향 출력만 다루고 있다. 브리핑을 수신한 사용자가 "그냥 간다(무시)", "대안을 본다", "일정을 취소한다"는 행동 이벤트가 시스템에 회수되어야 AI 모델의 정확도를 평가하고 개선할 수 있다. 최소한 `브리핑열람됨`, `브리핑무시됨`, `대안탐색시작됨` 이벤트를 명시적으로 클러스터에 추가하고 `Briefing` Aggregate에 `user_response: Enum` 필드를 포함할 것을 제안한다.

## 누락 사항

- **사용자 여행 스타일 프로파일 이벤트가 없다.**
  MVP에서 개인화 랭킹 모델을 위한 콜드 스타트 문제를 해결하려면 초기 여행 성향 수집(일정 등록 시 간단한 선호도 선택: "효율 우선 / 분위기 우선 / 가성비 우선")이 필요하다. 해당 이벤트가 클러스터 1(일정 관리)에 없으며, `Trip` Aggregate에도 여행 스타일 속성이 보이지 않는다. 이 데이터가 없으면 대안 추천 개인화는 클릭 이력이 수십 건 쌓일 때까지 불가능하다.

- **상태 판정 히스토리(`StatusHistory`)의 보존 정책과 집계 주기가 정의되지 않았다.**
  Aggregate로는 `StatusHistory`가 명시되어 있으나, 얼마나 오래, 어떤 해상도로 보존할지 정책이 없다. 시계열 이상 감지(예: 특정 장소가 매주 금요일 오후에 반복적으로 빨강이 되는 패턴)를 학습하려면 최소 4~8주 분량의 1시간 단위 집계 데이터가 필요하다. MVP 설계 시점에 보존 기간(예: 원본 90일, 1시간 집계 1년)을 확정해야 스토리지 용량과 쿼리 성능 설계가 가능하다.

- **외부 API 실패 패턴 이벤트가 기록되지 않는다.**
  정책 P3("3회 연속 실패 시 회색 상태")이 있지만, API 실패 자체를 도메인 이벤트로 기록하는 흐름이 없다. `외부API호출실패됨` 이벤트를 별도로 기록해야 "어떤 장소에서, 어떤 시간대에, 어떤 데이터 소스가 자주 실패하는가"를 분석할 수 있다. 이 패턴은 특정 지역/시간대의 데이터 신뢰도를 사전에 낮게 예측하는 모델의 학습 신호로 직접 활용된다.

- **대안 카드의 비선택(rejection) 이유 수집 메커니즘이 없다.**
  `07-대안장소검색.puml`은 카드 생성과 제시까지만 다루고 있다. 사용자가 제시된 3장 중 어떤 카드도 선택하지 않고 화면을 닫는 경우(묵시적 거부)나, 카드를 봤지만 결국 원래 장소를 고수하는 경우를 이벤트로 포착해야 한다. 이 데이터가 있어야 "왜 대안 카드가 채택되지 않았는가"를 분석하고 추천 품질을 개선할 수 있다.

- **다국어/다문화 컨텍스트가 브리핑 생성에 반영되어 있지 않다.**
  MVP 타깃이 "해외 자유여행자"임에도 불구하고, 브리핑 콘텐츠 생성 이벤트에 `locale`, `timezone`, `currency` 같은 컨텍스트 필드가 없다. Claude API를 활용한 자연어 브리핑 생성 시 이 정보는 프롬프트 컨텍스트로 필수 입력이 된다. 여행 일정 등록 시점에 여행 국가/도시를 `Trip` Aggregate에 기록하고, 브리핑 생성 이벤트에 전달하는 흐름을 추가해야 한다.
