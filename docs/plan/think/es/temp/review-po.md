# 이벤트 스토밍 리뷰 - 피오 (제품 책임자)

> 리뷰 관점: 비즈니스 가치, 고객 가치 전달, 수익화 연결
> 리뷰일: 2026-02-22

---

## 긍정 평가

### 1. S05(배지) -> S04(브리핑) -> S06(대안 카드)의 통합 UX 흐름이 이벤트 수준에서 완벽하게 구현되었다

핵심솔루션 문서에서 정의한 **"인지(Pull: 배지) -> 알림(Push: 브리핑) -> 행동(Action: 대안 카드)"** 흐름이 유저플로우 01~08과 userflow.puml에 빠짐없이 반영되어 있다. 04(상태 배지 조회)에서 노랑/빨강 탭 시 07(대안 장소 검색)으로 연결되고, 06(브리핑 수신)에서 주의/위험 시 07로 연결되는 두 가지 진입 경로가 모두 설계되었다. 이것은 S05와 S04가 각각 독립적으로 S06의 트리거가 된다는 비즈니스 설계 의도를 정확히 반영한 것이다.

### 2. Needs Statement "목적지에 도착하기 전에 미리 알 수 있는 경험"이 이벤트 흐름에 구조적으로 녹아 있다

05(출발 전 브리핑 생성)에서 "출발 예정 시간 15~30분 전 브리핑 트리거(P7)"가 정책으로 명시되고, 정상 시에도 "안심 브리핑"을 발송(P8)하는 정책이 시퀀스 다이어그램에 alt 분기로 정확히 표현되었다. "도착 전에 미리 안다"는 핵심 가치가 스케줄러 -> 브리핑업무 -> 알림업무 -> 여행자의 이벤트 체인으로 기술적 흐름까지 완결되었다.

### 3. 킹핀 B(사후 인지) -> A(대안 탐색) 인과 체인이 이벤트 간 의존 관계로 정확히 표현되었다

B 해결(S04+S05)에서 문제 감지 시 A 해결(S06)로 자동 연결되는 P9 정책이 06(브리핑 수신)의 alt 분기와 04(배지 조회)의 alt 분기에 모두 반영되어 있다. 비즈니스 모델에서 정의한 "B를 해결하면 A의 발생 빈도 자체가 감소하고, C의 트리거 빈도도 줄어든다"는 인과 구조가 이벤트 설계에 투영된 점이 좋다.

### 4. "탭 한 번으로 일정 반영" 3단계 UX가 08 시퀀스에서 충실히 구현되었다

08(대안 카드 선택 및 일정 반영)에서 여행자의 액션이 "대안 카드 선택" 단 하나이고, 이후 일정 교체 -> 후속 이동시간 재계산(P13) -> 새 장소 모니터링 등록(P14)이 시스템 간 자동으로 처리된다. S06의 핵심 가치인 "23.5분 -> 3분 이하" 검색 노동 제거가 최소 인터랙션으로 설계된 점이 비즈니스 가치와 정합한다.

### 5. 14개 정책/규칙이 빠짐없이 시퀀스 다이어그램에 note로 명시되었다

정책이 특정 문서에만 존재하고 다이어그램에서 빠지면 구현 단계에서 누락될 위험이 있는데, P1~P14가 해당 시퀀스의 note 블록에 모두 표현되었다. 특히 P3(API 실패 시 캐시/3회 연속 실패 시 회색), P6(회색 = 데이터 미확인 투명 표시), P8(정상 시에도 안심 브리핑)처럼 엣지 케이스 관련 정책까지 포함된 점이 구현 리스크를 줄인다.

## 개선 의견

### 1. [06-출발전브리핑수신.puml] 무료/유료 티어 분기가 이벤트 흐름에 없다

비즈니스 모델에서 **Free 티어는 1일 1회 브리핑, Trip Pass/Pro는 무제한 브리핑**으로 정의했다. 그러나 06(출발 전 브리핑 수신) 시퀀스에서 "브리핑 Push 알림 발송 요청" 시점에 사용자 티어를 확인하고 발송 여부/횟수를 제한하는 이벤트나 정책이 없다. 수익화의 핵심 게이트웨이인 브리핑 횟수 제한 로직이 이벤트 설계에서 빠져 있으면, 무료 사용자가 무제한으로 브리핑을 받게 되어 유료 전환 동기가 사라진다.

**제안**: 05 또는 06 시퀀스에 `[정책/규칙] Free 티어: 1일 1회 브리핑 발송 제한. Trip Pass/Pro: 무제한.` 정책을 추가하고, 브리핑 발송 전 "사용자 구독 상태 확인" 커맨드와 "브리핑 발송 제한됨(무료 한도 초과)" 이벤트를 분기로 추가할 것.

### 2. [07-대안장소검색.puml] 대안 카드(S06) 역시 유료 기능인데 접근 제어 이벤트가 없다

비즈니스 모델에서 대안 카드(S06)는 **Trip Pass 이상**에서만 사용 가능한 유료 기능이다. 그러나 07(대안 장소 검색)에서 여행자가 "대안 장소 검색 요청"을 보낼 때 구독 상태를 검증하는 이벤트가 없다. Free 사용자가 배지(S05)에서 빨강을 탭하거나 브리핑에서 "대안 보기"를 탭했을 때, 유료 전환 유도 화면(Paywall)으로 분기하는 흐름이 필요하다.

**제안**: 07 시퀀스 시작부에 `[정책/규칙] 대안 카드는 Trip Pass/Pro 전용. Free 사용자는 유료 전환 안내.` 정책을 추가하고, Free 사용자에게 "유료 전환 안내됨" 이벤트를 alt 분기로 추가할 것. 이 Paywall 지점이 수익화 전환의 핵심 퍼널이므로 명시적으로 설계되어야 한다.

### 3. [01-여행일정등록.puml] 온보딩/첫 일정 등록 시 가치 인지 이벤트가 없다

01(여행 일정 등록)은 사용자가 서비스를 처음 접하는 Activation 퍼널의 시작점이다. 그런데 현재 시퀀스는 순수한 CRUD(일정 생성 -> 장소 추가 -> 모니터링 등록)만 있고, 사용자에게 "이 앱이 왜 다른지"를 인지시키는 이벤트가 없다. AARRR 지표에서 Activation의 핵심인 "일정 등록 완료율 60%"를 달성하려면, 등록 과정에서 핵심 가치(상태 배지, 출발 전 브리핑)를 미리 안내하는 흐름이 필요하다.

**제안**: 장소 추가 완료 직후 "상태 배지 초기 표시됨(프리뷰)" 이벤트를 추가하여, 일정 등록 완료 시점에 배지가 바로 표시되는 경험을 설계할 것. "등록하자마자 이미 모니터링이 시작되었다"는 즉각적 가치 인지가 Activation 전환율을 높인다.

### 4. [이벤트 스토밍 결과 전체] 사용자 행동 추적(Analytics) 이벤트가 전혀 없다

비즈니스 모델의 Key Metrics(AARRR)에서 브리핑 열람률, 배지 탭 비율, 대안 채택률, 유료 전환율 등을 측정해야 한다. 그러나 이벤트 스토밍 결과에 "브리핑 열람됨", "배지 탭됨", "대안 카드 노출됨", "대안 카드 채택됨" 같은 분석용 이벤트가 정의되어 있지 않다. MVP 검증 KPI를 측정할 수 없으면 "Fast fail, Learn and Pivot"을 실행할 데이터 기반이 없다.

**제안**: 각 시퀀스에서 사용자 인터랙션이 발생하는 시점에 분석용 이벤트를 추가할 것. 최소한 다음 4개는 필요하다:
- "브리핑 열람됨" (06 시퀀스, 브리핑 상세 조회 시점)
- "배지 상세 조회됨" (04 시퀀스, 상태 상세 조회 시점)
- "대안 카드 노출됨" (07 시퀀스, 카드 제시 시점)
- "대안 카드 채택됨" (08 시퀀스, 카드 선택 시점)

### 5. [08-대안카드선택및일정반영.puml] "대안을 선택하지 않음" 분기가 없다

08 시퀀스는 여행자가 대안 카드를 선택하는 Happy Path만 존재한다. 그러나 대안 카드 3장이 모두 마음에 들지 않아 "이전 일정 유지" 또는 "직접 검색"을 선택하는 경우가 현실적으로 빈번하다. 대안 채택률 60% 목표는 곧 40%는 채택하지 않는다는 의미이므로, 이 40%의 흐름이 설계되어야 한다.

**제안**: 08 시퀀스에 `alt 대안 선택 / else 대안 미선택` 분기를 추가하고, 미선택 시 "기존 일정 유지됨" 이벤트와 함께 "대안 미채택 사유 기록됨" 분석 이벤트를 추가할 것. 미채택 사유 데이터는 대안 추천 알고리즘 개선의 핵심 피드백 루프가 된다.

## 누락 사항

### 1. 사용자 인증/구독 관리 컨텍스트가 완전히 부재하다

이벤트 스토밍 결과에서 5개 바운디드 컨텍스트(일정/장소/모니터링/브리핑/대안)를 정의했는데, **사용자(User) 컨텍스트**가 없다. 비즈니스 모델의 수익 구조(Free/Trip Pass/Pro 3개 티어)를 운영하려면 최소한 다음 이벤트가 필요하다:
- "사용자 가입됨", "구독 티어 변경됨", "Trip Pass 구매됨", "Pro 구독 시작됨/해지됨"
- 결과 문서 6-4절에 "Phase 4: 사용자 서비스 분리"가 로드맵에 있으나, MVP에서도 구독 상태 확인은 필수이다

MVP에서 별도 마이크로서비스까지는 불필요하지만, **일정 서비스나 브리핑 서비스 내부에 사용자 구독 상태를 확인하는 이벤트/정책이 최소한 존재해야** 유료 전환 퍼널이 작동한다.

### 2. 알림 설정/수신 거부 흐름이 없다

P8(정상 시에도 안심 브리핑 발송)으로 습관 형성을 유도하지만, 사용자가 알림을 끄거나 빈도를 조절하는 흐름이 없다. 앱스토어 리뷰에서 가장 많은 부정 피드백이 "알림 과다"인 만큼, MVP에서도 알림 설정 이벤트("알림 설정 변경 요청됨", "알림 빈도 조정됨")가 있어야 한다. 이것이 없으면 Push 알림 수신 거부(OS 레벨)로 이어져 브리핑 열람률 70% 목표 자체가 위험해진다.

### 3. 여행 완료/피드백 수집 흐름이 없다

비즈니스 모델의 Referral 지표(바이럴 계수 0.3, NPS 40+)와 Unfair Advantage("도시별 실시간 POI 정확도 데이터 플라이휠")를 작동시키려면, **여행 완료 후** 피드백을 수집하는 흐름이 필요하다:
- "여행 완료됨" -> "정확도 피드백 요청됨" -> "피드백 제출됨(정확/부정확)"
- "NPS 설문 요청됨" -> "NPS 점수 제출됨"
- "공유 링크 생성됨" -> "추천 코드 사용됨"

이 흐름이 없으면 Accuracy Flywheel이 작동하지 않고, 바이럴 계수 측정/개선이 불가능하다. 유저플로우 09번("여행 완료 및 피드백")을 추가하는 것을 강력히 권고한다.

### 4. 오프라인 모드 대응 이벤트가 없다

비즈니스 모델 MVP 범위(MoSCoW)에서 **Should Have로 "오프라인 캐싱"**이 명시되어 있다. 해외 여행 중 데이터 로밍이 끊기거나 Wi-Fi가 불안정한 상황은 매우 빈번하다. 그러나 02(외부 데이터 수집)에서 네트워크 불가 시의 흐름이 P3(API 실패 시 캐시 사용)으로만 처리되어 있고, 사용자 앱 단에서 오프라인 상태 감지 -> 캐시 데이터 표시 -> 온라인 복귀 시 동기화하는 이벤트 흐름이 없다. 해외 여행 중 네트워크 불안정은 "일상"이므로, 이 흐름이 없으면 핵심 고객(해외 자유여행자)의 실사용 경험에 큰 구멍이 생긴다.

## 종합 의견

아키가 작성한 이벤트 스토밍은 핵심 솔루션 S04/S05/S06의 기술적 흐름을 매우 충실하게 분석했다. 8개 유저플로우, 7개 이벤트 클러스터, 5개 바운디드 컨텍스트, 5개 마이크로서비스 정의, 14개 정책/규칙 모두 비즈니스 요구사항과 정합한다.

다만 **수익화 퍼널(구독 티어 분기, Paywall)과 PMF 측정(분석 이벤트, 피드백 수집)**이라는 두 가지 비즈니스 핵심 축이 이벤트 수준에서 누락되어 있다. 이 두 가지가 보완되지 않으면, 기술적으로 완벽한 서비스를 만들어도 "돈을 버는 구조"와 "개선하는 구조"가 작동하지 않는다.

개선 우선순위를 정리하면 다음과 같다:

| 우선순위 | 항목 | 근거 |
|:--------:|------|------|
| 1 | 무료/유료 티어 분기 정책 추가 (개선 1, 2) | 수익화 게이트웨이. 없으면 유료 전환율 8% 달성 불가 |
| 2 | 분석 이벤트 추가 (개선 4) | MVP 검증 KPI 측정 불가. "Fast fail" 실행 기반 |
| 3 | 여행 완료/피드백 흐름 추가 (누락 3) | Accuracy Flywheel + NPS + 바이럴 계수의 전제 조건 |
| 4 | 대안 미선택 분기 추가 (개선 5) | 대안 추천 알고리즘 개선 피드백 루프 |
| 5 | 온보딩 가치 인지 이벤트 (개선 3) | Activation 전환율 60% 달성 지원 |
| 6 | 알림 설정/수신 거부 (누락 2) | 브리핑 열람률 70% 방어 |
| 7 | 오프라인 모드 대응 (누락 4) | Should Have이나 해외 여행 특성상 중요 |

위 1~3번은 MVP 유저스토리 작성 전에 반드시 반영되어야 하며, 4~7번은 유저스토리 단계에서 보완 가능하다고 판단한다.
