# 이벤트 스토밍 리뷰 - 인사이트 (도메인 전문가)

> 리뷰 관점: 도메인 정합성, 규제, 외부 시스템 현실성
> 리뷰일: 2026-02-22

---

## 긍정 평가

1. **상태 배지 4단계 색상 체계의 현실 반영도가 높다.**
   `03-실시간상태모니터링.puml` 및 `event-storming-결과.md` 정책 P4에서 초록/노랑/빨강/회색으로 구분한 것은 여행 현장에서 즉각적인 시각 판단이 필요한 상황에 잘 맞는다. 특히 회색(데이터 미확인)을 별도 상태로 두어 "불확실성을 숨기지 않는다"는 점은 해외 여행처럼 정보 신뢰도가 불균등한 환경에서 중요한 도메인 판단이다.

2. **API 실패 대응 정책(P3)이 명시적으로 정의되어 있다.**
   `02-외부데이터수집.puml`에서 "3회 연속 실패 시 회색 상태"를 명시한 것은 해외에서 빈번하게 발생하는 네트워크 불안정, 로밍 환경을 고려한 현실적인 정책이다. 단순히 에러를 던지지 않고 서비스 연속성을 유지하려는 설계 의도가 도메인 관점에서 타당하다.

3. **출발 전 브리핑의 "안심 브리핑" 개념(P8)은 사용자 습관 형성 전략으로 유효하다.**
   `05-출발전브리핑생성.puml`과 `06-출발전브리핑수신.puml`에서 정상 상황에도 브리핑을 발송하는 정책은 "알림 = 위기 신호"가 아니라 "알림 = 안심 신호"로 브랜드 인식을 형성하는 데 효과적이다. 맥킨지 여행 서비스 연구에서도 신뢰 습관 형성이 재방문율과 직결된다는 점에서 도메인 전략적으로 올바른 방향이다.

4. **대안 카드 3장 선정 기준(P11)의 우선순위 논리가 여행 맥락에 맞다.**
   `07-대안장소검색.puml`에서 거리 > 평점 > 혼잡도 순으로 우선순위를 둔 것은 "지금 당장 이동해야 하는" 여행 중 돌발 상황의 특성을 잘 반영한다. 먼 곳의 명소보다 가깝고 지금 갈 수 있는 곳이 우선이어야 한다는 판단은 현장 경험과 일치한다.

5. **일정 교체 시 후속 이동시간 자동 재계산(P13)을 명시한 것은 연쇄 영향 관리 측면에서 중요하다.**
   `08-대안카드선택및일정반영.puml`에서 이 정책을 명확히 기재한 점은 평가한다. 대안 장소 선택이 후속 일정 전체에 미치는 영향을 인지하고 있다는 것을 보여준다.

## 개선 의견

### 개선 1. 영업시간 정책(P1)에 현지 시간대(Timezone) 처리가 누락되어 있다
**대상 파일: `01-여행일정등록.puml`, `event-storming-결과.md` 클러스터 1/3**

현재 P1("장소별 방문 시간은 영업시간 내여야 함")과 `01-여행일정등록.puml`에서 "방문 일시"를 다루지만, 방문 일시가 여행자 기기의 로컬 시간인지 목적지 현지 시간인지 구분이 없다. 해외 여행자는 한국 시간으로 기기를 사용하면서 일본·유럽 등 현지 장소를 등록하는 경우가 일반적이다.

- 구체적 문제: 서울 거주자가 파리 식당 13:00 방문을 등록할 때, 시스템이 이를 KST 13:00로 해석하면 CET 기준 영업시간 검증이 불가능해진다.
- 개선 제안: `CollectedData` 또는 `ScheduleItem` 데이터에 `timezone` 필드를 명시하고, 영업시간 검증은 반드시 장소 현지 시간(Place Timezone)을 기준으로 수행하는 정책을 추가할 것. Google Places API는 `utc_offset_minutes` 또는 Time Zone API를 통해 현지 시간대를 제공하므로 기술적 구현도 가능하다.

### 개선 2. Google Popular Times는 독립 API가 아닌 비공개 추정값이다
**대상 파일: `02-외부데이터수집.puml`, `event-storming-결과.md` 섹션 8**

현재 외부 시스템 목록에 "혼잡도 API (Google Popular Times)"가 독립된 REST API처럼 기술되어 있으나, 이는 사실과 다르다.

- 실제 상황: Google Popular Times는 공식 독립 API가 아니다. Google Places API의 `current_popularity` 필드로 제한적으로 제공되며, 이 값은 실시간이 아닌 Google이 집계한 과거 패턴 기반 추정값이다. 또한 모든 장소에서 지원되지 않으며, 소규모 로컬 매장은 데이터가 아예 없는 경우가 많다.
- 개선 제안: "혼잡도" 데이터 소스를 다음과 같이 현실화할 것.
  - 1차 소스: Google Places API의 `current_popularity` (지원 장소에 한함)
  - 2차 소스: Foursquare Venue Stats API (대안)
  - 미지원 장소: 혼잡도를 회색(미확인)으로 처리하는 fallback 정책 명시
- 이 수정이 없으면 P4 상태 판정 로직 자체가 흔들린다. 혼잡도 데이터 부재가 빈번할 경우 노랑 배지가 과다 발생할 수 있다.

### 개선 3. 대안 검색 반경 1km 기준이 도시별 이동 맥락을 고려하지 않았다
**대상 파일: `07-대안장소검색.puml`, `event-storming-결과.md` 정책 P10**

P10에서 "반경 1km 이내"를 고정 기준으로 명시하고 있으나, 이는 도시 밀도와 이동 수단에 따라 실효성이 크게 달라진다.

- 문제 사례 1: 도쿄·홍콩처럼 고밀도 도시에서는 1km 내에 수십 개의 대안이 존재하여 선별 기준의 의미가 희석된다.
- 문제 사례 2: 교외 지역(나파 밸리, 제주 외곽)이나 관광지 집중 구역에서는 1km 내에 동일 카테고리 장소가 전무할 수 있다. 이 경우 대안 카드 3장을 채우지 못하는 빈 결과가 발생한다.
- 개선 제안: 반경을 고정값이 아닌 동적 확장 정책으로 변경할 것.
  - 1km 내 결과 3개 미만 시 → 2km로 확장 재검색
  - 2km 내에도 부족 시 → 3km 확장 및 "조금 먼 대안" 레이블 표시
  - 대도시 고밀도 환경: 도보 이동 기준 15분(약 1.2km)을 상한으로 설정
  - 이 정책 변경이 P10 및 `07-대안장소검색.puml`의 노트에 반영되어야 한다.

### 개선 4. 출발 전 브리핑 트리거 타이밍(P7)이 복수 일정 연속 방문 시 충돌한다
**대상 파일: `05-출발전브리핑생성.puml`, `event-storming-결과.md` 정책 P7**

P7에서 "출발 예정 시간 15~30분 전 브리핑 트리거"를 명시하고 있으나, 여행자가 하루에 4~6개 장소를 방문하는 전형적인 자유여행 패턴에서 현실적인 문제가 발생한다.

- 구체적 문제: 장소 A 방문 14:00, 장소 B 방문 15:00로 등록된 경우, 장소 A에 대한 브리핑이 13:30~13:45에 도착하고 장소 B에 대한 브리핑이 14:30~14:45에 도착한다. 이 두 브리핑이 겹치거나 연속 발송되어 알림 피로도가 높아진다. 더 심각한 경우, 장소 A 방문 중(14:00~14:50)에 장소 B 브리핑이 발송되어 사용자가 이미 이동 준비 중일 때 혼란을 야기한다.
- 개선 제안: 브리핑 트리거 정책을 다음과 같이 보완할 것.
  - 직전 방문지 체류 시간 중 최소 10분이 경과한 이후에만 다음 장소 브리핑 발송
  - 연속 방문 간격이 60분 미만인 경우, 현재 장소 도착 확인(GPS 또는 수동 체크인) 후 브리핑 트리거
  - 이 정책 변경을 `05-출발전브리핑생성.puml`의 스케줄러 노트에 추가 기재 필요

### 개선 5. 개인정보(위치정보) 수집에 대한 동의 및 법적 근거 이벤트가 전체 흐름에서 누락되었다
**대상 파일: `01-여행일정등록.puml`, `userflow.puml`**

전체 유저플로우 어디에도 위치정보 수집 동의, Push 알림 수신 동의 이벤트가 존재하지 않는다.

- 법적 배경: 한국 위치정보법(위치정보의 보호 및 이용 등에 관한 법률) 제18조는 개인위치정보 수집·이용 시 사전 동의를 의무화하고 있다. EU GDPR Article 6, 7도 위치 데이터를 특별 범주 데이터에 준하여 명시적 동의를 요구한다. 서비스 대상이 해외 여행자이므로 GDPR 적용 가능성이 실질적이다.
- FCM Push 알림 역시 iOS(APNs)와 Android 모두 사용자 명시적 허용이 필요하며, 이 허용 이벤트 없이 `06-출발전브리핑수신.puml`의 Push 알림 수신 흐름이 완성되지 않는다.
- 개선 제안: `01-여행일정등록.puml` 또는 별도 온보딩 플로우에 다음 이벤트를 추가할 것.
  - `[커맨드] 위치정보 수집 동의 요청` → `[이벤트] 위치정보 수집 동의 완료됨`
  - `[커맨드] Push 알림 수신 허용 요청` → `[이벤트] Push 알림 수신 허용됨`
  - 미동의 시 서비스 제공 범위(오프라인 일정 관리만 가능 등) 정의 필요

## 누락 사항

### 누락 1. 공휴일·임시 휴무 처리 로직이 없다

P1("장소별 방문 시간은 영업시간 내여야 함")에서 영업시간을 정적 데이터로만 다루고 있다. 실제 해외 여행에서 가장 빈번하게 발생하는 문제 중 하나가 현지 공휴일 또는 비정기 임시 휴무다.

- Google Places API의 `opening_hours` 필드는 정기 영업시간만 제공하며, 현지 공휴일 적용 여부는 `exceptional_opening_hours`(Google Maps Platform) 또는 별도 공휴일 데이터베이스(예: Nager.Date API, Calendarific) 연동이 필요하다.
- 현재 설계대로라면 일본 골든위크·프랑스 바스티유 데이, 이탈리아 페라고스토 기간에 "영업 중" 오판단이 발생할 수 있다.
- 추가 필요 정책: 현지 공휴일 데이터 소스 명시 및 공휴일 전날 영업시간 단축(Reduced Hours) 처리 정책

### 누락 2. 오프라인 모드(네트워크 단절 상황) 동작 정의가 없다

해외 자유여행 중 지하철, 건물 내부, 농촌 지역 등 네트워크 단절은 일상적이다. 현재 설계는 모든 흐름이 실시간 API 호출을 전제로 하며, 오프라인 상황에서의 서비스 동작이 정의되어 있지 않다.

- P3("API 실패 시 최근 캐시 사용")은 API 서버 장애를 다루지만, 클라이언트 네트워크 단절 상황과는 다르다.
- 누락된 정책 예시:
  - 오프라인 상태에서 일정표 조회 시 마지막 캐시 데이터 표시 및 "오프라인 모드" 표시 정책
  - 오프라인 상태에서 대안 검색 요청 시 사전 다운로드된 주변 장소 데이터 활용 가능 여부
  - 브리핑 알림이 네트워크 복귀 후 지연 수신될 때 시간 유효성 검증 (이미 출발 시각을 넘긴 브리핑은 무효 처리)

### 누락 3. 사용자 실수 취소(Undo) 및 일정 복원 흐름이 없다

`08-대안카드선택및일정반영.puml`에서 "탭 한 번으로 일정 반영"이 강조되어 있으나, 빠른 탭이 실수 선택으로 이어질 경우 이를 되돌리는 이벤트가 없다.

- 도메인 현실: "탭 한 번"은 UX 편의성이지만 돌이킬 수 없는 일정 교체가 되면 신뢰를 깎는다. 특히 후속 일정 이동시간이 자동 재계산되는(P13) 연쇄 변경이 발생하기 때문에, 취소 시 원래 일정 전체를 원상복구해야 하는 트랜잭션 범위가 복잡해진다.
- 추가 필요 이벤트: `[커맨드] 일정 교체 취소 요청` → `[이벤트] 일정 원복됨` (교체 후 일정 시간 내, 예: 30초~2분 이내에 한해 허용)

### 누락 4. 동일 장소 중복 등록 방지 정책이 없다

`01-여행일정등록.puml`에서 장소 추가 흐름이 있으나, 같은 여행 일정 내에 동일 장소를 다른 날/시간대에 중복 등록하는 경우에 대한 처리가 정의되지 않았다. 또한 대안 카드로 선택된 장소가 이미 같은 날 일정에 등록된 장소와 동일한 경우도 처리가 없다.

### 누락 5. 브리핑 수신 거부(Do Not Disturb) 및 알림 설정 관리 흐름이 없다

여행 중 야간·수면 시간대, 기내 모드, 집중 방해 금지 설정 상태에서 브리핑이 발송될 경우에 대한 처리가 없다. Push 알림 과다 발송은 앱 삭제의 주요 원인 중 하나로, 알림 수신 시간대 설정 및 일시 정지 기능이 도메인 필수 요소임에도 현재 이벤트 흐름에서 완전히 누락되어 있다.
