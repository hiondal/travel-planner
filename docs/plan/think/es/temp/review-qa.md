## 긍정 평가

1. **정책/규칙이 시퀀스 다이어그램 내에 인라인으로 명시됨**
   - 각 .puml 파일에 `[정책/규칙]` 노트가 흐름과 함께 표기되어 테스트 케이스 도출 기준이 명확하다.
   - 특히 P3(3회 연속 실패 시 회색 처리), P5(상태 변경 방향성), P7(15~30분 전 트리거), P11(3장 선정 우선순위) 등이 구체적 수치로 정의되어 경계값 테스트 설계가 즉시 가능하다.

2. **배지 상태 전이 방향이 명확히 정의됨 (03-실시간상태모니터링.puml)**
   - 초록->노랑, 초록->빨강, 노랑->빨강의 단방향 전이만 알림을 트리거하도록 설계됨.
   - 역방향(빨강->노랑, 빨강->초록 등) 전이는 알림 없이 처리되는 정책이 명시되어 테스트 경계가 분명하다.

3. **외부 API 실패 대응 정책이 존재함 (02-외부데이터수집.puml)**
   - API 실패 시 캐시 사용, 3회 연속 실패 시 회색 상태 전환이라는 fallback 정책이 존재한다.
   - 이 정책이 테스트 가능한 수치(3회)로 정의된 점이 긍정적이다.

4. **유저플로우 간 연결 구조가 명시됨 (event-storming-결과.md)**
   - 08번 흐름 종료 후 03번(실시간 상태 모니터링)으로 피드백되는 순환 구조가 다이어그램으로 표현됨.
   - 이를 통해 통합 테스트 시나리오(일정 교체 후 신규 장소 모니터링 정상 작동)를 도출할 수 있다.

5. **5개 마이크로서비스 경계와 통신 방식이 명확히 구분됨**
   - 동기/비동기 구분(컨텍스트 맵 5-2), Customer-Supplier 관계(5-3)가 정의되어 서비스 간 경계 테스트 범위 설정이 용이하다.

## 개선 의견

### 1. 외부 API 부분 실패 시나리오 미흡 (02-외부데이터수집.puml)

현재 다이어그램은 4개 외부 API(Google Places, 날씨, 혼잡도, 교통)를 순차적으로 호출하지만, **일부만 성공하는 부분 실패(Partial Failure) 케이스**가 정의되어 있지 않다.

- 예: 영업상태만 성공하고 날씨 API가 타임아웃되는 경우 상태 판정 로직이 어떻게 동작해야 하는가?
- 제안: `alt [영업상태 조회 실패]`, `alt [날씨 API 타임아웃]` 분기를 각 API 호출 직후에 추가하고, "단일 항목 실패 시 해당 항목은 캐시 값 사용, 나머지는 신규 값 사용"과 같은 정책을 명시할 것.
- 테스트 포인트: 4개 API 각각을 개별 실패시켰을 때 상태 판정 결과가 올바른지 검증하는 케이스 필요.

### 2. 배지 상태 역전이 처리 미정의 (03-실시간상태모니터링.puml)

P5 정책에서 초록->노랑->빨강 방향만 알림을 트리거한다고 명시했으나, **역방향 전이(빨강->노랑->초록)** 에 대한 처리가 다이어그램 어디에도 정의되어 있지 않다.

- 예: 빨강 상태에서 혼잡도가 낮아져 초록으로 회복되는 경우, 알림을 보내는가? 조용히 배지만 변경하는가? Push가 오지 않으면 사용자는 회복 사실을 알 수 없다.
- 제안: "상태 회복 시 알림 정책" 분기를 03 다이어그램에 추가하고, P5에 "역방향 전이 시 정책"을 명시할 것.
- 테스트 포인트: 빨강->초록 회복 시 배지 업데이트 여부, 알림 발송 여부 경계값 테스트.

### 3. 브리핑 트리거 타이밍 경계값 정의 누락 (05-출발전브리핑생성.puml)

P7은 "출발 예정 시간 15~30분 전" 브리핑 트리거를 정의하지만, **구체적인 판단 시점과 중복 발송 방지 정책**이 없다.

- 예: 출발 25분 전에 트리거가 실행되었는데 브리핑 생성에 5분이 걸려 도착 시 출발 20분 전이 된 경우 정상인가?
- 예: 15분~30분 사이에 스케줄러가 2회 실행되면 동일 브리핑이 2번 발송되는가?
- 제안: "출발 N분 전 최초 1회만 발송" 정책을 명시하고, 멱등성(idempotency) 처리 방식을 다이어그램에 추가할 것. 브리핑 발송 완료 이력을 기록하는 이벤트(`브리핑 발송 이력 저장됨`)를 추가하는 것이 적절하다.
- 테스트 포인트: 14분 전(미트리거), 15분 전(트리거), 30분 전(트리거), 31분 전(미트리거) 경계값 테스트 및 중복 호출 시 1회만 발송되는지 검증.

### 4. Push 알림 실패 시나리오 완전 누락 (06-출발전브리핑수신.puml)

FCM 발송 후 알림이 미수신되거나 지연되는 케이스가 다이어그램에 전혀 정의되어 있지 않다.

- 누락된 시나리오:
  - FCM 서버 응답 오류(5xx) 발생 시 재시도 정책
  - 디바이스 오프라인으로 알림 지연 수신 후 출발 시간이 이미 지난 경우
  - 동일 브리핑 알림 중복 수신(네트워크 재시도로 인한 중복)
  - 사용자가 알림을 탭하지 않고 무시한 경우의 후속 처리
- 제안: `alt [FCM 발송 실패]` 분기를 추가하고, "N분 내 재시도 X회, 최종 실패 시 인앱 알림 fallback" 정책을 명시할 것.
- 테스트 포인트: FCM 타임아웃, 400/500 응답, 중복 전송 방지 토큰 검증.

### 5. 대안 카드 3장 미만 결과 처리 미정의 (07-대안장소검색.puml)

P10 조건(동일 카테고리, 반경 1km, 영업 중)을 모두 만족하는 장소가 3개 미만이거나 0개인 경우의 처리 흐름이 없다.

- 예: 현재 위치가 외진 지역이라 반경 1km 내에 동일 카테고리 영업 장소가 0개인 경우 사용자에게 무엇을 보여주는가?
- 제안: `alt [대안 장소 0개]`, `alt [대안 장소 1~2개]` 분기를 추가하고, "반경 확장(1km->2km->5km) 재시도" 또는 "카테고리 완화" 폴백 정책을 명시할 것.
- 테스트 포인트: 후보 0개/1개/2개/3개/3개 초과(정확히 3개 선정되는지) 경계값 테스트.

## 누락 사항

### A. 인증/세션 예외 시나리오 전체 누락 (01-여행일정등록.puml ~ 08-대안카드선택및일정반영.puml)

모든 8개 흐름에서 **인증 실패, 세션 만료, 미인증 요청** 케이스가 단 한 번도 정의되지 않았다.

- 해외 여행 중 네트워크 전환(Wi-Fi->LTE) 또는 장시간 백그라운드 전환으로 세션이 만료된 상태에서 알림을 탭하면 어떤 흐름이 발생하는가?
- 일정 교체 요청(08번) 중 세션이 만료되면 교체는 반영되는가?
- 추가 필요: 모든 커맨드 진입점에 `[정책/규칙] 미인증 요청 시 로그인 페이지로 리다이렉트` 처리 명시.

### B. 동시성 시나리오 누락

**같은 장소에 대한 복수 요청 동시 처리** 케이스가 정의되어 있지 않다.

- 같은 시각에 동일 장소를 일정에 추가한 두 사용자가 대안 검색을 동시에 요청하면, 상태판정업무의 후보 장소 상태 일괄 조회가 경합하는가?
- 일정 장소 교체(08번)와 모니터링 수집 스케줄러(02번)가 동일 장소 ID에 대해 동시에 실행될 경우 데이터 정합성은 보장되는가?
- 추가 필요: 03-실시간상태모니터링.puml에 낙관적 잠금 또는 상태 변경 원자성 보장 정책 명시.

### C. 캐시와 실시간 데이터 불일치 시나리오 누락 (02, 04 파일)

- 02번에서 수집 데이터가 캐시에 저장되고, 04번(상태 배지 조회)에서 실시간으로 읽히는 구조이나, **캐시 TTL이 정의되지 않았다.**
- 예: 방문 예정 2시간 전부터 15분 주기로 수집하지만, 사용자가 마지막 수집 후 14분 59초 뒤에 배지를 조회하면 최대 15분 오래된 데이터를 보게 된다.
- 추가 필요: 캐시 TTL 명시, "캐시 생성 시각" 표시 정책, 신선도(staleness) 임계값 정의.

### D. 일정 교체 롤백 시나리오 누락 (08-대안카드선택및일정반영.puml)

- 일정 교체 후 모니터링 등록이 실패하면(상태판정업무 오류) 교체된 일정은 어떻게 되는가?
- 일정은 교체됐으나 새 장소에 대한 모니터링이 없는 "반쪽 완료" 상태가 발생할 수 있다.
- 추가 필요: `alt [모니터링 등록 실패]` 분기를 추가하고 보상 트랜잭션(Saga 패턴) 또는 재시도 정책을 명시할 것.

### E. 사용자 알림 수신 거부/설정 미반영 (06-출발전브리핑수신.puml)

- OS 레벨 Push 알림 권한 거부 상태에서 FCM 발송 시 처리 흐름이 없다.
- 사용자가 알림을 끈 경우 인앱 메시지 또는 배지 갱신으로 fallback하는 정책이 필요하다.
- 추가 필요: "Push 권한 미보유 사용자" 처리 분기 및 `알림 설정` 상태를 사용자 프로파일 데이터로 명시.

### F. 여행 기간 외 접근 시나리오 누락 (01-여행일정등록.puml)

- 여행 기간이 종료된 이후에도 사용자가 해당 일정에 접근하여 배지를 조회하거나 대안을 요청하는 경우의 처리가 없다.
- 여행 종료 후 모니터링 자동 해제 정책(P2와 연관)이 명시되어야 한다.
- 추가 필요: `[정책/규칙] 여행 종료 후 모니터링 대상 자동 해제` 이벤트(`모니터링 대상 해제됨`) 추가.

### G. 구글 API 비율 제한(Rate Limit) 시나리오 누락 (02-외부데이터수집.puml, 07-대안장소검색.puml)

- 02번과 07번 모두 Google Places API를 호출하나, **API 비율 제한(429 Too Many Requests)** 응답에 대한 처리가 없다.
- 여러 사용자의 모니터링이 동시에 활성화된 경우 단시간에 다수의 API 호출이 발생할 수 있다.
- 추가 필요: Rate Limit 초과 시 지수 백오프(Exponential Backoff) 재시도 정책, 또는 API 호출 버킷 관리 정책 명시.

> 특히 06~08번 흐름은 사용자가 실제로 가장 스트레스를 받는 돌발 상황 대응 흐름임에도 오류 처리가 전혀 없어 우선적으로 보완이 필요하다.
