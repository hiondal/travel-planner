# 이벤트 스토밍 리뷰 - 데브-백 (백엔드 개발자)

> 리뷰 관점: 기술 타당성, 구현 가능성, 서비스 아키텍처
> 리뷰일: 2026-02-22

---

## 긍정 평가

### 1. 서비스 분리 기준이 명확하다
5개 서비스의 바운디드 컨텍스트 경계가 데이터 소유권 기준으로 명확하게 정의되어 있다. 특히 `모니터링 서비스`에서 데이터 수집과 상태 판정을 하나의 트랜잭션 경계로 묶은 결정(수집 데이터 저장 + 상태 갱신의 원자성)은 구현 관점에서 올바른 판단이다. 별도 서비스로 쪼갰다면 분산 트랜잭션 문제가 발생했을 것이다.

### 2. 비동기/동기 통신 방식 구분이 적절하다
일정 변경 이벤트(일정 -> 모니터링)처럼 즉각 응답이 불필요한 흐름은 비동기로, 장소 정보 조회(일정 -> 장소)처럼 사용자 응답에 직결된 흐름은 동기로 구분한 방식이 타당하다. 이는 서비스 간 결합도를 낮추는 데 효과적이다.

### 3. 폴백 정책(P3)이 명시되어 있다
`API 실패 시 최근 캐시 사용, 3회 연속 실패 시 회색 상태`라는 폴백 규칙이 이벤트 스토밍 단계에서 명시된 점은 백엔드 구현 시 큰 도움이 된다. 외부 API 의존 서비스에서 이 정책이 없으면 장애 대응 코드가 누락되는 경우가 많다.

### 4. 상태 전이 방향이 단방향으로 명확하다 (P5)
`초록 -> 노랑 -> 빨강` 방향의 알림 트리거 정책만 정의하고, 역방향(빨강 -> 초록) 알림은 별도 정의하지 않은 점이 구현을 단순하게 유지한다. 알림 피로도 방지 측면에서도 올바른 설계다.

### 5. 확장 로드맵이 서비스 분리 기준과 연결되어 있다
`알림 채널 2개 이상 추가 시 알림 서비스 분리`라는 Phase 2 기준이 구체적이다. MVP에서는 브리핑+알림을 통합하고 이후 분리하는 점진적 전략은 구현 복잡도를 줄이면서도 확장성을 확보하는 현실적 접근이다.

## 개선 의견

### 1. [05-출발전브리핑생성.puml] 브리핑 트리거 스케줄러의 구현 방식 명확화 필요
현재 `출발 예정 시간 15~30분 전 브리핑 트리거` 정책(P7)이 있지만, 스케줄러가 어떻게 각 사용자의 출발 시간을 감시하는지 흐름에 표현되어 있지 않다.

- **문제**: 사용자가 수천 명이면 수천 개의 출발 예정 시간을 주기적으로 스캔해야 한다. 단순 Cron(`@Scheduled`) 방식으로는 초 단위 정밀도가 떨어지고, 대량 사용자 환경에서 DB 풀 스캔 부하가 발생한다.
- **제안**: Delayed Queue 패턴(Redis Sorted Set + TTL 또는 RabbitMQ Dead Letter Exchange)을 활용하여 일정 등록 시점에 `출발 시간 - 20분` 지연 메시지를 발행하고, 브리핑 서비스가 이를 소비하는 방식으로 변경할 것을 제안한다. 동일한 방식을 `02-외부데이터수집.puml`의 `방문 예정 시간 2시간 전 수집 시작` 트리거에도 적용해야 한다.

### 2. [02-외부데이터수집.puml] Google Popular Times는 공식 REST API가 아님
외부 시스템 의존성 목록에 `혼잡도 API (Google Popular Times)`가 포함되어 있으나, Google Popular Times는 공식 API로 제공되지 않는다.

- **문제**: 비공식 스크래핑 방식(`serpapi`, `outscraper` 등 3rd-party)에 의존하면 서비스 약관 위반, 데이터 신뢰성 저하, 비용 예측 불가 문제가 발생한다. 또한 Google Places API의 `current_opening_hours`에 포함된 `popular_times` 필드는 실시간 데이터가 아닌 히스토리 통계값이다.
- **제안**: 혼잡도 데이터 소스를 `Google Places API의 popular_times 필드(통계 기반)`로 명시적으로 변경하고, 실시간 혼잡도가 필요하다면 MVP 범위 외로 분류하거나 대체 지표(이동시간 증가율, 날씨 기반 추정)를 활용하는 방향으로 조정이 필요하다.

### 3. [08-대안카드선택및일정반영.puml] 일정 장소 교체 후 모니터링 등록 주체가 잘못되어 있다
`일정업무 -> 상태판정업무: 새 장소 모니터링 등록 요청` 흐름에서 `일정 서비스`가 `상태판정업무`를 직접 동기 호출하고 있다.

- **문제**: 컨텍스트 맵(5-2절)에는 일정 서비스 -> 모니터링 서비스 통신이 `비동기`로 정의되어 있으나, 08번 시퀀스에서는 동기 호출로 표현되어 있다. 두 다이어그램이 불일치한다. 또한 `상태판정업무`는 모니터링 서비스 내부의 업무 단위인데, 일정 서비스가 이를 직접 호출하면 서비스 경계가 무너진다.
- **제안**: 일정 서비스는 `일정 장소 교체됨` 이벤트를 비동기 발행하고, 모니터링 서비스가 해당 이벤트를 구독하여 내부적으로 모니터링 대상을 등록하는 방식으로 수정한다. 이렇게 하면 `01-여행일정등록.puml`의 흐름(`모니터링 대상 등록됨` 내부 이벤트)과도 일관성이 유지된다.

### 4. [05-출발전브리핑생성.puml] 브리핑 생성 시 데이터 재수집 경로가 순환 의존을 유발한다
브리핑 서비스가 `데이터수집업무`를 동기 호출하고(최신 데이터 수집 요청), 이후 `상태판정업무`를 다시 동기 호출(종합 상태 판정 요청)하는 2단계 동기 체인이 있다.

- **문제**: 브리핑 서비스 -> 모니터링 서비스(데이터수집) -> 브리핑 서비스 방향의 순환 의존이 발생할 수 있다. 또한 두 번의 동기 호출 지연이 합산되어 Push 알림 발송 시점이 15분 트리거 정밀도를 벗어날 위험이 있다. 외부 API 4개(Places, Weather, Popular Times, Directions) 호출 시간까지 포함하면 타임아웃 리스크가 크다.
- **제안**: 15분 주기로 수집되어 캐싱된 최신 데이터를 브리핑 생성 시 그대로 읽어오는 방식(Read-Through Cache)을 우선 고려한다. 브리핑 트리거 시점에 `온디맨드 재수집`이 꼭 필요하다면 수집 완료 이벤트를 기다리는 비동기 패턴(이벤트 기반 파이프라인)으로 변경해야 한다.

### 5. [07-대안장소검색.puml] 대안 검색 시 후보가 충분하지 않을 경우 처리 흐름 부재
`초록 상태인 장소만 필터` 조건과 `반경 1km` 조건을 동시에 적용하면, 실제 여행지(도심 외곽, 소도시)에서 대안 카드 3장을 채우지 못하는 경우가 빈번하게 발생할 수 있다.

- **문제**: 후보 부족 시 처리 경로(반경 확장, 조건 완화, 0장 반환 등)가 정의되어 있지 않아 빈 카드 화면이 노출될 위험이 있다.
- **제안**: `반경 1km 이내 초록 장소 3개 미만 시 반경을 2km로 확장`하는 폴백 조건과, `3개를 채우지 못할 경우 노랑 상태 장소도 포함(단, 주의 표시)`하는 조건 완화 정책을 정책/규칙(P10 연계)으로 추가 정의한다.

## 누락 사항

### 1. 사용자 인증/인가(Auth) 서비스가 전혀 언급되지 않음
모든 시퀀스 다이어그램에 인증 흐름이 없다. `여행자 -> 일정업무` 호출 시 어떤 사용자인지 식별하는 수단(JWT, 세션)이 없으면 다중 사용자 환경에서 데이터 격리가 불가능하다. MVP라도 `사용자 ID` 기반의 인증 토큰 발급 경로와 서비스별 토큰 검증 방식(API Gateway 레벨 vs. 서비스 자체 검증)이 정의되어야 한다.

### 2. 외부 API Rate Limit 및 일일 비용 한도 관리 전략 부재
Google Places API, OpenWeatherMap, Google Directions API는 모두 호출 횟수 기반 과금 구조다. 15분 주기 수집(P2) 정책 하에서 사용자 100명, 장소 3개 기준으로 하루 수집 호출 수를 계산하면:

```
사용자 100명 x 평균 3개 장소 x 4개 API x (여행 중 평균 8시간 / 15분) = 약 38,400 호출/일
```

Google Places API 기본 무료 할당량($200/월 크레딧)을 초과할 수 있다. 서비스별 API 호출 횟수 상한(Rate Limiter), 동일 장소 중복 요청 제거(Place ID 기반 캐시 TTL), 비용 알림 임계값 설정이 아키텍처 수준에서 정의되어야 한다.

### 3. 모니터링 서비스의 스케줄러 분산 실행 방지 정책 부재
`방문 예정 시간 2시간 전부터 15분 주기 수집(P2)` 스케줄러가 모니터링 서비스 인스턴스가 2개 이상 실행될 경우, 동일 장소에 대한 수집 작업이 중복 실행되는 문제가 발생한다. `ShedLock` 또는 Redis 기반 분산 락을 활용한 중복 실행 방지 정책이 명시되어야 한다.

### 4. 장소 서비스의 데이터 이중 호출 문제 (01번 vs. 02번 흐름)
`01-여행일정등록` 시 장소 서비스가 Google Places API를 호출하여 장소 상세 정보를 저장하고, `02-외부데이터수집` 시 모니터링 서비스도 Google Places API를 통해 영업 상태를 다시 조회한다. 동일한 `Place ID`에 대해 두 서비스가 각각 Google API를 호출하는 구조로, 캐시 공유 전략(장소 서비스가 TTL 기반 캐시를 제공하고 모니터링 서비스는 장소 서비스 캐시를 경유) 또는 데이터 소유 경계 재정의가 필요하다.

### 5. Push 알림(FCM) 실패 시 재발송 전략 부재
`06-출발전브리핑수신.puml`에서 FCM 발송 실패 처리 흐름이 없다. FCM은 네트워크 오류, 토큰 만료 등으로 실패할 수 있으며, `출발 15~30분 전`이라는 시간 제약이 있는 알림이므로 재시도 정책(최대 2회 재시도, 재시도 간격 30초)과 실패 시 인앱 알림으로의 폴백 경로가 정의되어야 한다.

### 6. 대안 서비스의 일정 교체 요청 실패 시 롤백 경로 부재
`08-대안카드선택및일정반영.puml`에서 `일정 서비스`의 `일정 장소 교체` 요청이 실패했을 때의 흐름이 없다. 사용자에게 "일정 반영 실패" 피드백을 제공하고 대안 카드 화면을 유지하는 오류 처리 경로가 시퀀스에 표현되어야 한다.
