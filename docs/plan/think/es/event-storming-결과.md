## 1. 유저플로우 식별

핵심 솔루션 3개의 통합 UX 흐름 **S05(배지/Pull) -> S04(브리핑/Push) -> S06(대안 카드/Action)**을 기반으로 8개 유저플로우를 식별하였다.

| 순번 | 유저플로우 | 관련 솔루션 | 설명 |
|:----:|-----------|:----------:|------|
| 01 | 여행 일정 등록 | 공통 기반 | 여행자가 일정표에 장소를 등록하고 관리 |
| 02 | 외부 데이터 수집 | 공통 인프라 | 등록된 장소의 영업상태/날씨/혼잡도/이동시간 수집 |
| 03 | 실시간 상태 모니터링 | S05 기반 | 수집 데이터를 종합하여 장소 상태(초록/노랑/빨강/회색) 판정 |
| 04 | 상태 배지 조회 | **S05** | 일정표에서 장소별 상태 배지 확인 및 상세 정보 팝업 |
| 05 | 출발 전 브리핑 생성 | **S04** | 출발 15~30분 전 최신 데이터 기반 브리핑 데이터 생성 |
| 06 | 출발 전 브리핑 수신 | **S04** | Push 알림으로 안심/주의 브리핑 수신 및 확인 |
| 07 | 대안 장소 검색 | **S06** | 돌발 상황 시 맥락 기반 대안 장소 3개 검색 |
| 08 | 대안 카드 선택 및 일정 반영 | **S06** | 대안 카드 탭하여 일정에 즉시 반영 |

### 유저플로우 간 연결 구조

```
[여행 준비]
  01. 여행 일정 등록
      |
      v
[실시간 감시]
  02. 외부 데이터 수집 --> 03. 실시간 상태 모니터링
      |                        |              |
      |                        v              v
[S05 상태 배지]          [S04 출발 전 브리핑]
  04. 상태 배지 조회       05. 브리핑 생성 --> 06. 브리핑 수신
      |                                            |
      +--------------------------------------------+
      |
      v
[S06 대안 카드]
  07. 대안 장소 검색 --> 08. 대안 카드 선택 및 일정 반영
                              |
                              v
                        03. 실시간 상태 모니터링 (새 장소 모니터링 시작)
```

> 다이어그램 파일: `userflow.puml` 및 `01~08-*.puml` 참조

## 3. Aggregate 식별

이벤트 클러스터에서 Aggregate Root를 식별하였다.

| 클러스터 | Aggregate Root | 주요 엔티티 | 설명 |
|----------|---------------|------------|------|
| 일정 관리 | **Trip** | Trip, ScheduleItem | 여행 일정과 장소 목록을 관리하는 핵심 도메인 |
| 장소 정보 | **Place** | Place, PlaceDetail | 장소 마스터 데이터를 관리 |
| 데이터 수집 | **CollectedData** | CollectedData, DataSource | 외부 API에서 수집한 원시 데이터 관리 |
| 상태 판정 | **PlaceStatus** | PlaceStatus, StatusHistory | 수집 데이터 기반 상태 배지 판정 결과 |
| 브리핑 | **Briefing** | Briefing, BriefingItem | 출발 전 브리핑 콘텐츠 생성/관리 |
| 알림 | **Notification** | Notification, NotificationLog | Push 알림 발송 및 이력 관리 |
| 대안 검색 | **Alternative** | AlternativeCard, SearchContext | 맥락 기반 대안 카드 생성/관리 |

## 4. 바운디드 컨텍스트 정의

Aggregate를 바운디드 컨텍스트로 그룹핑하였다.

| 바운디드 컨텍스트 | Aggregate | 핵심 책임 |
|-----------------|-----------|----------|
| **일정 컨텍스트** | Trip | 여행 일정 CRUD, 장소 추가/교체, 일정표 조회 |
| **장소 컨텍스트** | Place | 장소 마스터 데이터 관리, 장소 검색, 외부 장소 API 연동 |
| **모니터링 컨텍스트** | CollectedData, PlaceStatus | 외부 데이터 수집, 상태 배지 판정, 상태 변경 감지 |
| **브리핑 컨텍스트** | Briefing, Notification | 브리핑 생성, Push 알림 발송, 안심/주의 분기 |
| **대안 컨텍스트** | Alternative | 맥락 기반 대안 검색, 카드 생성, 일정 반영 중개 |

## 5. 컨텍스트 맵핑

### 5-1. 서비스 간 관계

```
                    ┌─────────────┐
                    │  일정 서비스  │
                    │  (Trip)      │
                    └──────┬──────┘
                           │ 일정 등록/변경 이벤트 (비동기)
              ┌────────────┼────────────┐
              v            v            v
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ 모니터링 서비스│ │ 브리핑 서비스  │ │  대안 서비스   │
    │(CollectedData│ │(Briefing     │ │(Alternative) │
    │ PlaceStatus) │ │ Notification)│ │              │
    └───────┬──────┘ └──────┬──────┘ └──────┬───────┘
            │               │               │
            │  상태 조회     │  최신 데이터    │  후보 상태 조회
            │  (동기)       │  요청 (동기)    │  (동기)
            └───────┬───────┘               │
                    v                       │
            ┌──────────────┐                │
            │  장소 서비스   │ <──────────────┘
            │  (Place)     │   주변 장소 검색 (동기)
            └──────────────┘
```

### 5-2. 통신 방식

| 호출자 | 피호출자 | 방식 | 목적 |
|--------|---------|:----:|------|
| 일정 서비스 | 장소 서비스 | 동기 | 장소 정보 조회 |
| 일정 서비스 | 모니터링 서비스 | 비동기 | 모니터링 대상 등록/해제 |
| 모니터링 서비스 | 장소 서비스 | 동기 | 외부 API 통한 장소 데이터 수집 |
| 모니터링 서비스 | 브리핑 서비스 | 비동기 | 상태 변경 이벤트 발행 |
| 브리핑 서비스 | 모니터링 서비스 | 동기 | 최신 수집 데이터 요청 |
| 대안 서비스 | 장소 서비스 | 동기 | 주변 장소 검색 |
| 대안 서비스 | 모니터링 서비스 | 동기 | 후보 장소 상태 일괄 조회 |
| 대안 서비스 | 일정 서비스 | 동기 | 일정 장소 교체 요청 |
| 일정 서비스 | 모니터링 서비스 | 비동기 | 대안 선택 후 모니터링 대상 변경 이벤트 발행 |

### 5-3. 컨텍스트 맵 관계 유형

| 관계 | 유형 | 설명 |
|------|------|------|
| 일정 <-> 장소 | Customer-Supplier | 일정이 장소 데이터를 소비 |
| 일정 -> 모니터링 | Customer-Supplier | 일정 변경 시 모니터링 대상 갱신 |
| 모니터링 -> 브리핑 | Published Language | 상태 변경 이벤트를 표준 포맷으로 발행 |
| 모니터링 <-> 장소 | Customer-Supplier | 모니터링이 장소 데이터 수집을 위임 |
| 대안 -> 장소 | Customer-Supplier | 대안이 주변 장소 검색을 위임 |
| 대안 -> 모니터링 | Customer-Supplier | 대안이 후보 장소 상태를 조회 |
| 대안 -> 일정 | Customer-Supplier | 대안 선택 시 일정 교체를 요청 |

## 6. 분할/병합 결정 (MVP)

### 6-1. 분석 매트릭스

| 바운디드 컨텍스트 | 차별화 핵심 | 부하 변동 | 변경 빈도 | 데이터 소유 | 판정 |
|:---------------:|:----------:|:--------:|:--------:|:----------:|:----:|
| 일정 | - | 낮음 | 중간 | Trip | **독립** |
| 장소 | - | 중간 | 낮음 | Place | **독립** |
| 모니터링 (수집+판정) | - | **높음** | 높음 | CollectedData, PlaceStatus | **독립** |
| 브리핑 (브리핑+알림) | 핵심(S04) | 중간 | 높음 | Briefing, Notification | **MVP 통합** |
| 대안 | 핵심(S06) | 중간 | 높음 | Alternative | **독립** |

### 6-2. 병합 근거

**데이터수집 + 상태판정 -> 모니터링 서비스**
- 근거: 데이터 수집과 상태 판정은 순차적 파이프라인 (수집 -> 판정)
- 동기 호출 비율 95% 이상
- 트랜잭션 경계 공유 (수집 데이터 저장과 상태 갱신이 원자적)
- MVP 예상 코드량: ~3,000줄

**브리핑 + 알림 -> 브리핑 서비스**
- 근거: 브리핑 생성과 알림 발송이 1:1 대응
- MVP에서 알림 채널이 Push 1종뿐
- MVP 예상 코드량: ~2,000줄
- 확장 시점: 알림 채널 추가(SMS, 카카오톡 등) 시 알림 서비스 분리

| # | 서비스명 | 바운디드 컨텍스트 | Aggregate | 핵심 책임 |
|:-:|---------|:---------------:|-----------|----------|
| 1 | **일정 서비스** (schedule-service) | 일정 | Trip, ScheduleItem | 여행 일정 CRUD, 장소 추가/교체, 모니터링 등록 요청 |
| 2 | **장소 서비스** (place-service) | 장소 | Place, PlaceDetail | 장소 검색/조회, Google Places API 연동, 장소 캐싱 |
| 3 | **모니터링 서비스** (monitoring-service) | 모니터링 | CollectedData, PlaceStatus | 외부 데이터 주기적 수집, 상태 배지 판정, 상태 변경 감지 |

| 5 | **대안 서비스** (alternative-service) | 대안 | Alternative, SearchContext | 맥락 기반 대안 검색, 카드 3장 생성, 추천 이유 제공 |

> 총 5개 서비스로 팀당 3~5개 서비스 적정 기준에 부합한다.

## 9. 인증/구독 최소 요구사항

> MVP에서 사용자 인증/구독 관리는 별도 바운디드 컨텍스트로 분리하지 않고, 각 서비스에서 공통 미들웨어/라이브러리를 통해 처리한다.

### 9-1. 구독 티어 정의

| 티어 | 브리핑 | 대안 카드 | 가격 |
|:----:|:------:|:--------:|------|
| **Free** | 1일 1회 | 미제공 (Paywall) | 무료 |
| **Trip Pass** | 무제한 | 제공 | 여행별 과금 |
| **Pro** | 무제한 | 제공 | 월정액 |

### 9-2. 티어 확인 지점

| 확인 지점 | 서비스 | 분기 동작 |
|----------|:------:|----------|
| 브리핑 생성 트리거 시 | 브리핑 | Free: 일일 한도 확인 -> 초과 시 스킵 |
| 대안 장소 검색 요청 시 | 대안 | Free: Paywall 표시 -> 업그레이드 안내 |

### 9-3. MVP 인증 최소 요구사항

- 소셜 로그인 (Google, Apple) 기반 인증
- JWT Access Token + Refresh Token 방식
- API Gateway 레벨에서 토큰 검증
- 구독 티어 정보는 토큰 클레임에 포함하여 각 서비스에서 참조

## 10. MVP 배포 전략

> 파이프(DevOps) 의견(D6)을 반영하여 MVP 배포 전략을 정리한다.

### 10-1. 단계별 전략

| 단계 | 구조 | 설명 | 전환 조건 |
|:----:|------|------|----------|
| **v0 (MVP)** | 모놀리스 우선 | 5개 서비스를 논리적 모듈로 분리하되 단일 배포 단위로 운영. 서비스 간 통신은 내부 메서드 호출 | MVP 출시 및 초기 사용자 확보 |
| **v1** | 모니터링 서비스 분리 | 부하 변동이 가장 큰 모니터링 서비스를 독립 배포 단위로 분리 | 수집 주기 증가 또는 동시 모니터링 장소 1,000개 초과 시 |
| **v2+** | 점진적 분리 | 브리핑, 대안, 장소 서비스를 필요에 따라 순차 분리 | 개별 서비스의 독립 스케일링 필요 시 |

### 10-2. 모놀리스 우선 근거

- 팀 규모(8명)와 MVP 범위에서 마이크로서비스 운영 오버헤드가 개발 속도 저하 유발
- 논리적 모듈 경계(패키지 구조)를 명확히 하여 추후 분리 비용 최소화
- 서비스 간 인터페이스는 처음부터 API 계약으로 정의 (분리 시 HTTP/이벤트로 교체 용이)
- 모니터링 서비스는 스케줄러 부하 패턴이 다른 서비스와 상이하므로 1차 분리 대상

### 10-3. 분리 준비 설계 원칙

- 각 모듈은 독립 패키지 구조 유지 (schedule, place, monitoring, briefing, alternative)
- 모듈 간 직접 의존 금지 -> 인터페이스를 통한 간접 의존
- 데이터베이스 스키마는 모듈별 논리적 분리 (스키마 프리픽스 또는 별도 스키마)
- 비동기 통신 포인트는 이벤트 인터페이스로 추상화 (MVP에서는 인메모리 이벤트 버스)
