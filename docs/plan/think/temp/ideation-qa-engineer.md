## Steal & Synthesize 분석

| # | 산업 | 사례 | 핵심 메커니즘 | 우리 문제 적용 가능성 |
|---|------|------|--------------|----------------------|
| 1 | 보험 | 자동차 보험 텔레매틱스(UBI — Usage-Based Insurance) | 운전 중 실시간 주행 데이터를 수집하여, 사고 직전 급제동/급가속 패턴 감지 시 사전 경고 알림을 발송. 사고가 발생하기 **전에** 리스크 신호를 포착해 운전자 행동을 교정한다. 핵심은 "이미 일어났다"가 아니라 "곧 일어날 것 같다"의 예측 알림 구조. | 여행자가 목적지에 **도착하기 전** 해당 장소의 혼잡·날씨·운영 리스크 신호를 감지해 경로 변경을 유도하는 Push 알림 구조에 직접 대입 가능. 80% 도착 후 인지 문제를 '도착 전 사전 경고'로 전환하는 핵심 메커니즘과 완전히 일치. |
| 2 | 헬스케어 | 연속혈당측정기(CGM) + 저혈당 예측 알림 (Dexcom G7) | 혈당이 실제로 위험 수치에 **도달하기 전에** "20분 후 저혈당 예상" 알림을 발송. 단순 현재 상태 모니터링이 아니라, 트렌드(변화 속도)를 분석해 미래 상태를 예측. 이 알림이 오면 사용자는 간식을 먹거나 인슐린을 조정하는 **사전 행동**을 취한다. | 여행 중 "지금 그 식당이 혼잡하다"가 아니라 "지금 출발하면 도착 시 대기 45분 예상"이라는 트렌드 기반 예측 알림 구조로 적용. 변화의 속도와 방향성을 분석해 여행 흐름 붕괴를 선제 차단. |
| 3 | 사이버 보안 | SIEM(보안 정보·이벤트 관리) + 위협 인텔리전스 플랫폼 (Splunk, CrowdStrike) | 개별 이벤트(로그인 실패 1회)는 무시하지만, 복수 신호를 **상관 분석(correlation)**하면 침해 시도로 판단해 알림 발송. 임계값을 넘은 순간이 아니라, 패턴 조합이 위험 시나리오와 일치할 때 사전 차단. | 날씨 악화 + 혼잡도 급증 + 교통 지연이 동시에 발생할 때 "연쇄 붕괴 위험" 경보를 발령하는 복합 신호 상관 분석에 직접 적용. 단일 변수 알림이 아닌 문제 C(연쇄 붕괴) 차단을 위한 다변수 감지 로직의 근거 모델. |
| 4 | 항공 | 비행 운항 통제(OCC — Operations Control Center) | 항공사는 기상, 공항 용량, 항공기 상태, 승무원 가용성을 실시간 통합 모니터링해, 지연이 **확정되기 전** 대안 항공편·게이트를 자동 배정. 승객은 이미 게이트 변경 알림을 받고 이동 중. | 여행자의 "개인 OCC" 개념. 내 일정에 영향을 줄 모든 변수를 통합 관제하는 허브가 없다는 문제 A(통합 허브 부재)에 대한 직접적 해법 모델. 탑승 전 대안 배정처럼, 출발 전 대안 코스를 자동 준비. |

## 아이디어

| # | 구분 | 아이디어 제목 | SCAMPER/S&S 관점 | 아이디어 상세 설명 |
|---|------|--------------|-----------------|-------------------|
| 1 | Big Idea | 여행 이상 탐지 레이더 — "출발 전 30분 경보 시스템" | **Steal (CGM + 보험 텔레매틱스)** + **Substitute (현재 상태 알림 → 미래 상태 예측 알림)** | QA 엔지니어로서 가장 두려운 것은 "테스트를 통과했는데 프로덕션에서 터지는 것"이다. 여행도 똑같다. 지금 내 화면엔 아무 문제가 없는데, 목적지에 도착하면 이미 닫혀 있다. 이 서비스의 핵심은 "도착 전 30분 이내" 이상 탐지 Push 알림이다. 연속혈당측정기(CGM)가 혈당 트렌드로 저혈당을 예측하듯, 이 시스템은 장소별 혼잡도 변화 속도, 날씨 악화 속도, SNS 부정 반응 급증 속도를 트렌드 분석해 "지금 출발하면 도착 시 상황이 X일 확률 Y%"를 계산한다. 단순 현재 상태가 아니라 도착 시점의 예측 상태를 알린다. 사용자는 이미 이동 중이므로 "지금 당장 방향을 바꿀 수 있는 타이밍"에 알림이 온다. 정보 정확도 90%+ 전제를 충족하기 위해 알림에는 신뢰도 수치를 함께 표시하고, 낮은 신뢰도 알림은 발송하지 않는 필터를 둔다. 이것이 킹핀 문제 B(도착 전 인지)를 직접 해결하는 핵심 메커니즘이다. |
| 2 | Big Idea | 연쇄 붕괴 차단 시뮬레이터 — "내 일정의 도미노 리스크 지도" | **Steal (SIEM 상관 분석)** + **Combine (단일 장소 알림 → 일정 전체 영향도 분석)** | 보안에서 알림 피로(Alert Fatigue)가 가장 큰 문제다. 경보가 너무 많으면 사람은 무시하기 시작한다. 여행 알림도 같다. "그 식당 혼잡합니다"라는 알림은 무시하기 쉽다. 하지만 "이 식당 지연 시 오늘 나머지 일정 3개가 연쇄 영향을 받습니다. 지금 대안 루트로 바꾸면 전부 살릴 수 있습니다"라는 알림은 무시할 수 없다. SIEM의 상관 분석처럼, 단일 이벤트가 내 전체 일정 파이프라인에 미치는 연쇄 영향을 그래프로 시각화한다. 도미노처럼 연결된 일정 중 어느 하나가 흔들릴 때 그 파동이 뒤의 일정들에 어떻게 전파되는지 즉각 계산해 보여준다. 사용자는 "지금 이 문제 하나를 해결하면 나머지가 살아난다"는 것을 직관적으로 이해한다. 문제 C(연쇄 붕괴)의 근본 해결이다. |
| 3 | Big Idea | 개인 여행 OCC (운항 통제 센터) — "내 주머니 속 관제탑" | **Steal (항공 OCC)** + **Adapt (기업용 관제 → 개인용 여행 관제)** | 항공사의 운항 통제 센터(OCC)는 수십 개의 변수를 동시에 모니터링하며 사전에 대안을 준비한다. 일반 여행자에게는 이런 시스템이 없다. "내 오늘 하루 여행 일정"을 하나의 운항 편으로 취급하고, 실시간으로 날씨·혼잡·운영시간·교통이라는 4개 변수를 통합 모니터링하는 개인용 관제탑을 만든다. 홈 화면은 대시보드 형태로, 내 일정의 현재 상태를 신호등(초록/노랑/빨강)으로 표시한다. 문제가 생기면 이미 대안 코스가 준비되어 있어서 사용자는 "수락/거절" 버튼 하나로 경로 변경 완료. 대안 탐색 23.5분을 제로로 만드는 것이 목표다. 통합 허브 부재(문제 A)를 단일 대시보드로 해결한다. QA 관점에서 중요한 것은 이 시스템이 "정보가 없을 때 어떻게 행동하는가"인데, 데이터 부재 시 해당 항목을 "확인 불가" 상태로 표시해 사용자가 직접 판단할 수 있게 투명성을 보장한다. |
| 4 | Little Win | 목적지 도착 전 5분 체크인 카드 — "지금 들어가도 될까요?" | **Simplify (복잡한 예측 시스템 → 단순 OK/NG 카드)** + **S&S (비행기 탑승 전 게이트 확인 경험)** | 복잡한 분석 없이, 목적지로 이동 중일 때 자동으로 화면에 카드 하나가 뜬다. "A 식당, 지금 도착하면: 대기 20분 예상 / 날씨 맑음 / 영업 중". 딱 이 세 줄이다. OK면 그대로 가고, NG면 카드를 옆으로 밀면 대안 장소 하나가 나온다. 인터페이스는 비행기 탑승 전 게이트 변경 알림처럼 단순하게 설계한다. 사용자가 결정을 내리기 위한 인지 부하를 최소화한다. QA 관점에서 "잘못된 정보를 단순하게 보여주는 것이 복잡한 정보보다 더 위험하다"는 것을 알기에, 이 카드는 신뢰도 낮은 데이터가 있을 경우 해당 항목을 숨기고 확인 가능한 항목만 표시하는 안전 설계를 기본으로 한다. 작지만 킹핀 문제 B를 직접 터치하는 가장 빠르게 구현 가능한 MVP 후보다. |
| 5 | Little Win | 여행 루틴 복구 스냅샷 — "이전에 이런 상황에서 나는 어떻게 했나" | **Modify (개인화) + Reuse (과거 행동 패턴 재활용)** | 사용자가 이미 경험한 여행에서의 돌발 상황 대처 방법을 스냅샷으로 저장해두고, 유사 상황 발생 시 "지난번 도쿄 여행에서 비슷한 상황에 이 코스로 바꿨더니 만족도 높았어요"라고 알려준다. 헬스케어의 개인 건강 이력 기반 처방처럼, 여행자의 과거 패턴이 미래 대안 추천의 가장 신뢰할 수 있는 데이터가 된다. 타인의 리뷰가 아닌 나 자신의 검증된 경험에서 나온 대안이므로 신뢰도가 높고 결정이 빠르다. 신규 사용자는 이 기능을 쓸 수 없지만, 재방문 사용자에게는 강력한 잠금 효과(Lock-in)를 만든다. 구현 난이도가 낮고, 리텐션(재방문율) 향상에 직접 기여하는 작은 승리다. |
| 6 | Crazy Idea | 여행 보험 연동 자동 클레임 — "일정이 망가지면 자동으로 보상받는다" | **Steal (보험 텔레매틱스 자동 사고 감지)** + **Eliminate (클레임 신청 절차 완전 제거)** | 가장 미친 생각이지만, 사용자 관점에서 가장 통쾌하다. 예약된 레스토랑이 당일 임시 휴업하면, 우리 앱이 이를 탐지하는 순간 사전 연동된 여행 보험사 API에 자동으로 클레임을 신청하고, 사용자 계좌로 소액 보상금이 입금된다. 사용자는 아무것도 할 필요가 없다. 자동차 사고 시 블랙박스 영상이 자동으로 보험사로 전송되는 것처럼, 우리 앱이 수집한 실시간 장소 상태 데이터가 보험 청구의 근거 증빙이 된다. 이것이 실현되면 서비스의 가치 제안이 "알려드립니다"에서 "보상해드립니다"로 격상된다. 보험사 입장에서는 정확한 데이터 기반 클레임으로 허위 청구가 줄고, 여행자 입장에서는 스트레스가 보상으로 바뀐다. 현실화를 막는 장벽은 보험사 API 연동 계약과 규제 문제지만, 이것이 풀린다면 세계 최초의 여행 실시간 자동 보상 플랫폼이 된다. |
