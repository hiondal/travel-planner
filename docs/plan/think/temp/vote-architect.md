## B 투표 (비즈니스 가치) -- 3표

| 솔루션 | 투표수 | 투표 근거 |
|--------|:------:|----------|
| S01 여행 일정 실시간 모니터링 & 사전 알림 | 2표 | 아키텍처 관점에서 S01은 단일 솔루션이 아니라 **전체 플랫폼의 코어 엔진**이다. S02(대시보드), S03(재조정), S04(브리핑), S05(배지), S06(대안카드), S08(단계적 대응) 모두가 S01의 모니터링 인프라 위에 구축된다. 즉 S01을 구축하면 다른 6개 솔루션의 기반이 동시에 만들어지는 **기술적 레버리지가 가장 큰** 솔루션이다. 킹핀 문제 B(사후 인지)를 Pull에서 Push로 패러다임 전환하는 핵심이며, 9명 전원이 독립 발상했다는 합의 강도도 기술 투자 리스크를 낮추는 근거가 된다. 카카오내비의 실시간 경로 모니터링, FDS의 이상 감지, 스마트 팩토리의 예측 정비 -- 모두 성숙한 아키텍처 패턴으로 기술적 참조 모델이 충분하다. |
| S03 AI 자동 일정 재조정 엔진 | 1표 | B->A->C 인과 체인 전체를 **하나의 end-to-end 파이프라인**으로 해결하는 유일한 솔루션이다. 항공사 IROPS의 "감지->영향 산정->대안 생성->통보" 자동 파이프라인을 여행 도메인에 적용한 것으로, 기술적 차별화 장벽(moat)이 가장 높다. MVP에서 완전 구현은 어렵지만, S01(감지) -> S06(대안 제시)의 파이프라인 위에 "후속 일정 영향 시뮬레이션"을 점진적으로 추가하는 **확장 경로가 명확**하다. During-trip 카테고리에서 경쟁자가 모방하기 가장 어려운 기술적 해자(moat)를 형성하며, 장기 비즈니스 가치가 가장 크다. |

## F 투표 (실현 가능성) -- 3표

| 솔루션 | 투표수 | 투표 근거 |
|--------|:------:|----------|
| S05 실시간 상태 배지 | 1표 | **기술적 리스크가 사실상 제로**인 솔루션이다. 백엔드는 Google Places API 상태 조회 -> 4단계 매핑(초록/노랑/빨강/회색) -> REST 응답이 전부이고, 프론트엔드는 아이콘 렌더링 하나다. MVP Sprint 1에서 1~2주 내 구현이 확실하며, 이후 WebSocket 기반 실시간 업데이트로 점진적 고도화가 가능하다. "데이터 부재도 정보(회색 배지)"라는 설계는 정보 정확도 90% 전제조건을 유연하게 처리하는 아키텍처적으로 건전한 접근이다. 무엇보다 S01(모니터링)의 시각적 프론트엔드 컴포넌트로서, 전체 아키텍처의 첫 번째 빌딩 블록이 된다. |
| S04 출발 전 브리핑 | 1표 | Google Places API(영업 상태) + 기상 API(날씨) + Google Maps API(이동 시간) 3개의 **성숙한 REST API 조합**만으로 구현 가능하다. 복잡한 AI나 실시간 스트리밍이 필요 없고, 스케줄러(cron) 기반의 트리거 + 캐싱 전략으로 API 비용 최적화까지 단순하게 해결된다. 아키텍처적으로 API Gateway -> Scheduler -> Notification Service라는 **검증된 마이크로서비스 패턴**으로 구현하면 2~3주 내 MVP 가능하다. 킹핀 문제 B를 가장 직접적으로, 가장 가볍게 해결하면서 동시에 S01 모니터링 엔진의 **데이터 수집 파이프라인을 자연스럽게 구축**하는 효과가 있어, 이후 S01으로의 확장 경로가 매끄럽다. |
| S06 맥락 맞춤 대안 카드 | 1표 | MVP 단계에서는 LLM 없이 **규칙 기반 엔진**(현재 위치 반경 + 동일 카테고리 + 영업 중 필터 + 거리/평점 정렬)으로 충분히 구현 가능하다. Google Places API의 Nearby Search가 핵심 데이터를 제공하고, 카드 UI는 장소명+거리+평점+혼잡도+추천이유 5개 필드로 표준화된다. 기술적으로 킹핀 A(대안 탐색 23.5분)를 해결하는 가장 직접적인 경로이면서, 이후 LLM 기반 개인화 추천으로 **점진적 고도화가 자연스러운** 아키텍처를 설계할 수 있다. S04(브리핑)에서 "주의 필요" 판정 시 S06(대안 카드)으로 연결되는 **자연스러운 UX 파이프라인**이 형성되어, 두 솔루션의 시너지가 크다. |

## 투표 총평

아키텍트로서 나의 투표는 **"플랫폼 코어 엔진의 비즈니스 가치"와 "점진적 확장이 가능한 기술 기반"** 두 축으로 구성된다. B 투표에서 S01에 2표를 집중한 이유는, 이 솔루션이 단독 기능이 아니라 S02~S08까지 6개 솔루션이 의존하는 **인프라 플랫폼**이기 때문이다. 하나를 만들면 여섯의 기반이 완성되는 기술적 레버리지는 비즈니스 가치 그 자체다. S03에 1표를 준 것은 B->A->C 전체를 자동화하는 end-to-end 파이프라인이 경쟁자가 모방하기 가장 어려운 기술적 해자(moat)를 형성하기 때문이다. F 투표에서 S05+S04+S06 조합을 선택한 이유는, 이 세 솔루션이 MVP Sprint 1~2(4~6주) 내에 **검증된 REST API와 성숙한 아키텍처 패턴만으로 안정적으로 구현 가능**하면서, 동시에 킹핀 B(사전 인지)와 A(대안 탐색)를 동시에 커버하고, 이후 S01(모니터링 엔진)으로의 확장 경로가 자연스럽게 열리는 **아키텍처적으로 건전한 MVP 기반**을 형성하기 때문이다. 기술 투자의 핵심 원칙은 "작게 시작하되, 확장 경로를 처음부터 설계에 반영하라"이며, 이 투표는 그 원칙을 반영한다.
